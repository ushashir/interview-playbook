JavaScript and the Browser

JavaScript has more ways to surprise you than any other mainstream language. The candidate who says "I know JavaScript well" is often the most dangerous person in the room — because they don't know what they don't know.

What you'll learn

  • Why var, let, and const behave differently — and the classic bugs each one creates
  • How this works — and why it changes depending on how a function is called
  • What closures actually are, why they're powerful, and why they trip up interviewers
  • How event delegation works and why it matters for performance
  • The prototype chain — enough to answer any inheritance question confidently
  • Promises, async/await, and when to reach for Promise.all vs Promise.allSettled
  • The event loop — call stack, microtask queue, macrotask queue, and exactly why 1, 4, 3, 2
  • TypeScript's unknown vs any — the type-safe difference that senior engineers know cold

1. var, let, and const — Scoping and Hoisting

The Party Announcement Analogy

var is like shouting your name at a party — everyone in the whole building hears it, from the basement to the penthouse. let and const are like whispering to the person next to you — only the people in your immediate conversation group know your name. When you leave the room, outsiders have no idea you were ever there.

Definitions

  • var: Function-scoped (or global if declared outside a function). Hoisted to the top of its function and initialized to undefined immediately. This means you can use a var variable before the line that declares it — it just evaluates to undefined.
  • let: Block-scoped (confined to the nearest {}). Hoisted but NOT initialized — accessing it before the declaration throws a ReferenceError. This period between the top of the block and the declaration is called the Temporal Dead Zone (TDZ).
  • const: Block-scoped, must be initialized at declaration, cannot be reassigned. The object or array it points to can still be mutated.
// var hoisting gotcha
console.log(name); // undefined — not a ReferenceError!
var name = "Alice";
console.log(name); // "Alice"

// let temporal dead zone
console.log(city); // ReferenceError: Cannot access 'city' before initialization
let city = "Lagos";

// const: the binding is immutable, not the object
const user = { name: "Alice" };
user.name = "Bob"; // Fine — we're mutating the object
user = {};         // TypeError: Assignment to constant variable

The Classic var-in-a-Loop Bug

This is one of the most commonly asked JavaScript interview questions. It looks simple and trips up even experienced candidates who haven't thought about it carefully.

// BUG: var is function-scoped, so i is shared across all closures
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Output: 3, 3, 3  (not 0, 1, 2)
// By the time the timeouts fire, the loop has finished and i === 3

// FIX 1: Use let (block-scoped — each iteration gets its own i)
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Output: 0, 1, 2

// FIX 2: Use an IIFE to capture i by value (pre-ES6 solution)
for (var i = 0; i < 3; i++) {
  ((capturedI) => {
    setTimeout(() => console.log(capturedI), 100);
  })(i);
}
// Output: 0, 1, 2

Why does the first example output 3, 3, 3? Because var i is not block-scoped to each loop iteration — it's the same variable throughout. When the setTimeout callbacks finally run (after the loop finishes), they all look up the current value of i, which is 3. let creates a new binding per iteration, so each callback captures its own independent copy.

2. The this Keyword

The "This Building" Analogy

"This building" means something different depending on where you're standing when you say it. If you're in London, "this building" is Big Ben. If you're in Lagos, "this building" is your office. JavaScript's this works the same way — it always refers to the execution context, the "where" in which the code is running at that moment. Move the code to a different context, and this points somewhere new.

// 1. Regular function — this is determined by HOW it's called
function greet() {
  console.log(this.name);
}

const person = { name: "Alice", greet };
person.greet();   // "Alice" — called on the object, so this = person
greet();          // undefined (or window.name in browsers) — called standalone

// 2. Arrow function — this is lexically bound (from surrounding scope)
const counter = {
  count: 0,
  start() {
    // Arrow function captures `this` from start()'s scope = the counter object
    setInterval(() => {
      this.count++;
      console.log(this.count); // Works correctly!
    }, 1000);
  }
};

// 3. The common bug: losing `this` by passing a method as a callback
class Timer {
  constructor() {
    this.seconds = 0;
  }
  tick() {
    this.seconds++; // `this` here depends on how tick() is called
  }
}

const t = new Timer();
setInterval(t.tick, 1000); // BUG: `this` inside tick() will be undefined (strict mode)
                            // or the global object — not the Timer instance

// FIX 1: bind()
setInterval(t.tick.bind(t), 1000);

// FIX 2: arrow function wrapper
setInterval(() => t.tick(), 1000);

// FIX 3: class field arrow function (binds at instance creation time)
class Timer2 {
  seconds = 0;
  tick = () => {        // arrow function as class field
    this.seconds++;     // `this` is always the instance
  }
}

The one-line rule for this

In a regular function, this is determined at call time. In an arrow function, this is determined at definition time. That's the whole model.

3. Closures

The Lunchbox Analogy

You packed a lunchbox before school. Even after you leave the house and the front door is locked — the house's scope is gone — the lunchbox still has your sandwich inside. You carry it with you all day. A closure is a function that packed its own lunchbox: it carries the variables from the scope where it was created, even after that outer scope has finished executing and its local variables would normally be garbage collected.

Technical definition

A closure is a function bundled together with references to its lexical environment — the variables in scope at the time the function was defined. The function "closes over" those variables, keeping them alive as long as the function itself is reachable.

// Basic closure: the returned function remembers `count`
function makeCounter() {
  let count = 0; // This variable lives in the closure
  return {
    increment() { count++; },
    decrement() { count--; },
    value()     { return count; }
  };
}

const counter = makeCounter();
counter.increment();
counter.increment();
counter.increment();
console.log(counter.value()); // 3

// count is private — there's no way to set it directly from outside
// This is the Module Pattern in its simplest form

// ------------------------------------------------
// Practical: memoization using a closure
function memoize(fn) {
  const cache = new Map(); // cache lives in the closure
  return function(...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) {
      return cache.get(key); // cache hit
    }
    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}

function slowSquare(n) {
  // Simulate expensive computation
  return n * n;
}

const fastSquare = memoize(slowSquare);
fastSquare(10); // Computes: 100
fastSquare(10); // Cache hit: 100 — no recomputation

// ------------------------------------------------
// Classic closure bug in loops (with var — see section 1)
// The fix using let or IIFE was shown above

Closures are the mechanism behind private state in JavaScript. Without class private fields or closures, there was historically no native way to create truly private data. The counter example above is the classic module pattern — count is inaccessible from outside because nothing outside has a reference to the lexical environment where it lives.

4. Event Delegation

The Front Desk Analogy

Instead of hiring a personal receptionist for every single office in a skyscraper, you hire one receptionist at the front desk. Every visitor goes through the front desk, which checks who they're visiting and routes them to the right floor. Event delegation works the same way — instead of attaching a listener to every child element, you attach one listener to a parent, and let the event bubble up to it. The parent checks event.target to figure out which child was actually clicked.

// WITHOUT delegation: one listener per item (bad for large lists)
document.querySelectorAll('.task-item').forEach(item => {
  item.addEventListener('click', handleClick);
});
// Problem: if you add new items dynamically, they don't get the listener.
// Problem: 1000 items = 1000 listeners = memory pressure.

// WITH delegation: one listener on the parent
const taskList = document.getElementById('task-list');

taskList.addEventListener('click', function(event) {
  // event.target: the element the user actually clicked
  // event.currentTarget: the element with the listener (taskList)

  const item = event.target.closest('.task-item');
  if (!item) return; // Click was somewhere else in the parent — ignore

  const taskId = item.dataset.taskId;
  console.log('Clicked task:', taskId);
});

// Dynamically added items automatically work — no re-binding needed
function addTask(id, text) {
  const li = document.createElement('li');
  li.className = 'task-item';
  li.dataset.taskId = id;
  li.textContent = text;
  taskList.appendChild(li); // The delegation listener catches clicks on this too
}

Event delegation works because of event bubbling: when you click a <li>, the click event fires on the <li>, then bubbles up to the <ul>, then to <body>, then to window. The parent listener intercepts it at the <ul> level. event.target tells you where the event originated. event.currentTarget is always the element with the listener attached.

5. The Prototype Chain

The Family Inheritance Analogy

If you ask a person "do you own a car?" and they say no, you don't give up — you check their parents. If their parents don't own one either, you check the grandparents. The search walks up the family tree until you either find it or run out of family. JavaScript's prototype chain works exactly this way: when you access a property on an object, JavaScript looks at that object first, then its prototype, then the prototype's prototype, all the way up to null.

// Low-level: Object.create()
const animal = {
  breathe() { return "inhale, exhale"; }
};

const dog = Object.create(animal); // dog's prototype IS animal
dog.bark = function() { return "woof"; };

dog.bark();    // Found directly on dog
dog.breathe(); // Not on dog — found on dog's prototype (animal)

// How it works internally:
// dog.__proto__ === animal         (true)
// animal.__proto__ === Object.prototype (true)
// Object.prototype.__proto__ === null  (end of chain)

// ------------------------------------------------
// class is syntactic sugar over the same prototype system
class Animal {
  constructor(name) {
    this.name = name;
  }
  breathe() {
    return `${this.name} inhales, exhales`;
  }
}

class Dog extends Animal {
  bark() {
    return `${this.name} says woof`;
  }
}

const rex = new Dog("Rex");
rex.bark();    // Own method
rex.breathe(); // Inherited via prototype chain from Animal.prototype

// Verify:
// Object.getPrototypeOf(rex) === Dog.prototype    (true)
// Object.getPrototypeOf(Dog.prototype) === Animal.prototype (true)

Interview-ready one-liner

"JavaScript uses prototypal inheritance — every object has a hidden link ([[Prototype]]) to another object. Property lookup walks this chain upward until the property is found or we hit null. The class keyword doesn't change this model — it's syntactic sugar that sets up the same prototype links."

6. Promises and async/await

The Restaurant Order Analogy

You walk into a restaurant and place your order. The waiter hands you a buzzer and walks away — that's the Promise being created (pending state). The kitchen is working on your food. Eventually, either your buzzer goes off and you get your meal (fulfilled), or the waiter comes back and tells you they're out of the dish (rejected). async/await is just a more readable syntax for waiting for the buzzer — you sit at the table and "await" your food, but the kitchen is still asynchronous underneath.

// Promise states: pending → fulfilled OR rejected

// Creating a promise manually
const fetchUser = (id) => new Promise((resolve, reject) => {
  setTimeout(() => {
    if (id > 0) {
      resolve({ id, name: "Alice" }); // fulfilled
    } else {
      reject(new Error("Invalid user ID")); // rejected
    }
  }, 500);
});

// .then()/.catch() chaining
fetchUser(1)
  .then(user => {
    console.log("Got user:", user.name);
    return fetchUser(2); // chain another promise
  })
  .then(user2 => console.log("Also got:", user2.name))
  .catch(err => console.error("Error:", err.message))
  .finally(() => console.log("Done, either way"));

// async/await — same logic, more readable
async function loadUsers() {
  try {
    const user1 = await fetchUser(1);
    console.log("Got user:", user1.name);
    const user2 = await fetchUser(2);
    console.log("Also got:", user2.name);
  } catch (err) {
    console.error("Error:", err.message);
  } finally {
    console.log("Done, either way");
  }
}

// ------------------------------------------------
// Promise combinators — know when to use each

// Promise.all: all must succeed; fails fast if any reject
// Use when results are independent but you need all of them
const [users, posts] = await Promise.all([
  fetchUsers(),
  fetchPosts()
]); // Both start simultaneously — takes max(time1, time2), not sum

// Promise.race: resolves/rejects with the first one to settle
// Use for timeout patterns
const result = await Promise.race([
  fetchData(),
  new Promise((_, reject) => setTimeout(() => reject(new Error("Timeout")), 5000))
]);

// Promise.allSettled: waits for all, never fails-fast
// Returns array of { status: 'fulfilled'|'rejected', value|reason }
// Use when you want to know which succeeded and which failed
const results = await Promise.allSettled([
  fetchUser(1),  // might succeed
  fetchUser(-1), // will reject
  fetchUser(3)   // might succeed
]);

results.forEach(r => {
  if (r.status === 'fulfilled') console.log("OK:", r.value);
  else console.log("FAIL:", r.reason.message);
});

// ------------------------------------------------
// Common mistake: not handling rejections
async function badCode() {
  const data = await fetchUser(1); // If this rejects and there's no try/catch...
  // ...you get an UnhandledPromiseRejection — crashes Node.js in newer versions
}

// Always wrap await calls in try/catch or attach .catch() to the promise

7. The Event Loop

The Restaurant Chef Analogy

Picture a chef working alone in a kitchen. He can only cook one dish at a time — JavaScript is single-threaded. But he doesn't stand at the oven staring at a roast while it cooks. He starts the roast, sets a timer, and goes to chop vegetables. When the roast timer rings, he finishes what he's doing on the cutting board, then tends to the roast. The event loop is that chef's discipline: one task at a time, but never idle, checking queues between each task.

The four components of the event loop

  • Call Stack: Where synchronous code executes. One thing at a time. LIFO (last in, first out).
  • Web APIs / Node.js APIs: Where async operations live while they wait (timers, network, I/O). Managed by the browser or Node.js, not JavaScript itself.
  • Microtask Queue: Where Promise callbacks (.then(), await) wait. Drained completely before the next macrotask runs.
  • Macrotask Queue (Callback Queue): Where setTimeout, setInterval, and I/O callbacks wait. One macrotask is run per event loop "tick."
// The classic interview question — predict the output:
console.log(1);

setTimeout(() => console.log(2), 0);

Promise.resolve().then(() => console.log(3));

console.log(4);

// Output: 1, 4, 3, 2

Let's trace exactly what happens, step by step:

  1. console.log(1) — pushed to call stack, executes synchronously, prints 1, popped.
  2. setTimeout(() => console.log(2), 0) — the browser's Timer API receives the callback. Even with 0ms delay, it goes to the macrotask queue after the minimum delay. Call stack continues.
  3. Promise.resolve().then(() => console.log(3)) — the Promise is already resolved. The .then() callback is scheduled into the microtask queue immediately.
  4. console.log(4) — pushed to call stack, executes, prints 4, popped.
  5. Call stack is now empty. Before the event loop picks the next macrotask, it drains the microtask queue completely. The () => console.log(3) callback runs: prints 3.
  6. Microtask queue is now empty. The event loop picks the next macrotask: the setTimeout callback runs: prints 2.

The critical rule: microtasks always run before the next macrotask. This is why Promises (microtasks) always resolve before timeouts (macrotasks), even if the timeout has a delay of zero.

// More complex example — can you predict this?
console.log('start');

setTimeout(() => console.log('timeout 1'), 0);

Promise.resolve()
  .then(() => {
    console.log('promise 1');
    return Promise.resolve();
  })
  .then(() => console.log('promise 2'));

setTimeout(() => console.log('timeout 2'), 0);

console.log('end');

// Output:
// start
// end
// promise 1
// promise 2
// timeout 1
// timeout 2

Note that "promise 2" still appears before both timeouts, even though it's chained off promise 1. Each .then() in a Promise chain schedules a new microtask, but all microtasks drain before any macrotask runs.

Junior answer to "Explain the event loop" "JavaScript is single-threaded, so it uses a queue for async stuff. Callbacks go into the queue and run when the main thread is free."
Senior answer to "Explain the event loop" "JavaScript has a single call stack — it can only do one thing at a time. Async operations are handled by the host environment (browser or Node.js) and their callbacks are placed into queues. There are two kinds of queues that matter: the microtask queue (Promises, queueMicrotask) and the macrotask queue (setTimeout, setInterval, I/O). After each synchronous task completes, the engine fully drains the microtask queue before picking the next macrotask. This is why Promise.resolve().then() fires before setTimeout with a zero delay — they're in different queues, and microtasks always go first. In practice, this means you need to be careful about long microtask chains blocking rendering, and you need to understand that 'setTimeout 0' doesn't mean 'immediate' — it means 'after current microtasks and at least one frame.'"

8. TypeScript: unknown vs any

The Package Label Analogy

any is like labeling a package "stuff" — you can put anything inside and take anything out, and the postal service will never question you. Zero checking. unknown is like labeling a package "unknown contents" — you have to open it and inspect it before you're allowed to use what's inside. One approach is convenient but dangerous. The other is safe by design.

// any: completely disables type checking
function processAny(value: any) {
  value.toUpperCase(); // TypeScript is silent — no error
  value.nonExistentMethod(); // Also silent — TypeScript trusts you
  return value * 2; // Still silent — may crash at runtime
}

// unknown: type-safe top type — you MUST narrow before using
function processUnknown(value: unknown) {
  value.toUpperCase(); // TypeScript ERROR: Object is of type 'unknown'

  // You must prove to TypeScript what the type is before using it
  if (typeof value === 'string') {
    console.log(value.toUpperCase()); // OK — narrowed to string
  }

  if (value instanceof Error) {
    console.log(value.message); // OK — narrowed to Error
  }

  if (typeof value === 'number') {
    return value * 2; // OK — narrowed to number
  }
}

// ------------------------------------------------
// Real-world use case: JSON parsing
function safeParseJSON(json: string): unknown {
  return JSON.parse(json); // JSON.parse returns `any` — we upgrade to `unknown`
}

const data = safeParseJSON('{"name": "Alice", "age": 30}');

// data.name — TypeScript ERROR: Object is of type 'unknown'

// Fix: narrow with a type guard
function isUser(value: unknown): value is { name: string; age: number } {
  return (
    typeof value === 'object' &&
    value !== null &&
    'name' in value &&
    'age' in value
  );
}

if (isUser(data)) {
  console.log(data.name); // OK — TypeScript knows this is a user
  console.log(data.age);
}

// ------------------------------------------------
// When to use each:
// - any: almost never in production code. Use when migrating a JS codebase to TS
//         and you need to ship quickly. Always treat it as temporary.
// - unknown: for values coming from external sources — API responses,
//             JSON parsing, user input, third-party library returns.
//             Forces you to handle all cases explicitly.

The silent any problem

any is contagious. A function that returns any spreads any to everything it touches downstream, silently disabling type safety across a large surface area. A codebase that uses any liberally gets the syntax of TypeScript without the safety guarantees. Prefer unknown — it forces you to be honest about what you know.

Practice Questions

Beginner

What is the output of the following code? Explain each line.

console.log(x);
var x = 5;
console.log(x);

console.log(y);
let y = 10;

Answer: Line 1 prints undefined — var x is hoisted and initialized to undefined. Line 3 prints 5. Line 5 throws a ReferenceError — let y is in the temporal dead zone; accessing it before the declaration is an error.

Intermediate

Implement a debounce function using closures. It should delay invoking a function until after a specified wait time has elapsed since the last call.

// Your implementation:
function debounce(fn, delay) {
  let timeoutId = null; // lives in the closure

  return function(...args) {
    // Cancel the previous pending call
    if (timeoutId) clearTimeout(timeoutId);

    // Schedule a new call
    timeoutId = setTimeout(() => {
      fn.apply(this, args); // preserve `this` context
      timeoutId = null;
    }, delay);
  };
}

// Usage:
const onSearch = debounce((query) => {
  console.log("Searching for:", query);
}, 300);

// If called 5 times in 200ms, only the last call fires (after 300ms of silence)
onSearch("j");
onSearch("ja");
onSearch("jav");
onSearch("java");
onSearch("javascript"); // This is the only one that actually fires

The closure is key here: timeoutId persists between calls because it lives in the lexical environment of the returned function. Each call can clear and reset it.

Advanced

Predict the output of the following snippet and trace each step of the event loop.

async function main() {
  console.log('A');

  await Promise.resolve();
  console.log('B');

  setTimeout(() => console.log('C'), 0);

  await new Promise(resolve => setTimeout(resolve, 0));
  console.log('D');
}

console.log('1');
main();
console.log('2');

Output: 1, A, 2, B, C, D

Trace:

  1. console.log('1') — synchronous, prints 1.
  2. main() is called. console.log('A') runs synchronously — prints A.
  3. await Promise.resolve() — Promise is already resolved; the continuation (everything after await) is scheduled as a microtask. main() suspends and control returns to the caller.
  4. console.log('2') — synchronous, prints 2. Call stack now empty.
  5. Microtask queue drains: the continuation of main() resumes. console.log('B') — prints B.
  6. setTimeout(() => console.log('C'), 0) — C's callback goes to macrotask queue.
  7. await new Promise(resolve => setTimeout(resolve, 0)) — the promise won't resolve until the setTimeout fires. main() suspends again.
  8. Macrotask queue: C's setTimeout fires first (it was queued first). Prints C. The resolve() callback from step 7 also fires, resolving the awaited promise.
  9. Promise resolves → continuation of main() queued as microtask → drains immediately → console.log('D') — prints D.

Chapter Summary

  • var is function-scoped and hoisted (initialized to undefined); let and const are block-scoped and sit in a temporal dead zone before their declaration — always prefer let/const.
  • this in regular functions is determined at call time; in arrow functions it is lexically bound at definition time — this distinction fixes a whole class of callback bugs.
  • A closure is a function plus its lexical environment; this enables private state, factory functions, and memoization, and is the mechanism behind the classic for-var-setTimeout bug.
  • The event loop drains the microtask queue completely after every synchronous task before picking the next macrotask — Promises always run before timeouts.
  • Prefer unknown over any in TypeScript; unknown forces type narrowing and keeps type safety intact across your codebase.
