Frontend Performance and DevTools

A user decides whether your app is fast or slow in about 300 milliseconds. That's not a UX preference — that's neuroscience. If you can't talk confidently about web performance, you will not get senior frontend roles at product companies.

What you'll learn

  • Core Web Vitals — Google's three performance metrics and exactly how to fix each one
  • The browser rendering pipeline from HTML bytes to pixels on screen
  • JavaScript performance: long tasks, code splitting, lazy loading, and tree shaking
  • Network performance: HTTP/2, resource hints, caching, and image optimization
  • How to use Chrome DevTools like a senior engineer — Performance, Network, Memory, and Coverage panels
  • Frontend memory leaks: how they happen and how to find them

Core Web Vitals

💡 Analogy: The Restaurant Health Inspection

A restaurant's health inspection doesn't measure everything — it focuses on three critical areas that determine if you're allowed to stay open: food temperature, cross-contamination prevention, and sanitation. If any of the three fails, you fail. Google's Core Web Vitals work the same way: three specific measurements that determine whether your page is considered "good" by search engines and users alike. Pass all three, you're healthy. Fail any one, and your search ranking pays the price.

Core Web Vitals are not abstract engineering metrics — they directly affect Google search ranking and user retention. A 100ms improvement in load time can produce measurable increases in conversion rates. This is why performance is a business conversation, not just a technical one.

LCP — Largest Contentful Paint

What it measures: How long until the largest content element visible in the viewport finishes rendering. This is typically the hero image, a large text block, or a video poster.

Good threshold: Under 2.5 seconds.

What causes poor LCP:

  • A large hero image served at full resolution without optimization
  • Render-blocking resources (large CSS or JavaScript files in <head>) that delay rendering
  • Slow server response time (TTFB — Time To First Byte over 600ms)
  • Client-side rendering where the page is blank until JavaScript runs

How to fix it:

<!-- 1. Preload the LCP image — tells the browser to fetch it immediately -->
<link rel="preload" as="image" href="hero.webp" fetchpriority="high">

<!-- 2. Use modern image formats -->
<picture>
  <source srcset="hero.avif" type="image/avif">
  <source srcset="hero.webp" type="image/webp">
  <img src="hero.jpg" alt="Hero" width="1200" height="600"
       fetchpriority="high">
  <!-- fetchpriority="high" on the LCP image is one of the highest-ROI optimizations -->
</picture>

<!-- 3. Never lazy-load above-the-fold images -->
<img src="hero.webp" loading="eager" alt="Hero">  <!-- eager = load immediately -->
<img src="below-fold.webp" loading="lazy" alt="..."> <!-- lazy for everything else -->

// 4. Measure LCP in code (useful for Real User Monitoring)
const observer = new PerformanceObserver((list) => {
  const entries = list.getEntries();
  const lastEntry = entries[entries.length - 1]; // Last LCP candidate wins
  console.log('LCP:', lastEntry.startTime, 'ms');
  console.log('LCP element:', lastEntry.element);
});
observer.observe({ type: 'largest-contentful-paint', buffered: true });

CLS — Cumulative Layout Shift

What it measures: The total amount of unexpected layout movement as the page loads. Every time content jumps — because an image loaded and pushed text down, or an ad appeared — that shift is scored.

Good threshold: Under 0.1 (it's a score, not a time unit).

What causes poor CLS:

  • Images without explicit width and height attributes — the browser doesn't know how much space to reserve
  • Dynamically injected content above existing content (ads, banners, cookie notices)
  • Web fonts causing text to reflow when they load (FOUT — Flash of Unstyled Text)
  • Animations that trigger layout (never animate width, height, margin, padding, top, left)

How to fix it:

<!-- Always set explicit dimensions on images -->
<img src="product.jpg" width="400" height="300" alt="Product">
<!-- The browser now knows this will be 400×300 and reserves the space
     before the image loads. No jump when the image appears. -->

<!-- For responsive images, use aspect-ratio in CSS instead -->

/* Use aspect-ratio to reserve space for responsive media */
.responsive-image-wrapper {
  aspect-ratio: 16 / 9;    /* Reserve the space */
  width: 100%;
  overflow: hidden;
  background: #f0f0f0;     /* Placeholder color while loading */
}

.responsive-image-wrapper img {
  width: 100%;
  height: 100%;
  object-fit: cover;
}

/* For web fonts: font-display: swap prevents invisible text but can shift */
/* font-display: optional avoids shift by using fallback if font isn't cached */
/* Preload critical fonts to minimize FOUT */

<!-- Preload critical fonts to reduce font-caused layout shift -->
<link rel="preload" href="/fonts/inter-var.woff2" as="font"
      type="font/woff2" crossorigin>

INP — Interaction to Next Paint

What it measures: How fast the page responds to user interactions — clicks, taps, keyboard input. INP replaced FID (First Input Delay) in 2024 because it measures all interactions throughout the page lifecycle, not just the first one.

Good threshold: Under 200ms.

What causes poor INP:

  • Long tasks on the main thread — JavaScript running for more than 50ms blocks the browser from responding to input
  • Expensive event handlers (doing too much work on click)
  • Forced synchronous layouts (reading layout properties after writing to the DOM)
  • Third-party scripts running on the main thread

How to fix it:

// BAD: expensive work runs synchronously on click, blocking the thread
button.addEventListener('click', () => {
  const data = fetchAndProcessLargeDataset(); // takes 400ms — terrible INP
  renderResults(data);
});

// GOOD: break long tasks using scheduler or setTimeout
button.addEventListener('click', async () => {
  // Show immediate feedback — critical for perceived responsiveness
  showLoadingState();

  // Yield to the browser between heavy operations
  await yieldToMain();
  const data = await fetchData();

  await yieldToMain(); // let the browser paint before the next heavy step
  renderResults(data);
  hideLoadingState();
});

// The yield-to-main pattern: gives the browser a chance to handle input/paint
function yieldToMain() {
  return new Promise(resolve => {
    // scheduler.postTask is the modern API (Chrome 94+)
    if ('scheduler' in window && 'postTask' in scheduler) {
      scheduler.postTask(resolve, { priority: 'user-visible' });
    } else {
      // Fallback: setTimeout(0) yields to the event loop
      setTimeout(resolve, 0);
    }
  });
}

// Measure INP in code
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.interactionId) {
      console.log('Interaction duration:', entry.duration, 'ms');
    }
  }
});
observer.observe({ type: 'event', buffered: true, durationThreshold: 16 });

The Browser Rendering Pipeline

💡 Analogy: Printing a Newspaper

Think of the browser rendering a page like a newspaper going through a printing press. First, typesetters lay out all the articles (parsing HTML into structure). The layout team arranges everything on the page grid (applying CSS). Then the paste-up team calculates exactly where each element sits (layout/reflow). The press then applies ink to the paper (paint). Finally, the printed sections are stacked together (compositing). If any stage changes, some or all of the stages after it must run again — which is why understanding which CSS properties trigger which stages is a performance superpower.

HTML Bytes ↓ Parse HTML → DOM Tree Parse CSS → CSSOM Tree ↓ Combine → Render Tree (only visible nodes, with styles attached) ↓ Layout (Reflow) Calculate exact position and size of every element ↓ Paint Fill in pixels: colors, borders, shadows, text ↓ Composite Combine painted layers in GPU memory (transform and opacity live here) ↓ Screen

Understanding this pipeline lets you answer "why is my animation janky?" definitively. The answer always traces back to which stage your code is triggering repeatedly.

What Triggers What

Change Triggers Cost Examples
Geometry changes Layout → Paint → Composite Most expensive width, height, margin, padding, top, left, font-size
Visual changes Paint → Composite Moderate color, background-color, border-color, box-shadow
Composited properties Composite only Cheapest (GPU) transform, opacity 
/* WRONG: animating left triggers layout on every frame — janky */
.element {
  position: absolute;
  left: 0;
  transition: left 0.3s ease;
}
.element:hover {
  left: 100px;
}

/* RIGHT: animating transform stays on the compositor — smooth 60fps */
.element {
  position: absolute;
  left: 0;
  transition: transform 0.3s ease;
  will-change: transform;   /* hint to browser: promote this to its own layer */
}
.element:hover {
  transform: translateX(100px);
}

/* will-change: use sparingly. Each layer uses GPU memory.
   Promoting everything to its own layer can actually HURT performance. */

Forced Synchronous Layout — The Performance Killer

// WRONG: forces the browser to recalculate layout on every iteration
// This is called "layout thrashing"
const elements = document.querySelectorAll('.box');

elements.forEach(el => {
  const height = el.offsetHeight;     // READ — forces layout calculation
  el.style.height = height * 2 + 'px'; // WRITE — invalidates layout
  // On the next iteration, READ again forces ANOTHER layout calculation
  // This can produce hundreds of layout calculations per second
});

// RIGHT: batch reads first, then writes
const heights = Array.from(elements).map(el => el.offsetHeight); // all reads

heights.forEach((height, i) => {
  elements[i].style.height = height * 2 + 'px'; // all writes
});
// Layout is calculated once. All writes apply to the invalidated layout.

// Even better: use CSS for layout changes when possible, not JavaScript

JavaScript Performance

JavaScript is the most expensive resource on the web. A JavaScript file doesn't just cost network time — it costs parse time, compile time, and execution time. An equivalent PNG costs only download time. This is why JavaScript budgets matter.

Long Tasks and the 50ms Rule

The browser's main thread handles JavaScript execution, layout, painting, and user input. It can only do one thing at a time. Any task that runs longer than 50ms is considered a "long task" and blocks user input during that window. This is the root cause of poor INP.

// Identifying long tasks in the Performance Observer API
const observer = new PerformanceObserver((list) => {
  list.getEntries().forEach(entry => {
    console.warn(`Long task: ${entry.duration}ms at ${entry.startTime}ms`);
    // Check entry.attribution to find what caused it
  });
});
observer.observe({ type: 'longtask', buffered: true });

Code Splitting

Ship only the code the current page needs. Load everything else on demand.

// Static import: always bundled, always downloaded
import { HeavyChart } from './HeavyChart';

// Dynamic import: loaded on demand — returns a Promise
async function showChart() {
  // The HeavyChart module is NOT in the initial bundle
  // It's fetched from the server only when this function is called
  const { HeavyChart } = await import('./HeavyChart');
  return new HeavyChart();
}

// React.lazy() — component-level code splitting
import React, { lazy, Suspense } from 'react';

const AdminPanel = lazy(() => import('./AdminPanel'));
// AdminPanel is not in the initial bundle. It's fetched when first rendered.

function App() {
  return (
    <Suspense fallback={<div>Loading...</div>}>
      <AdminPanel />
    </Suspense>
  );
}

// Route-based splitting (most impactful split point)
// In React Router:
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Settings  = lazy(() => import('./pages/Settings'));
const Profile   = lazy(() => import('./pages/Profile'));
// Each route is a separate chunk downloaded only when navigated to

Tree Shaking — Dead Code Elimination

// Tree shaking only works with ES module syntax (import/export)
// It does NOT work with CommonJS (require/module.exports)

// BAD: imports entire lodash library (~70KB gzipped)
const _ = require('lodash');
const result = _.debounce(fn, 300);

// GOOD: import only what you use — bundler eliminates the rest
import { debounce } from 'lodash-es';  // lodash-es is the ES module version
const result = debounce(fn, 300);

// Even better for small utilities: write your own or use a tiny alternative
const debounce = (fn, delay) => {
  let timeout;
  return (...args) => {
    clearTimeout(timeout);
    timeout = setTimeout(() => fn(...args), delay);
  };
};

// Bundle analysis — find what's actually in your bundle
// Run: npx webpack-bundle-analyzer stats.json
// Or with Vite: npx vite-bundle-visualizer
// This shows you a treemap of every module's size contribution

Memoization in a Performance Context

We covered memoization in Chapter 12 from a React perspective. In a performance interview, the framing shifts: memoization prevents expensive re-renders and recalculations that would otherwise block the main thread.

// React.memo: prevent re-render if props haven't changed
const ExpensiveList = React.memo(({ items, onSelect }) => {
  // This component only re-renders when items or onSelect reference changes
  return items.map(item => <Row key={item.id} item={item} onSelect={onSelect} />);
});

// useMemo: cache expensive computed values
function DataGrid({ rawData, filters }) {
  const processedData = useMemo(() => {
    // This runs only when rawData or filters changes
    // Without useMemo, it runs on every render — potentially 60x per second during animations
    return rawData
      .filter(row => matchesFilters(row, filters))
      .sort((a, b) => a.value - b.value)
      .slice(0, 100);
  }, [rawData, filters]);

  return <Grid data={processedData} />;
}

// useCallback: stable function reference for memoized children
function Parent({ items }) {
  const handleSelect = useCallback((id) => {
    setSelectedId(id);
  }, []); // stable reference — ExpensiveList won't re-render due to this prop

  return <ExpensiveList items={items} onSelect={handleSelect} />;
}

Network Performance

HTTP/2 vs HTTP/1.1

HTTP/1.1 is serial: browsers open multiple TCP connections (typically 6 per host) and send one request per connection. Requests queue behind each other — this is called head-of-line blocking. HTTP/2 is multiplexed: a single TCP connection carries multiple requests simultaneously, interleaved in frames. You can make 100 requests and get responses in parallel over one connection.

Feature HTTP/1.1 HTTP/2
Requests per connection 1 (queued) Many (multiplexed)
Header compression No Yes (HPACK)
Server push No Yes (rarely used)
Text vs binary Text Binary frames
Implication for perf Bundle aggressively, fewer requests Many small files can be fine

Resource Hints

<!-- preconnect: establish TCP/TLS connection to an origin early
     Use for: API servers, CDNs, analytics, font providers -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://api.example.com" crossorigin>

<!-- preload: fetch a specific resource early, at high priority
     Use for: LCP image, critical fonts, critical scripts -->
<link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="/hero.webp" as="image" fetchpriority="high">

<!-- prefetch: fetch a resource you expect to need for the NEXT navigation
     Low priority — happens in idle time, doesn't hurt current page -->
<link rel="prefetch" href="/dashboard" as="document">
<link rel="prefetch" href="/chunk-dashboard.js" as="script">

<!-- dns-prefetch: resolve DNS for an origin — cheaper than preconnect -->
<link rel="dns-prefetch" href="https://third-party-analytics.com">

Caching Strategy

// Cache-Control header patterns
// Served from your server (Node/Express example)

// Static assets with content hash in filename (e.g., main.a3f9b2c1.js)
// These can be cached forever — if the file changes, the hash changes
res.setHeader('Cache-Control', 'public, max-age=31536000, immutable');
// max-age=31536000 = 1 year in seconds
// immutable: tells the browser never to revalidate (no conditional requests)

// HTML files — should not be cached aggressively (they reference the hashed assets)
res.setHeader('Cache-Control', 'no-cache');
// no-cache doesn't mean "don't cache" — it means "validate before using cache"
// The browser still caches it, but checks the server first (using ETag or Last-Modified)
// If unchanged, server returns 304 Not Modified with no body — very fast

// API responses — usually short-lived or private
res.setHeader('Cache-Control', 'private, max-age=60');   // user-specific, 1 minute
res.setHeader('Cache-Control', 'no-store');               // never cache (sensitive data)

// ETag workflow:
// Server sends: ETag: "abc123"
// Browser caches with this tag
// Next request: browser sends If-None-Match: "abc123"
// If unchanged: server returns 304 (just headers, no body) — huge bandwidth saving
// If changed: server returns 200 with new body and new ETag

Image Optimization

<!-- Format comparison (same visual quality):
  JPEG:  100KB (established, good support)
  WebP:  65KB  (25-35% smaller than JPEG, 95%+ browser support)
  AVIF:  45KB  (40-50% smaller than JPEG, modern browsers only)
  Strategy: serve AVIF with WebP fallback with JPEG ultimate fallback -->

<picture>
  <source srcset="image.avif" type="image/avif">
  <source srcset="image.webp" type="image/webp">
  <img src="image.jpg" alt="Description" width="800" height="600"
       loading="lazy" decoding="async">
</picture>

<!-- loading="lazy": defers loading until near the viewport — use for everything below the fold
     decoding="async": decodes the image off the main thread — minor but free win
     Never use loading="lazy" on above-the-fold images — it delays LCP -->

DevTools Profiling — Practical Walkthrough

Knowing that performance is important is table stakes. Knowing how to find the problem is the skill that gets you hired. Here's how to use Chrome DevTools for each type of investigation.

Performance Panel — Reading a Flame Chart

// Step 1: Open DevTools → Performance tab
// Step 2: Click record, perform the interaction you're investigating, stop
// Step 3: Read the flame chart

// The flame chart shows call stacks over time:
// - Horizontal axis = time
// - Vertical axis = call stack (top of stack at top)
// - Width of a bar = how long that function ran
// - Long, wide bars at the bottom of the stack = expensive work
// - Look for red triangles = browser's warning for long tasks

// What to look for:
// 1. Long tasks (red) — anything over 50ms
// 2. The widest bars — those are your bottlenecks
// 3. "Recalculate Style" and "Layout" entries — CSS-triggered work
// 4. "Evaluate Script" entries — JavaScript parse/compile cost

// Practical script to mark your own code in the timeline:
performance.mark('component-render-start');
// ... render work ...
performance.mark('component-render-end');
performance.measure('component-render', 'component-render-start', 'component-render-end');
// Your measure now appears as a labeled bar in the Performance panel

Network Panel — Finding Render-Blocking Resources

// In the Network panel waterfall view:
// - Each row = one resource
// - Blue bar = Time to First Byte (server processing + network)
// - Gray bar = queued/blocked time
// - Resources with "Render Blocking" in the Type column are critical

// Common render-blocking patterns:
// 1. Large CSS files in <head> without media queries
// 2. <script> tags without async or defer
// 3. Synchronous third-party scripts (analytics, chat widgets)

// Fixes in HTML:
// CSS that's not needed immediately:
// <link rel="stylesheet" href="print.css" media="print">  — not render-blocking

// JavaScript loading strategies:
// <script src="app.js"></script>            — blocks HTML parsing (worst)
// <script src="app.js" defer></script>      — parsed in parallel, runs after HTML parsed
// <script src="app.js" async></script>      — parsed in parallel, runs as soon as ready
// Use defer for most scripts. Use async only for fully independent scripts (analytics).

Memory Panel — Detecting Memory Leaks

// Step 1: Open DevTools → Memory tab
// Step 2: Take a heap snapshot (before the suspected leak)
// Step 3: Perform the action that might leak (navigate away and back, etc.)
// Step 4: Take another heap snapshot
// Step 5: Change the view to "Comparison" and look for objects that increased

// What to look for in the snapshot:
// - "Detached HTMLElement" — DOM nodes held in memory after removal
// - Closures with large retained sizes
// - Event listeners that accumulated

// Programmatic memory monitoring:
performance.measureUserAgentSpecificMemory().then(memory => {
  console.log('JS heap:', memory.bytes / 1024 / 1024, 'MB');
  // Requires cross-origin isolation headers to use
});

// Simple leak detection:
const startHeap = performance.memory?.usedJSHeapSize;
// ... perform operation ...
const endHeap = performance.memory?.usedJSHeapSize;
console.log('Memory delta:', (endHeap - startHeap) / 1024, 'KB');
// Note: performance.memory is Chrome-only and not standardized

Coverage Tool — Unused CSS and JS

// Open DevTools → More Tools → Coverage (Ctrl+Shift+P → "coverage")
// Click record, interact with the page, stop
// The tool shows each file with a bar indicating used (green) vs unused (red) bytes

// Typical finding: 60-80% of CSS in many apps is never used on load
// Action: use PurgeCSS or the equivalent for your framework to eliminate unused CSS

// For unused JavaScript:
// - Code splitting (covered above) is the systematic fix
// - The coverage tool shows you which functions were never called — candidates for lazy loading

// Example PurgeCSS config (PostCSS plugin):
// purgecss({
//   content: ['./src/**/*.html', './src/**/*.jsx', './src/**/*.tsx'],
//   defaultExtractor: content => content.match(/[\w-/:]+(?<!:)/g) || []
// })
// This scans your HTML/JS and removes any CSS class not found in those files

Frontend Memory Leaks

Memory leaks in frontend applications are a class of bug where JavaScript holds references to objects that should have been garbage collected. The application grows in memory over time, eventually causing slowness, janky animations, or a browser tab crash.

Cause 1: Detached DOM Nodes (Most Common)

// The classic detached DOM leak:
let detachedNode;

function createLeak() {
  const div = document.createElement('div');
  document.body.appendChild(div);
  detachedNode = div;  // We store a reference...
  document.body.removeChild(div); // ...then remove it from the DOM
  // The div is gone from the page, but detachedNode still holds a reference
  // The garbage collector cannot free it — it's a "detached DOM node"
}

// Fix: null out references when you're done
function noLeak() {
  const div = document.createElement('div');
  document.body.appendChild(div);
  // ... use div ...
  document.body.removeChild(div);
  // No stored reference — garbage collector can free the memory
}

// In React: this is mostly handled for you.
// But be careful with imperative refs stored in module scope or closures.

Cause 2: Event Listeners Never Removed

// Covered in Chapter 12 — a brief performance-framing recap:

// LEAK: the DOM element is removed but the handler still references it
class Component {
  constructor() {
    this.handleResize = () => this.onResize(); // bound method
    window.addEventListener('resize', this.handleResize); // registered
  }
  destroy() {
    // Component is destroyed but listener is still attached to window
    // window.removeEventListener is never called — memory leak
  }
}

// FIX: always pair addEventListener with removeEventListener
class Component {
  constructor() {
    this.handleResize = () => this.onResize();
    window.addEventListener('resize', this.handleResize);
  }
  destroy() {
    window.removeEventListener('resize', this.handleResize); // clean up
  }
}

// In React: use the useEffect cleanup function
useEffect(() => {
  const handleResize = () => setSize(window.innerWidth);
  window.addEventListener('resize', handleResize);
  return () => window.removeEventListener('resize', handleResize); // cleanup
}, []);

Cause 3: Closures Holding Large Data

// A closure that accidentally retains a large object
function processLargeArray(data) {
  const results = expensiveTransformation(data); // data might be 50MB

  return function getResult(index) {
    // This closure captures the entire 'data' array in its scope,
    // even though it only needs 'results'
    return results[index];
  };
}

// Fix: don't capture more than you need
function processLargeArray(data) {
  const results = expensiveTransformation(data);
  data = null; // explicitly release the reference (won't help if captured in closure)

  // Better: process outside the closure scope
  return (index) => results[index];
  // data is not captured here — only results is
}

Senior vs Junior: Diagnosing a Slow Page

Interviewer: "How would you improve the performance of a slow e-commerce product page?"

Junior Answer

"I'd minimize the JavaScript and compress the images."

This is technically correct but useless. It's like a doctor saying "I'd prescribe medicine" without examining the patient. You haven't measured anything, you don't know what's actually slow, and minification is something your build tool does automatically anyway.

Senior Answer

"I'd start by measuring — never optimize what you haven't measured. I'd run Lighthouse to get a baseline score and identify which Core Web Vital is failing. Then I'd dig into the specific problem:

If LCP is poor, I'd identify the largest element in the viewport. Is it a hero image? I'd switch to WebP/AVIF format, add an explicit preload hint, add explicit dimensions, and make sure it's not accidentally lazy-loaded. If LCP is a text element, I'd look at render-blocking CSS or fonts.

If CLS is poor, I'd look for images without explicit dimensions, late-loading ads or banners pushing content, and web font swaps. I'd add width/height attributes to all images and add font-display: optional or preload the critical font.

If INP is poor, I'd open the Performance panel, record an interaction, and find the long task. Is it JavaScript? I'd check the bundle analyzer for large unnecessary imports. Is it a carousel initialization? I'd defer it until after the user scrolls or interacts. Is it layout thrashing? I'd batch my DOM reads and writes.

I'd also check the Network panel waterfall for render-blocking resources and confirm that static assets are served with long cache headers from a CDN. Then I'd measure again with Lighthouse to verify the improvements and set a performance budget in CI to prevent regressions."

Practice Questions

Beginner Explain Core Web Vitals to a non-technical product manager who wants to know why the team is spending a sprint on "web vitals." What do LCP, CLS, and INP measure in plain English, and why should the business care?

Beginner A team member complains that a CSS animation is "janky" on mobile. Without looking at the code, name the three most likely causes and the one question you'd ask first to narrow it down.

Intermediate Write a custom debounce function that won't block the main thread for expensive scroll handlers. Then explain the difference between debouncing, throttling, and using requestAnimationFrame for scroll events, and when you'd choose each.

Intermediate A React app has a list component that re-renders 200 list items every time the parent state changes. The user reports lag when typing in a search field. Describe a complete fix using the tools covered in this chapter and Chapter 12.

Advanced A single-page application has a memory leak — users report the tab consuming 2GB of RAM after an hour of use. Walk through your entire debugging process: which DevTools panels you'd use, what you'd look for in each, and how you'd isolate the root cause. Then describe how you'd prevent this class of bug in a code review.

Chapter Summary

  • Core Web Vitals: LCP (main content loads fast — <2.5s), CLS (page doesn't shift — <0.1), INP (interactions feel instant — <200ms). These affect SEO and user retention directly.
  • The browser rendering pipeline: Parse HTML → DOM, Parse CSS → CSSOM, Render Tree, Layout (reflow), Paint, Composite. Animations should only touch transform and opacity — they skip to composite and run on the GPU.
  • Long tasks (>50ms) are the enemy of INP. Break them up with the yield-to-main pattern. Code splitting, lazy loading, and tree shaking reduce the amount of JavaScript that runs at all.
  • Network: Use HTTP/2 for multiplexing. Resource hints (preconnect, preload, prefetch) guide the browser's loading priority. Cache static assets forever with content hashes. Serve images in WebP/AVIF from a CDN.
  • DevTools: Performance panel for flame charts and long tasks, Network panel for render-blocking resources, Memory panel for heap snapshots and leak detection, Coverage for unused code.
  • Always measure before you optimize. A senior engineer gives a diagnosis before a prescription.
