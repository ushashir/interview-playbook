CSS, Layout, and Responsive Design

CSS questions have a reputation for being the "easy" part of the frontend interview. Then someone asks you to center a div and you suddenly forget everything you know. CSS is deceptively simple and surprisingly deep.

What you'll learn

  • The box model explained clearly — and why border-box should be your default
  • Five distinct ways to center a div, and when to use each
  • Flexbox and CSS Grid from mental model to production code
  • CSS specificity and how to avoid the wars it creates
  • Mobile-first responsive design with media queries and container queries
  • How senior engineers think about CSS at the design system level

The Box Model

💡 Analogy: The Gift Box

Think of every HTML element as a gift box on a shelf. The content is the actual gift inside — the text, the image, whatever the element contains. The padding is the bubble wrap surrounding the gift, protecting it from the edges. The border is the box itself — the physical container. The margin is the empty space on the shelf between this box and the one next to it. You can't put anything in the margin — it's just breathing room between boxes.

Every element in a browser is a rectangular box. The box model defines how the four layers of that box — content, padding, border, and margin — relate to each other and to the total dimensions of the element. Understanding this model cold is the foundation of every CSS layout problem.

+-----------------------------------------------+ | MARGIN | | +---------------------------------------+ | | | BORDER | | | | +-------------------------------+ | | | | | PADDING | | | | | | +-----------------------+ | | | | | | | | | | | | | | | CONTENT | | | | | | | | (width × height) | | | | | | | | | | | | | | | +-----------------------+ | | | | | | | | | | | +-------------------------------+ | | | | | | | +---------------------------------------+ | | | +-----------------------------------------------+

box-sizing: content-box vs border-box

This is where the model gets confusing and where interviews often expose gaps. By default, browsers use content-box, which means when you set width: 200px, that's the width of the content only. Padding and border are added on top. So a box with width: 200px; padding: 20px; border: 2px solid is actually 244px wide on screen — and that surprises every developer who hasn't internalized this.

border-box changes the math so that width: 200px includes padding and border. The content area shrinks to accommodate them, but the total box stays 200px. This is what your brain intuitively expects when you write a width. This is why every serious codebase applies a global reset for it.

/* The standard box-sizing reset — put this at the top of every stylesheet */
*, *::before, *::after {
  box-sizing: border-box;
}

/* Demonstration of why this matters */
.content-box-example {
  box-sizing: content-box; /* default — you almost never want this */
  width: 200px;
  padding: 20px;
  border: 2px solid #333;
  /* Actual rendered width: 200 + 20 + 20 + 2 + 2 = 244px */
}

.border-box-example {
  box-sizing: border-box; /* what you actually want */
  width: 200px;
  padding: 20px;
  border: 2px solid #333;
  /* Actual rendered width: exactly 200px */
  /* Content area shrinks to: 200 - 40 - 4 = 156px */
}

The only time you'd ever consciously use content-box is if you're working in a legacy codebase that wasn't reset and you need to match existing behavior. In all new work: always use border-box.

Centering — Five Ways to Center a Div

This is one of the most reliably asked CSS interview questions. The reason interviewers love it is that there are genuinely multiple correct answers, each with different tradeoffs, and a candidate who knows all five has clearly spent time in the trenches. Knowing only one method tells the interviewer you've solved it once and moved on.

Method 1: Flexbox (Most Common, Most Versatile)

.container {
  display: flex;
  justify-content: center;   /* horizontal centering */
  align-items: center;       /* vertical centering */
  height: 100vh;             /* container must have height for vertical to work */
}

/* The child is centered both horizontally and vertically */
.centered-child {
  width: 300px;
  padding: 20px;
}

This is what you reach for by default. Readable, well-supported everywhere, easy to understand at a glance. Use this unless you have a specific reason not to.

Method 2: CSS Grid (Cleanest Syntax)

.container {
  display: grid;
  place-items: center;  /* shorthand for align-items + justify-items */
  height: 100vh;
}

/* place-items: center is the cleanest single-line centering solution in CSS */

place-items: center is a shorthand that sets both align-items and justify-items to center in a single declaration. It's the shortest correct answer to "center something in its container." It only works inside a grid context.

Method 3: Absolute Positioning + Transform

.container {
  position: relative;
}

.centered-child {
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%);
  /* top:50% and left:50% move the top-left corner to center */
  /* translate(-50%, -50%) shifts the element back by half its own width/height */
}

The trick here is subtle: top: 50% moves the element's top edge to the center, not the element's center. The transform pulls it back by half its own dimensions. This works without knowing the container's dimensions and without display: flex — useful for overlays, modals, and tooltips where the parent is constrained.

Method 4: margin: auto (Horizontal Only)

.centered-child {
  width: 300px;         /* must have an explicit width */
  margin: 0 auto;       /* auto margins distribute leftover horizontal space equally */
}

/* This does NOT vertically center. For vertical centering, you need flexbox or grid. */

This is the oldest method — works since CSS2. It only centers horizontally. The element must have an explicit width, otherwise it stretches to fill the container and there's no leftover space for the auto margins to split. Good for centering a content column on a page.

Method 5: text-align: center (Inline/Inline-Block Only)

.container {
  text-align: center;
}

/* Works for: */
/* - Text content */
/* - Inline elements (span, a, img by default) */
/* - inline-block elements */

.inline-block-child {
  display: inline-block;
  /* Now text-align: center on the parent will center this */
}

/* Does NOT work for block-level elements with a set width */

This one has the narrowest application. It centers text and inline/inline-block children. If a candidate says "use text-align: center" as their only answer to centering a div, that's a red flag — it only works in specific circumstances and it's inherited, which can cause unintended side effects down the tree.

Flexbox

💡 Analogy: The Factory Conveyor Belt

Imagine a factory conveyor belt. The flex container is the belt itself. Items sit on the belt and move along it. You control: which direction does the belt run (left-to-right? top-to-bottom?)? Do items grow to fill available space if there's room? Do they shrink if they're too big? How is the extra space at the end of the belt distributed — all at one end, spread between items, or equal on both sides? Flexbox gives you fine-grained control over all of these decisions.

Flexbox is a one-dimensional layout system — it works along a single axis at a time (either a row or a column). It excels at distributing space and aligning items in a line. For two-dimensional layouts, you want Grid.

Core Properties

/* === CONTAINER PROPERTIES === */
.flex-container {
  display: flex;

  /* Direction of the main axis */
  flex-direction: row;            /* row (default) | row-reverse | column | column-reverse */

  /* Distribution along the main axis */
  justify-content: flex-start;   /* flex-start | flex-end | center | space-between | space-around | space-evenly */

  /* Alignment along the cross axis */
  align-items: stretch;          /* stretch (default) | flex-start | flex-end | center | baseline */

  /* What happens when items overflow the container */
  flex-wrap: nowrap;             /* nowrap (default) | wrap | wrap-reverse */

  /* When wrapping occurs: aligns rows along cross axis */
  align-content: flex-start;    /* same values as justify-content */
}

/* === ITEM PROPERTIES === */
.flex-item {
  /* Grow factor: how much does this item expand to fill free space? */
  flex-grow: 0;     /* 0 = don't grow (default). 1 = take all free space. */

  /* Shrink factor: how much does this item shrink when space is tight? */
  flex-shrink: 1;   /* 1 = shrink proportionally (default). 0 = never shrink. */

  /* Base size before growing/shrinking */
  flex-basis: auto; /* auto = use the item's natural width/height */

  /* Shorthand: flex: grow shrink basis */
  flex: 1;          /* flex: 1 1 0% — grow, shrink, and start from 0 */
  flex: none;       /* flex: 0 0 auto — fixed size, never grow or shrink */
}
⚠️ Common Gotcha: align-items vs align-content

align-items aligns items within a single row (the cross-axis). align-content only activates when flex-wrap: wrap is set and there are multiple rows — it aligns those rows within the container. Using align-content when there's only one row of items does nothing. This trips up a lot of candidates.

⚠️ Common Gotcha: flex-basis overrides width

If flex-basis is set to anything other than auto, it overrides the width property on a flex item. Setting width: 200px on a flex item that also has flex-basis: 0 means the width is ignored. This is one of the most confusing behaviors in flexbox.

Real Code: Navigation Bar with Flexbox

/* Responsive navigation bar — logo on left, links on right */
.navbar {
  display: flex;
  justify-content: space-between;  /* pushes logo left, links right */
  align-items: center;             /* vertically centers both */
  padding: 0 24px;
  height: 64px;
  background: #1a1a2e;
}

.navbar__logo {
  font-size: 1.25rem;
  font-weight: 700;
  color: #fff;
  text-decoration: none;
}

.navbar__links {
  display: flex;
  gap: 24px;         /* modern way to add space between flex items */
  list-style: none;
  margin: 0;
  padding: 0;
}

.navbar__links a {
  color: rgba(255, 255, 255, 0.8);
  text-decoration: none;
  font-size: 0.9rem;
  transition: color 0.2s;
}

.navbar__links a:hover {
  color: #fff;
}

Real Code: Card Layout with Flexbox

/* A row of cards that wrap to the next line when they don't fit */
.card-grid {
  display: flex;
  flex-wrap: wrap;
  gap: 16px;
  padding: 24px;
}

.card {
  flex: 1 1 280px;   /* grow, shrink, base width of 280px */
  /* Items will stretch to fill row, but won't shrink below 280px */
  /* On wide screens: 3-4 cards per row */
  /* On narrow screens: 1 card per row */

  background: #fff;
  border-radius: 8px;
  padding: 20px;
  box-shadow: 0 2px 8px rgba(0,0,0,0.08);
  display: flex;
  flex-direction: column;
}

.card__body {
  flex: 1;   /* takes all remaining space, pushing footer to the bottom */
}

.card__footer {
  margin-top: 16px;
  padding-top: 16px;
  border-top: 1px solid #eee;
}

CSS Grid

💡 Analogy: The Spreadsheet

CSS Grid works exactly like a spreadsheet. You define how many rows and columns you want, and their sizes. Then you place items into specific cells, or let them flow automatically from left to right, top to bottom. A cell can span multiple rows or columns — just like a merged cell in Excel. You're thinking in two dimensions from the start, which is the fundamental difference from flexbox.

Grid is the right tool when you have a two-dimensional layout — rows and columns at the same time. Flexbox is one-dimensional (you pick a direction). Grid is two-dimensional. A good mental model: use Grid for page-level layout and major structural sections, use Flexbox for the content inside those sections.

Core Properties

.grid-container {
  display: grid;

  /* Define columns — fixed sizes, percentages, or fractions */
  grid-template-columns: 200px 1fr 1fr;
  /* Three columns: first is 200px, second and third split the remaining space equally */

  /* The fr unit: fractional unit — takes up a fraction of available space */
  grid-template-columns: repeat(3, 1fr);     /* 3 equal columns */
  grid-template-columns: 1fr 2fr 1fr;        /* middle column is twice as wide */

  /* Define rows */
  grid-template-rows: auto 1fr auto;         /* header, main (stretches), footer */

  /* Gap between cells */
  gap: 16px;                /* shorthand: row-gap and column-gap together */
  row-gap: 16px;
  column-gap: 24px;

  /* Named grid areas */
  grid-template-areas:
    "header header header"
    "sidebar main main"
    "footer footer footer";
}

/* Place items using named areas */
.header  { grid-area: header; }
.sidebar { grid-area: sidebar; }
.main    { grid-area: main; }
.footer  { grid-area: footer; }

/* Or place items by line numbers */
.item {
  grid-column: 1 / 3;    /* starts at column line 1, ends at line 3 (spans 2 columns) */
  grid-row: 2 / 4;       /* starts at row line 2, ends at line 4 */

  /* Shorthand */
  grid-column: span 2;   /* span 2 columns from wherever the item naturally lands */
}

auto-fill vs auto-fit

Both work with repeat() to create columns without specifying a fixed count. The difference matters for empty space when there are fewer items than columns.

/* auto-fill: fills the row with as many columns as will fit, even if empty */
.grid-fill {
  grid-template-columns: repeat(auto-fill, minmax(250px, 1fr));
  /* Empty column tracks still take up space */
}

/* auto-fit: collapses empty column tracks to zero width */
.grid-fit {
  grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
  /* Items stretch to fill the full row — usually what you want */
}

Real Code: Responsive Card Grid (1 → 3 Columns)

/* A responsive grid that adapts from 1 column (mobile) to 3 (desktop)
   WITHOUT any media queries — pure grid magic */
.responsive-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(280px, 1fr));
  /* auto-fit: use as many columns as will fit
     minmax(280px, 1fr): each column is at least 280px, max is 1 equal fraction
     At 320px viewport: 1 column
     At 620px viewport: 2 columns
     At 900px+ viewport: 3 columns */
  gap: 20px;
  padding: 20px;
}

.card {
  background: #fff;
  border-radius: 8px;
  overflow: hidden;
  box-shadow: 0 2px 12px rgba(0,0,0,0.08);
  transition: transform 0.2s, box-shadow 0.2s;
}

.card:hover {
  transform: translateY(-4px);
  box-shadow: 0 8px 24px rgba(0,0,0,0.12);
}

.card__image {
  width: 100%;
  aspect-ratio: 16 / 9;
  object-fit: cover;
}

.card__content {
  padding: 16px;
}

.card__title {
  margin: 0 0 8px;
  font-size: 1.1rem;
  font-weight: 600;
}

.card__description {
  margin: 0;
  color: #666;
  font-size: 0.9rem;
  line-height: 1.5;
}

That single grid declaration replaces what used to require three separate media queries. This is one of the most powerful patterns in modern CSS and shows up in real production code constantly.

CSS Specificity

💡 Analogy: Military Rank

Think of CSS rules as soldiers in an army. A general outranks a captain, who outranks a sergeant, who outranks a private. When two soldiers give conflicting orders to the same element, the higher-ranking one is obeyed. If they're the same rank, the one who spoke most recently (later in the stylesheet) wins. Specificity is the ranking system for CSS rules.

The Specificity Score

Every CSS selector has a specificity score calculated as a three-part number: (A, B, C).

Selector Type Score Examples
Inline style (1, 0, 0) = 1000 style="color: red"
ID selector (0, 1, 0) = 100 #header, #nav
Class, attribute, pseudo-class (0, 0, 1) = 10 .btn, [type="text"], :hover
Element, pseudo-element (0, 0, 1) = 1 div, p, ::before
Universal selector, combinators 0 *, >, +, ~

The scores are compared left to right, not summed into a single number. (0, 1, 0) — a single ID — always beats (0, 0, 99) — 99 classes — because the first non-zero digit from the left wins.

/* Specificity examples — which one wins? */

p { color: black; }                     /* (0,0,1) = 1 */
.text { color: blue; }                  /* (0,1,0) = 10 — wins over p */
#intro .text { color: green; }          /* (1,1,0) = 110 — wins over .text */
style="color: red"                      /* (1,0,0,0) = 1000 — wins over everything */

/* The cascade: when specificity is equal, the later rule wins */
.btn { background: blue; }
.btn { background: red; }   /* This wins — same specificity, comes later */

!important — The Nuclear Option

/* !important overrides ALL specificity calculations */
.button {
  color: blue !important;   /* This wins, even over inline styles */
}

/* Why you should avoid it:
   - It breaks the cascade — the predictable algorithm CSS relies on
   - Once used, the only way to override it is another !important
   - This creates an arms race of increasing specificity
   - It makes debugging CSS extremely painful
   - Legitimate uses: user stylesheets for accessibility, utility class overrides in design systems */

The problem with !important is not that it's wrong — it's that it's addictive. You use it once to fix something quickly, then you need another one to override that, and six months later your stylesheet is incomprehensible. Design your selectors to have predictable, minimal specificity instead.

The Common Mistake: Over-Specific Selectors

/* BAD: overly specific, impossible to reuse, hard to override */
div#main-content .article-container > ul.article-list li.list-item a.link {
  color: blue;
}

/* GOOD: minimal, predictable, easy to work with */
.article-link {
  color: blue;
}

The BEM naming convention (covered in the interview Q&A section below) exists precisely to solve this problem by keeping everything at class-level specificity.

Responsive Design

Responsive design means your interface works on any screen size — from a 320px phone to a 5K monitor — without separate codebases or server-side user agent detection. It's not optional on any project made in the last decade.

Mobile-First vs Desktop-First

Always recommend mobile-first. Here's why it's not just a preference — it's a technical advantage.

/* MOBILE-FIRST: start with mobile styles, add complexity for larger screens */
/* Uses min-width media queries */

/* Base styles — what every device gets */
.container {
  padding: 16px;
  font-size: 1rem;
}

.card-grid {
  display: grid;
  grid-template-columns: 1fr;   /* 1 column on mobile */
  gap: 16px;
}

/* Tablet and up (min-width: 768px means "768px OR WIDER") */
@media (min-width: 768px) {
  .container {
    padding: 24px;
  }
  .card-grid {
    grid-template-columns: repeat(2, 1fr);
  }
}

/* Desktop and up */
@media (min-width: 1024px) {
  .container {
    max-width: 1200px;
    margin: 0 auto;
    padding: 32px;
  }
  .card-grid {
    grid-template-columns: repeat(3, 1fr);
  }
}

/* =================================================== */

/* DESKTOP-FIRST (avoid this): start with desktop, override for small screens */
/* Uses max-width — more overrides, more specificity fights */
.card-grid {
  grid-template-columns: repeat(3, 1fr);
}

@media (max-width: 1023px) {
  .card-grid { grid-template-columns: repeat(2, 1fr); }
}

@media (max-width: 767px) {
  .card-grid { grid-template-columns: 1fr; }
}

Mobile-first forces you to think about the essential experience first. You're not stripping things away for mobile — you're adding things for desktop. That's architecturally cleaner.

Common Breakpoints

/* Industry-standard breakpoints (match Tailwind's defaults — they're widely adopted) */

/* Mobile-first breakpoints */
/* 0px - 639px: mobile (base styles, no media query) */
@media (min-width: 640px)  { /* sm: landscape phone / large phone */ }
@media (min-width: 768px)  { /* md: tablet */ }
@media (min-width: 1024px) { /* lg: laptop */ }
@media (min-width: 1280px) { /* xl: desktop */ }
@media (min-width: 1536px) { /* 2xl: large desktop */ }

Viewport Units and the svh Problem

/* Traditional viewport units */
.hero {
  height: 100vh;    /* 100% of viewport height */
  width: 100vw;     /* 100% of viewport width */
}

/* The problem: on mobile browsers, 100vh includes the browser chrome (address bar).
   The address bar hides when you scroll, which causes layout shifts.
   The fix: use svh (small viewport height) — the viewport height with chrome visible */

.hero {
  height: 100svh;   /* safe: uses the smaller of the two viewport measurements */
}

/* Full set of dynamic viewport units (modern CSS) */
/* svh = small viewport height (browser chrome visible) */
/* lvh = large viewport height (browser chrome hidden/scrolled) */
/* dvh = dynamic viewport height (changes as chrome shows/hides — use carefully) */

/* Safe default for full-screen sections on mobile: */
.full-screen-section {
  min-height: 100svh;
}

Responsive Images: srcset and sizes

<!-- Don't serve a 2400px image to a 375px phone screen -->

<img
  src="hero-800.jpg"
  srcset="
    hero-400.jpg   400w,
    hero-800.jpg   800w,
    hero-1200.jpg 1200w,
    hero-2400.jpg 2400w
  "
  sizes="
    (max-width: 600px) 100vw,
    (max-width: 1200px) 50vw,
    800px
  "
  alt="Hero image"
  loading="lazy"
  width="800"
  height="450"
/>
<!--
  srcset: tells the browser which image files are available and their widths
  sizes:  tells the browser how wide the image will be displayed at each breakpoint
  The browser picks the best file. It accounts for device pixel ratio automatically.

  Always include explicit width and height attributes to prevent layout shift (CLS).
  loading="lazy" defers off-screen images — huge performance win.
-->

CSS Container Queries — The Modern Alternative

Media queries respond to the viewport size. Container queries respond to the size of a specific element's container. This is a fundamental shift: components can now be truly responsive to where they're placed, not just the overall screen size.

/* Define a containment context */
.card-wrapper {
  container-type: inline-size;
  container-name: card;   /* optional, for named queries */
}

/* Style the card based on its container's width, not the viewport */
.card {
  padding: 12px;
  display: block;
}

@container card (min-width: 400px) {
  .card {
    display: flex;      /* Switch to horizontal layout when container is wide enough */
    gap: 16px;
    padding: 20px;
  }

  .card__image {
    flex-shrink: 0;
    width: 160px;
  }
}

/* Why this matters: the same .card component can now be placed in a sidebar
   (narrow container, stacked layout) OR in a main content area (wide container,
   horizontal layout) — without any JavaScript or separate component variants. */

Container queries have broad browser support as of 2023. For new projects, they should be your first instinct for component-level responsiveness. Media queries remain useful for page-level layout and global typography scaling.

Common Interview Questions

Q: "What is BEM and why would you use it?"

Strong Answer

BEM stands for Block, Element, Modifier. It's a CSS naming convention that prevents the two most common CSS problems: specificity wars and name collisions.

  • Block is the standalone component: .card, .navbar, .button
  • Element is a child of the block: .card__title, .card__image, .navbar__logo
  • Modifier is a variant: .card--featured, .button--disabled, .button--large
<!-- BEM HTML structure -->
<div class="card card--featured">
  <img class="card__image" src="..." alt="...">
  <div class="card__body">
    <h2 class="card__title">Title</h2>
    <p class="card__description">...</p>
  </div>
  <div class="card__footer">
    <button class="button button--primary">Read more</button>
  </div>
</div>

Every class is at the same specificity level — a single class selector. This means overriding styles is always predictable. You never need !important. The naming makes structure self-documenting. In a codebase with 20 developers, BEM means you can look at a class name and immediately know what it does and where it belongs.

Q: "display:none vs visibility:hidden vs opacity:0 — what's the difference?"

Strong Answer
Property Visible? Occupies space? Interactable? Transitions?
display: none No No — removed from layout flow entirely No No — can't animate to/from none
visibility: hidden No Yes — still takes up space No Yes (discrete — instant snap)
opacity: 0 No Yes — still takes up space Yes — can still be clicked! Yes — smooth fade

The critical gotcha with opacity: 0 is that the element is invisible but still interactive. This is a common accessibility bug — a transparent button that users can accidentally click. When fading out interactive elements, combine opacity: 0 with pointer-events: none to disable clicks. Or transition to visibility: hidden after the opacity transition completes.

Q: "What is a CSS preprocessor and when would you use one?"

Strong Answer

CSS preprocessors like Sass/SCSS compile a superset of CSS into regular CSS. They add variables, nesting, mixins, functions, and modules — things CSS historically lacked.

// SCSS example
$primary: #3b82f6;
$spacing-unit: 8px;

@mixin flex-center {
  display: flex;
  justify-content: center;
  align-items: center;
}

.button {
  @include flex-center;
  padding: $spacing-unit * 2 $spacing-unit * 4;
  background: $primary;

  &:hover {
    background: darken($primary, 10%);
  }

  &--large {
    padding: $spacing-unit * 3 $spacing-unit * 6;
  }
}

The honest answer in 2025: modern CSS has caught up significantly. CSS custom properties replace variables, native CSS nesting is now standard, and @layer handles cascade management. If you're starting a new project, you may not need a preprocessor at all. Where preprocessors still shine is in design systems with complex token hierarchies, existing large codebases, or when you need true compile-time functions like darken() and lighten() that don't have CSS equivalents. PostCSS with targeted plugins is often a better choice than a full preprocessor for new work.

Senior vs Junior: Writing CSS for a Design System

Interviewer: "How do you approach writing CSS for a design system that 20 developers will use?"

Junior Answer

"I'd write reusable classes and put them in a shared stylesheet that everyone imports."

Senior Answer

"I'd start with design tokens — CSS custom properties that are the single source of truth for every visual decision: colors, spacing scale, typography scale, border radii, shadow levels, animation durations. Every number in the system traces back to a token.

/* design-tokens.css */
:root {
  /* Color palette */
  --color-brand-50: #eff6ff;

On top of tokens, build a small set of utility classes for spacing, display, and text alignment — the things you reach for constantly. Component-level styles are scoped through CSS Modules or a CSS-in-JS solution so there are no name collisions between teams.

The goal is a system where adding a new component is predictable — a developer can look at existing components and immediately know how to style a new one. And deleting a component doesn't leave orphaned global styles because the styles live with the component. I'd also add a lint rule that bans hardcoded values in component files — every color and spacing value must reference a token."

Chapter Summary

  • The box model has four layers: content, padding, border, margin. Always use box-sizing: border-box so that width means what you think it means.
  • There are five ways to center a div. Flexbox with justify-content: center; align-items: center is your default. Grid with place-items: center is cleanest. Know all five.
  • Flexbox is one-dimensional (either a row or a column). Grid is two-dimensional. Use Grid for page structure and major sections, Flexbox for the content within them.
  • Specificity is a three-part score: inline (1,0,0), ID (0,1,0), class/attribute/pseudo-class (0,0,1,0), element (0,0,1). Higher rank always beats lower rank regardless of count. Avoid !important.
  • Mobile-first responsive design with min-width media queries is the correct default. Add svh for full-screen sections on mobile. Use container queries for component-level responsiveness.
  • Senior CSS means design tokens, scoped component styles, and a predictable specificity strategy — not just writing classes.
