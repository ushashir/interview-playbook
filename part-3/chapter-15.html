Frontend System Design

Frontend system design is the rarest skill and the one that separates $90k engineers from $200k engineers. Most developers never get asked it. The ones who can answer it confidently get the offers.

What you'll learn

  • A repeatable 8-step framework for any frontend system design question
  • Three full case studies: design system, analytics dashboard, infinite scroll feed
  • Micro-frontend architecture — when it's right and why it's usually wrong
  • Real-time strategies: polling, WebSockets, and Server-Sent Events with a decision framework
  • How to answer the hardest frontend design question: a collaborative text editor
💡 The Big Idea: Designing a Shopping Mall vs Designing a Shop

A single shop is one component — you think about the layout inside, the products, the counter. A shopping mall is the architecture. You think about how the shops relate to each other, how customers navigate from one to the next, where the utilities run (power, water, ventilation), how deliveries happen without disrupting shoppers, how a new tenant moves in without shutting down the whole building. Frontend system design is thinking at the mall level. Individual components are shops. The architecture is the mall. Both matter, but they are completely different disciplines — and the mall-level thinking is rarer and more valuable.

The Framework: How to Approach a Frontend System Design Question

Most candidates treat frontend system design like a coding question with more steps. It's not. It's a conversation about tradeoffs, constraints, and deliberate architectural choices. The goal is not to produce the "right" answer — it's to demonstrate that you think systematically and can justify your decisions.

Use this eight-step framework. You won't use every step every time, but having the structure prevents you from getting lost in implementation details too early.

Step 1: Clarify Scope

Never start designing without knowing the constraints. Ask:

  • Who are the users? Internal employees? Millions of consumers? Developers using an API?
  • What scale? 100 concurrent users or 10 million? Scale changes everything.
  • What devices and browsers? Mobile-first? IE11 support? PWA requirements?
  • What's the team context? One team or twenty? This drives component isolation decisions.
  • What exists already? Greenfield or integrating into a legacy system?

Step 2: Define User Interactions

Before touching architecture, describe what users actually do. Write user stories in plain English. These interactions drive everything downstream — state requirements, network calls, rendering choices, real-time needs.

Step 3: Design the Data Model

What data does the frontend need? Sketch the shape of the data — not database schemas, but the JSON structures your components will consume. This reveals normalization opportunities and catches N+1 fetch problems early.

// Example: data model for a product listing page
interface Product {
  id: string;

interface ProductListPage {

Step 4: Component Architecture

Break the system into components. Start at the page level, decompose into sections, sections into components. Identify which components are shared across the system and which are feature-specific. Draw this on the whiteboard — literally.

ProductListPage ├── PageHeader │ ├── Logo │ └── NavBar ├── FilterSidebar │ ├── SearchInput │ ├── CategoryFilter │ ├── PriceRangeFilter │ └── RatingFilter ├── ProductGrid │ └── ProductCard (×N) │ ├── ProductImage │ ├── ProductTitle │ ├── PriceDisplay │ ├── RatingBadge │ └── AddToCartButton └── Pagination

Step 5: State Management

Where does each piece of state live? Ask these questions for every piece of data:

  • Is it server state (fetched from an API)? Use React Query, SWR, or Apollo.
  • Is it global UI state (shared between unrelated components)? Use Zustand or Redux.
  • Is it local component state (used only in one component)? Use useState.
  • Should it be in the URL (so it survives reload and sharing)? Use the router's query params.

// URL state (survives reload, shareable)
// ?page=2&category=shoes&sort=price-asc&q=running

Step 6: Network Layer

How does the frontend communicate with the backend?

  • REST vs GraphQL: GraphQL shines when different views need different subsets of the same data (prevents over-fetching). REST is simpler for straightforward CRUD. Neither is universally correct.
  • Error handling: Retry strategy, exponential backoff, circuit breakers at the client.
  • Optimistic updates: Update the UI immediately, roll back on failure. Creates perceived responsiveness.
  • Real-time: WebSockets, SSE, or polling? (Covered in depth in the Real-Time section below.)

Step 7: Performance and Optimization

Refer back to Chapter 14. In a design interview, the relevant questions are:

  • What's the initial load strategy? Server-side rendering? Static generation? Client-side?
  • What gets code-split? (Routes, heavy libraries, below-fold sections)
  • What gets cached client-side and for how long?
  • Does any list need virtualization?
  • What are the LCP and INP critical paths?

Step 8: Accessibility and Error States

A design that doesn't account for failure modes and accessibility is not production-ready. Always address:

  • Loading states (skeleton screens, not spinners — more on this in Case Study 3)
  • Error states (what does the user see if the API call fails?)
  • Empty states (what does the user see with no data?)
  • Keyboard navigation and ARIA attributes for screen readers
  • Focus management for dynamic content changes

Case Study 1: Design a Component Library (Design System)

Prompt: "Design a component library that will be used by 20 frontend developers across 5 product teams."

Step 1: Scope Clarification

Before drawing a single box, ask: What framework? (React, Vue, both?) What's the design source of truth — does a design team maintain Figma? Is there a theming requirement (light/dark, white-label)? What's the publishing strategy — npm package, monorepo, CDN? What's the adoption path — greenfield only or migrating legacy code?

Core Components to Design First

Don't try to design 200 components. Start with the atoms — the smallest composable units — and build up.

Primitive components (atoms): Button, Input, Textarea, Select, Checkbox, Radio, Toggle, Badge, Avatar, Icon, Spinner, Tooltip, Tag Composite components (molecules): Modal, Toast/Snackbar, Dropdown, DatePicker, Table, Form, Accordion, Tabs, Card Layout components: Stack, Grid, Container, Divider, Spacer

Case Study 2: Analytics Dashboard with Real-Time Data

Prompt: "Design an analytics dashboard with 10 widgets showing real-time metrics. Users can apply date range filters."

Requirements Breakdown

  • 10 widgets: line charts, bar charts, KPI numbers, data tables
  • Real-time updates (numbers change as events come in)
  • Date range filter affects all widgets simultaneously
  • Each widget can independently fail or load

Data Flow Architecture

// State model: each widget is independent
interface DashboardState {
  dateRange: { start: Date; end: Date };
  widgets: Record<string, WidgetState>;
}

interface WidgetState {
  id: string;
  type: 'line-chart' | 'bar-chart' | 'kpi' | 'table';
  status: 'loading' | 'success' | 'error' | 'stale';
  data: unknown;
  lastUpdated: number;
  error?: string;
}

// Each widget subscribes to its own data slice
// When dateRange changes, all widget queries are invalidated and refetched
// Real-time updates patch only the affected widget's data — no full refetch

// React Query handles this cleanly:
function useWidgetData(widgetId: string, dateRange: DateRange) {
  return useQuery({
    queryKey: ['widget', widgetId, dateRange],
    queryFn: () => fetchWidgetData(widgetId, dateRange),
    refetchOnWindowFocus: true,
    staleTime: 30_000,
  });
}

Real-Time Strategy: WebSockets vs SSE vs Polling

For a metrics dashboard, Server-Sent Events (SSE) is the best fit. The server pushes updates to the client; the client never pushes to the server. WebSockets are bidirectional and are overkill for read-only dashboards.

// SSE connection for real-time metric updates
function useLiveMetrics(widgetIds: string[]) {
  const queryClient = useQueryClient();

  useEffect(() => {
    const url = `/api/metrics/stream?widgets=${widgetIds.join(',')}`;
    const eventSource = new EventSource(url, { withCredentials: true });

    eventSource.addEventListener('metric-update', (event) => {
      const update = JSON.parse(event.data);

      // Patch the specific widget's cached data without a full refetch
      queryClient.setQueryData(
        ['widget', update.widgetId, currentDateRange],
        (oldData: WidgetData) => mergeMetricUpdate(oldData, update)
      );
    });

    eventSource.addEventListener('error', () => {
      // SSE auto-reconnects on error — this is one of its advantages
      console.warn('SSE connection interrupted, reconnecting...');
    });

    return () => eventSource.close(); // cleanup on unmount
  }, [widgetIds.join(',')]);
}

Every Widget Needs Independent States

// A widget that handles all states properly
function Widget({ widgetId }: { widgetId: string }) {
  const { data, status, error, isStale } = useWidgetData(widgetId, dateRange);

  if (status === 'pending') return <WidgetSkeleton />;
  if (status === 'error')   return <WidgetError message={error.message} onRetry={refetch} />;

  return (

      <WidgetContent data={data} />
    </div>
  );
}

// The dashboard renders all widgets regardless of individual states
// If one widget fails, the others still work
function Dashboard() {
  return (
    <div className="dashboard-grid">
      {widgetIds.map(id => <Widget key={id} widgetId={id} />)}
    </div>
  );
}

Case Study 3: Infinite Scroll Feed

Prompt: "Design an infinite scroll feed for a social platform with 1 million+ items."

The Core Problem

You cannot render 1 million DOM nodes. You cannot fetch 1 million items. You cannot hold 1 million items in memory. You need to render only what's visible, fetch only what's nearby, and discard what's far away.

Intersection Observer for Fetch Triggering

// Detect when the user is near the bottom — trigger the next page fetch
function useInfiniteScroll(onLoadMore: () => void, threshold = 0.8) {
  const sentinelRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    const sentinel = sentinelRef.current;
    if (!sentinel) return;

    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting) {
          onLoadMore();
        }
      },
      {
        // rootMargin: start fetching when the sentinel is 200px from viewport bottom
        // This pre-fetches before the user hits the end, making scroll seamless
        rootMargin: '0px 0px 200px 0px',
        threshold: 0,
      }
    );

    observer.observe(sentinel);
    return () => observer.disconnect();
  }, [onLoadMore]);

  return sentinelRef;
}

// React Query's useInfiniteQuery handles the pagination
function useFeed() {
  return useInfiniteQuery({
    queryKey: ['feed'],
    queryFn: ({ pageParam }) => fetchFeedPage(pageParam),
    initialPageParam: null,
    getNextPageParam: (lastPage) => lastPage.nextCursor,
    // cursor-based pagination: more reliable than page numbers for live data
    // (page numbers break when items are inserted at the top)
  });
}

// Putting it together
function Feed() {
  const { data, fetchNextPage, hasNextPage, isFetchingNextPage } = useFeed();
  const sentinelRef = useInfiniteScroll(fetchNextPage);

  const allItems = data?.pages.flatMap(page => page.items) ?? [];

  return (
    <VirtualizedList items={allItems}>
      {/* Sentinel element at the bottom — triggers fetch when visible */}
      <div ref={sentinelRef} style={{ height: 1 }} />
      {isFetchingNextPage && <FeedSkeleton />}
      {!hasNextPage && <EndOfFeed />}
    </VirtualizedList>
  );
}

Virtualization for the Feed

// Problem: after scrolling through 500 items, you have 500 DOM nodes
// Each DOM node costs memory and slows down rendering
// Solution: only render items in or near the viewport

// @tanstack/virtual handles variable-height items (important for social feeds
// where posts can be any height: text-only, image, video, etc.)
import { useWindowVirtualizer } from '@tanstack/react-virtual';

function VirtualizedFeed({ items }: { items: FeedItem[] }) {
  const virtualizer = useWindowVirtualizer({
    count: items.length,
    estimateSize: () => 300,    // rough estimate for initial render
    overscan: 3,                // render 3 items beyond viewport
    // measureElement: built-in measurement corrects estimates after render
  });

  return (
    <div style={{ height: virtualizer.getTotalSize(), position: 'relative' }}>
      {virtualizer.getVirtualItems().map(virtualItem => (
        <div
          key={items[virtualItem.index].id}
          data-index={virtualItem.index}
          ref={virtualizer.measureElement}  // auto-measure actual height
          style={{
            position: 'absolute',
            top: 0,
            left: 0,
            width: '100%',
            transform: `translateY(${virtualItem.start}px)`,
          }}
        >
          <FeedPost item={items[virtualItem.index]} />
        </div>
      ))}
    </div>
  );
}

Skeleton Loaders vs Spinners

Always use skeleton loaders for content-heavy feeds. Spinners say "wait, I'm not ready." Skeletons say "here's where the content will go — it's loading right now." Skeletons reduce perceived load time because the user's brain has already started parsing the layout before the data arrives.

Micro-Frontend Architecture

💡 Analogy: A Shopping Mall Where Each Store Is Independently Owned

In a regular mall, one company owns everything and makes all decisions. In a micro-frontend architecture, each store (feature) is owned by an independent team, built with potentially different technologies, deployed on its own schedule, and only shares the physical building (the shell app). A customer walks the same mall, but each store has its own staff, its own inventory system, its own design. The stores cooperate on basics — shared entrance, shared parking — but operate independently.

What It Is

Micro-frontends extend the microservices philosophy to the frontend. Multiple independent frontend applications are composed into a single user interface. Each "micro-frontend" is owned by a different team, may use different frameworks or versions, is deployed independently, and has its own CI/CD pipeline.

Implementation Approaches

Approach How it works Best for Tradeoffs
Module Federation Webpack 5 shares modules at runtime between separately deployed builds React/Vue apps that need to share components across teams Webpack coupling, version negotiation complexity
iframes Each micro-frontend lives in an iframe Maximum isolation — third-party widgets, legacy apps Poor UX (scroll, focus, URL sync), accessibility nightmares
Web Components Custom HTML elements work across any framework Framework-agnostic shared components Limited ecosystem, SSR complexity
Build-time integration Each team publishes an npm package; shell app imports them Design systems, shared component libraries Requires coordination to release updates; not truly independent

The Hard Problems

// Problem 1: Shared state between micro-frontends
// Each app has its own state store — how do they share user auth, theme, cart?
// Solution: Custom events on the window (simple, framework-agnostic)

// App A fires an event when the user logs in
window.dispatchEvent(new CustomEvent('user:login', {
  detail: { userId: '123', token: 'abc' }
}));

// App B listens for it
window.addEventListener('user:login', (event) => {
  authStore.setUser(event.detail);
});

// This is loose coupling — apps don't import each other, just agree on event shapes

// Problem 2: Routing conflicts
// Each micro-frontend has its own router — they need to not step on each other
// Solution: each app "owns" a URL prefix
// /catalog/* → Catalog team's app
// /checkout/* → Checkout team's app
// /account/* → Account team's app
// Shell app routes to the right micro-frontend based on the path prefix

// Problem 3: Bundle duplication
// If both apps ship React, users download React twice
// Solution with Module Federation:
// Shell app exposes React as a shared singleton
// Micro-frontends declare React as a shared dependency
// Webpack negotiates: only one copy of React is loaded at runtime

When to Use It — Be Honest

⚠️ Honest Tradeoff Assessment

Micro-frontends add significant complexity: build tooling, deployment coordination, shared state protocols, routing negotiation, and bundle management. In most organizations, this is not the right choice. The scenario where it genuinely helps: large enterprise with 10+ teams who would otherwise create coordination bottlenecks on a monolithic frontend. For most companies with 1-5 frontend teams, a well-structured monorepo with clear ownership boundaries solves the same problem with a fraction of the operational overhead. When an interviewer asks about micro-frontends, showing that you understand the tradeoffs — including knowing when not to use them — is more impressive than just describing how to implement them.

// Decision tree: which real-time strategy to use?

function chooseRealTimeStrategy(requirements: {
  clientToServer: boolean;   // does the client need to SEND real-time data?
  highFrequency: boolean;    // updates more than once per second?
  manyUsers: boolean;        // thousands of concurrent connections?
}): 'polling' | 'sse' | 'websocket' {

  // WebSockets: when you need to send data from client to server in real-time
  if (requirements.clientToServer) {
    return 'websocket';
    // Use cases: chat, collaborative editing, multiplayer games, drawing apps
  }

  // SSE: server pushes to client, no client-to-server needed
  // Great for: notification feeds, live metrics, price tickers, activity streams
  if (!requirements.clientToServer) {
    return 'sse';
    // SSE is underused — it's simpler than WebSockets for most real-time needs
  }

  // Polling: when you can't or don't want persistent connections
  // Use cases: simple dashboards, low-frequency updates, edge cases where
  // persistent connections are restricted by the environment
  return 'polling';
}

// Examples:
// Chat application: WebSocket (bidirectional — you send and receive messages)
// Notification bell: SSE (server pushes notifications, client just receives)
// Live sports scores: SSE (read-only, high frequency from server)
// Collaborative text editor: WebSocket (both sides send changes)
// Dashboard refreshing every 30s: polling (simple, low frequency, acceptable UX)

Senior vs Junior: Design a Collaborative Text Editor

Interviewer: "Design a real-time collaborative text editor — like Google Docs — where multiple users can type simultaneously."

Junior Answer

"I'd use WebSockets to sync the changes between users. When someone types, I'd send the change to the server, and the server would broadcast it to all other connected clients."

This is technically a start, but it misses the entire hard problem. What happens when two users type at exactly the same time? User A deletes character 5. User B, who hasn't received that delete yet, inserts a character at position 5. Now their operations conflict. The junior answer has no resolution strategy for this.

Senior Answer

"Collaborative editing is a conflict resolution problem, not just a networking problem. The naive WebSocket approach breaks immediately when two users edit the same location simultaneously — you get race conditions and corrupted documents. The state of the art has two solutions, and they're conceptually different.

Operational Transforms (OT) is what Google Docs uses. The idea: when two operations are applied to the same document state, transform them relative to each other so they can be applied in any order and still produce the same result. For example: if I delete character 5 and you insert at position 5, after my delete your insert position must shift to 4. A central server coordinates the transformation. The math is complex — implementing OT from scratch is a research-level problem.

CRDTs (Conflict-free Replicated Data Types) are what Figma uses. A CRDT is a data structure with a merge operation that is commutative, associative, and idempotent — meaning any two replicas can merge in any order and always reach the same state, without a central coordinator. For text, the common implementation is a sequence CRDT like RGA or LSEQ. Yjs is the production-grade CRDT library for JavaScript.

For a new implementation, I'd use Yjs. It handles all the conflict resolution. My WebSocket server becomes a relay — it broadcasts Yjs update messages between clients without understanding their content. Here's the architecture:

// Client-side: Yjs + y-websocket
import * as Y from 'yjs';
import { WebsocketProvider } from 'y-websocket';
import { QuillBinding } from 'y-quill'; // or y-prosemirror, y-codemirror

// Each document is a Y.Doc — a CRDT
const ydoc = new Y.Doc();

// Connect to the relay server
const provider = new WebsocketProvider(
  'wss://your-relay-server.com',
  'document-id-abc123',  // room name = document ID
  ydoc,
  { connect: true }
);

// Y.Text is a CRDT-backed text type
const ytext = ydoc.getText('content');

// Bind to your editor
const binding = new QuillBinding(ytext, quillEditor, provider.awareness);

// Awareness: user cursor positions, names, colors
// This is the "who is editing where" overlay
provider.awareness.setLocalState({
  name: currentUser.name,
  color: currentUser.color,
  cursor: null,
});

// provider.awareness.getStates() returns a Map of all connected users' states

The secondary challenge is persistence. The CRDT state grows over time as operations accumulate. You need to periodically create a snapshot of the document state and store it in a database, so new clients don't have to replay the entire history from day one. I'd take a snapshot every N operations or every M minutes, stored in object storage (S3), and keep a reference to the latest snapshot in the database. New clients load the snapshot and only replay operations that came after it."

Chapter Summary

  • Use the eight-step framework: Clarify scope → User interactions → Data model → Component architecture → State management → Network layer → Performance → Accessibility/errors. Don't skip steps 1 and 8 — they're where junior engineers get caught.
  • A design system is built from design tokens up. Tokens are the single source of truth. CSS custom properties for theming. Storybook for documentation. Tree-shakeable npm package for distribution.
  • Analytics dashboards: each widget is independent and handles its own loading/error states. SSE for read-only real-time data. Virtualize large tables with @tanstack/virtual.
  • Infinite scroll: Intersection Observer to trigger fetches, cursor-based pagination for live data, virtualization for the list, skeleton loaders for loading states, and URL-encoded cursor for shareability.
  • Micro-frontends are usually the wrong choice. Know when they're right (large orgs, team autonomy, independent deployment) and be honest about the complexity cost.
  • Real-time: WebSockets for bidirectional (chat, collaborative editing), SSE for server-to-client only (notifications, dashboards), polling as a fallback for low-frequency updates.
  • Collaborative editing requires conflict resolution — OT (Google Docs) or CRDTs (Figma, Yjs). A WebSocket server alone does not solve the problem.
