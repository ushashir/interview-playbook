Frameworks and Component Architecture

The question "Do you know React?" is not what they're actually asking. They're asking: do you understand component thinking, state management, side effects, and why applications fall apart at scale? React is just the language of that conversation.

What you'll learn

  • The React component lifecycle through hooks — mount, update, cleanup
  • The three most common memory leaks in React and how to fix each one
  • A principled decision tree for state management: useState vs context vs Zustand vs Redux
  • How React.memo, useMemo, and useCallback actually work — and when NOT to use them
  • The underused composition pattern that eliminates most prop drilling
  • How NestJS backend architecture shapes the frontend API contract
  • How to discuss any framework in an interview by focusing on the underlying ideas

The LEGO Brick Analogy

React components are LEGO bricks. Each brick does one specific thing. You compose them to build complex structures. The rules are simple: each brick should be self-contained, reusable, and only aware of its own job. A brick that needs to know about the internal workings of every other brick in the set stops being a brick and starts being a dependency mess. When bricks know too much about each other's jobs, the structure becomes brittle — changing one brick requires changing twenty others.

The whole discipline of component architecture is enforcing this rule at scale. Small, focused components. Explicit data flow. Clear ownership of state. When you answer interview questions about React, this LEGO mental model is the frame to speak from.

1. Component Lifecycle (Hooks Era)

The Plant Life Cycle Analogy

A plant has stages: a seed pushes through the soil (mount), the plant grows and responds to its environment — sun, water, seasons (update), and eventually it dies and returns to the earth (unmount/cleanup). React components live the same cycle. They appear in the DOM (mount), re-render in response to state or prop changes (update), and when removed from the DOM they should clean up after themselves — subscriptions, timers, listeners — just like a plant composting back into the soil.

import { useEffect, useState } from 'react';

function ChatRoom({ roomId }) {
  const [messages, setMessages] = useState([]);

  // 1. MOUNT: runs once when component appears in the DOM
  useEffect(() => {

The setup/teardown rule

Every useEffect that sets something up should tear it down in the cleanup function. Subscriptions, event listeners, WebSocket connections, intervals — if you start it in the effect body, end it in the return function. This is not optional. Missing cleanup is a bug; it just might not surface until the component is unmounted at an inopportune time.

2. Memory Leaks from Missing Cleanup

Memory leaks in React components are almost always caused by the same three patterns. Knowing them makes you the person who spots them in code review before they reach production.

Leak 1: Event Listener Added Without Removal

// BUG: event listener never removed
function ResizeTracker() {
  const [width, setWidth] = useState(window.innerWidth);

  useEffect(() => {
    const handleResize = () => setWidth(window.innerWidth);
    window.addEventListener('resize', handleResize);
    // No cleanup! Every time this component mounts, another listener is added.
    // If the component mounts/unmounts repeatedly, you accumulate listeners.
  }, []);

  return 
Width: {width}px
;
}

// FIX: remove the listener in the cleanup function
function ResizeTracker() {
  const [width, setWidth] = useState(window.innerWidth);

  useEffect(() => {
    const handleResize = () => setWidth(window.innerWidth);
    window.addEventListener('resize', handleResize);

    return () => {
      window.removeEventListener('resize', handleResize); // cleanup
    };
  }, []); // handleResize is stable — no need to include it

  return 
Width: {width}px
;
}

3. State Management Patterns

The Office Communication Analogy

State management is like choosing how people in an office share information. Some information only one person needs — keep it on their own desk (local state). Some information two people need — put it in a shared folder between them (lifted state). Information that everyone needs — put it on the company intranet (context). A system so complex it needs version history and audit trails — hire a dedicated information manager with a full filing system (Redux or Zustand).

Decision Tree: Which State Solution?

Is the state used by only one component?
→ YES: useState — keep it local

Is the state shared between a few siblings?
→ YES: Lift state to the nearest common parent

Is the state global/cross-cutting (auth, theme, locale, user)?
→ YES: Context — but only for low-frequency updates

Does the state update frequently, have many writers, or need middleware?
→ YES: Zustand or Redux Toolkit — external store

Context is not a state management solution

Context is a dependency injection tool. It lets you pass values through the tree without prop drilling. But it has a critical problem: every component that consumes a context re-renders whenever the context value changes. If you put frequently-updating state (like a cart item count that updates on every keypress) in Context, you'll trigger cascading re-renders across your entire app. Context is great for: authentication state, theme, locale, feature flags. It's bad for: form state, server data, anything that updates on every user action. For those, use an external store (Zustand, Redux) or a server state library (React Query, SWR).

4. Performance: memo, useMemo, useCallback

The Cached Calculation Analogy

Imagine an accountant who recalculates the company's total revenue from scratch every time someone asks — even if no sales have happened since the last calculation. That's what React does by default: recalculate everything on every render. Memoization tells React: "If the inputs haven't changed, use the answer I already computed." But there's a cost to checking — the accountant has to compare the inputs, which takes time. For cheap calculations, the checking might cost more than just recalculating. Measure before you memoize.

The premature optimization trap

The wrong time to use memo, useMemo, and useCallback is before you've measured. Every memoization has a cost: memory to store the previous value, time to compare the current inputs to the previous inputs, and cognitive overhead when reading the code. For components that render quickly, memoization often costs more than it saves. The right process: build it without optimization, measure with React DevTools Profiler, find the actual bottleneck, then apply the appropriate fix. "It feels slow" is not a measurement.

Composition is the underused third option. When you see prop drilling, your first instinct shouldn't be "add Context." Ask instead: "Can I restructure the component tree so that the data is passed as JSX (a component) rather than as a primitive prop?" 6. NestJS and the Frontend API Contract

Frontend engineers who understand the backend they're consuming write better, more resilient UIs. When your backend is built with NestJS, you get a set of tools that make the frontend-backend contract explicit and type-safe.

DTOs Define the Contract

NestJS uses Data Transfer Objects (DTOs) to define the shape of request and response data. These are TypeScript classes decorated with validation rules. The frontend team should treat DTOs as the authoritative source of truth for what data looks like.

// Backend: NestJS DTO
// apps/user-service/src/dto/user.dto.ts
export class UserResponseDto {
  id: string;
  name: string;
  email: string;
  role: 'admin' | 'member' | 'viewer';
  createdAt: Date;
}

// Frontend: This shape is what the API returns — match it exactly
interface User {
  id: string;
  name: string;
  email: string;
  role: 'admin' | 'member' | 'viewer';
  createdAt: string; // Note: JSON serializes Date as string
}

OpenAPI Auto-Generation

NestJS generates a Swagger/OpenAPI spec automatically from your DTOs and decorators. The frontend team should never be guessing type shapes from network responses — that's how type drift happens. The right workflow:

  1. Backend generates the OpenAPI spec (automatically in NestJS with @nestjs/swagger)
  2. Frontend uses a code generator (like openapi-typescript or orval) to produce TypeScript types and API client code from the spec
  3. When the API changes, regenerate the types — TypeScript will tell you everywhere the frontend breaks
// Generated by openapi-typescript from the NestJS Swagger spec
// No more manually maintaining these types
export interface paths {
  '/users/{id}': {
    get: {
      parameters: { path: { id: string } };
      responses: {
        200: { content: { 'application/json': UserResponseDto } };
        404: { content: { 'application/json': { message: string } } };
      };
    };
  };
}

Shared Types in a Monorepo

The most powerful configuration is TypeScript on both sides in a monorepo (NestJS backend + React frontend in the same repo). A shared packages/types package holds DTOs that both the server and client import. When the server changes a type, the client immediately sees the TypeScript error — before the bug reaches production.

// packages/types/src/user.ts — shared between backend and frontend
export interface User {
  id: string;
  name: string;
  email: string;
  role: 'admin' | 'member' | 'viewer';
}

// apps/api (NestJS) imports from @myapp/types
// apps/web (React) imports from @myapp/types
// Change the type once → TypeScript errors in both apps immediately

7. Framework-Agnostic Interview Framing

If an interviewer asks about Vue and you only know React, don't panic. Every modern UI framework is a different implementation of the same underlying model. Articulating that model shows senior thinking.

The unified model

All reactive UI frameworks express the same idea: UI = f(state). Your UI is a pure function of your application state. When state changes, the framework re-runs that function (or an optimized approximation of it) to produce the new UI. React does this with a virtual DOM diff. Vue does it with a fine-grained reactivity system that tracks which state each component depends on. Svelte does it with a compile step that generates direct DOM mutations. Angular uses zone.js to detect changes. The mechanism differs; the model is the same.

In an interview, you can say: "I know React best, but I'm comfortable with the underlying reactive programming model. The patterns I use — explicit state ownership, unidirectional data flow, side effect isolation — are applicable in Vue, Angular, or Svelte. I'd expect a ramp-up period on the specific API, but the thinking is the same."

This answer signals that you understand your tools at the conceptual level, not just the syntactic one. That's senior thinking.

Senior vs. Junior: Architecting a Large React Application

Junior answer "I'd use React with a Redux store and organize my components into folders. I'd have a components folder and a pages folder. For API calls I'd use fetch or axios."
Senior answer

"I'd start before I write any code — by defining the domain. What are the core entities? What are the relationships between them? That domain model drives everything downstream.

For state, I'd separate concerns by type. Server state — data that lives on the server and is cached on the client — goes into React Query or SWR. That eliminates 80% of what people reach for Redux to solve: loading states, error states, cache invalidation, refetching. Global client state — UI things that aren't server data, like 'is the sidebar open', or 'which theme is selected' — goes into Zustand. Local component state stays in useState.

Component architecture follows feature folders, not file-type folders. Instead of /components, /hooks, /utils at the top level, I'd structure by domain: /features/cart/, /features/auth/, each containing the components, hooks, and types specific to that domain. This makes feature deletion trivial and prevents cross-feature coupling.

I enforce a boundary between presentational components (know how to render data, no API calls) and container components (connected to the store and API, pass data down). This makes the presentational components independently testable in Storybook.

Performance optimization is last. I set up the React Profiler in development and measure before I apply any memo. Bundle size is monitored from the start with a size budget in the CI pipeline — not addressed in a panic before launch.

And I'd negotiate the API contract with the backend team upfront. Bad API design creates bad frontends. If we're using NestJS on the backend, I want shared TypeScript types so the compiler tells us when the contract breaks, not our users."

Practice Questions

Beginner

What is the difference between useEffect(fn, []) and useEffect(fn)?

Answer: useEffect(fn, []) runs the effect once after the component first mounts, and never again. useEffect(fn) (no dependency array) runs after every render — mount and every update. The empty array is not "no dependencies" — it is "this effect doesn't depend on anything that changes," so it never needs to re-run. The missing array means "re-run this on every render," which is rarely what you want and often causes infinite loops if the effect updates any state.

Intermediate

Write a custom hook useLocalStorage that persists state to localStorage and stays in sync across renders.

import { useState, useEffect } from 'react';

function useLocalStorage(key, initialValue) {
  // Lazy initialization: read from localStorage once at mount
  const [storedValue, setStoredValue] = useState(() => {
    try {
      const item = window.localStorage.getItem(key);
      // If item exists, parse it; otherwise use the initialValue
      return item !== null ? JSON.parse(item) : initialValue;
    } catch (error) {
      // localStorage may be unavailable (private mode, SSR, etc.)
      console.warn('useLocalStorage: failed to read', key, error);
      return initialValue;
    }
  });

  // Wrap the setter so it also persists to localStorage
  const setValue = (value) => {
    try {
      // Allow value to be a function (same API as useState)
      const valueToStore = value instanceof Function ? value(storedValue) : value;
      setStoredValue(valueToStore);
      window.localStorage.setItem(key, JSON.stringify(valueToStore));
    } catch (error) {
      console.warn('useLocalStorage: failed to write', key, error);
    }
  };

  return [storedValue, setValue];
}

// Usage:
function Settings() {
  const [theme, setTheme] = useLocalStorage('theme', 'light');

  );
  // theme persists across page refreshes automatically
}
Advanced

You have a React list rendering 10,000 items. It's laggy. Walk through your complete debugging and optimization process.

Step 1 — Measure, don't guess. Open React DevTools Profiler. Record a interaction. Find which components are rendering slowly and how often. Separately, check the Chrome Performance tab for long tasks and layout thrashing.

Step 2 — Identify the root cause. Common culprits in this scenario:

  • All 10,000 DOM nodes exist in the document simultaneously — this is expensive to paint and lay out even if they're off-screen.
  • A parent component is re-rendering frequently and each re-render triggers all 10,000 child renders.
  • An expensive computation runs on every render (a filter or sort without memoization).
  • Missing key props causing React to destroy and recreate DOM nodes instead of reusing them.

Step 3 — Apply targeted fixes:

The senior move: start with virtualization. For a list of 10,000 items, it's almost always the highest-leverage fix, because it reduces the DOM node count from 10,000 to ~20, regardless of how fast each individual component renders.

Chapter Summary

  • Every useEffect that opens a connection, adds a listener, or starts a timer must return a cleanup function — missing cleanup is a memory leak waiting to surface at the worst moment.
  • Choose your state solution by the type of state: local → useState, lifted → parent state, global/stable → Context, global/frequent → Zustand or Redux, server data → React Query.
  • Context is a dependency injection tool, not a state management library — putting frequently-updating state in Context causes cascading re-renders.
  • Memoization (memo, useMemo, useCallback) has a cost — measure with React DevTools Profiler before applying it; premature memoization adds complexity without guaranteed benefit.
  • All reactive UI frameworks implement the same model (UI = f(state)); understanding the model lets you discuss any framework and position yourself above the level of API memorization.
