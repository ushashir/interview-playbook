Beginner Algorithms

These are the problems that seem too simple to study. They are not. Fumbling a FizzBuzz on camera is one of the most demoralizing experiences in tech. Let's make sure that never happens to you.

What you'll learn

  • Seven foundational algorithms — fully implemented, explained, and complexity-analyzed
  • The brute force first, then the optimized solution — every time
  • The common mistakes candidates make on "simple" problems
  • Why Fibonacci is secretly an introduction to dynamic programming
  • How to present beginner solutions with senior-level polish
💡 The Big Idea: Scrambled Eggs

Before Gordon Ramsay earned his Michelin stars, he mastered scrambled eggs. Not because scrambled eggs are impressive, but because they teach the fundamentals — heat control, timing, the feel of the pan. Every harder dish is built on those basics. Beginner algorithms are your scrambled eggs. FizzBuzz teaches conditionals and modulo. Fibonacci teaches recursion and why naive recursion is dangerous. Two Sum teaches the hash map pattern you'll use in dozens of harder problems. Skip these and you'll have gaps in your foundation. Master these and everything else builds naturally.

⚠️ Warning

Don't let the word "beginner" make you complacent. These problems are used as filters — a hiring manager screening for basic competence — not because they're trivial but because they're universally known. A 3-year developer who stumbles on a palindrome check raises a serious question about how much they've actually internalized vs. relied on Stack Overflow. Know these cold.

1. FizzBuzz

💡 Analogy: Sorting Mail by Rules

You're sorting mail into bins. The rules are: if the envelope is both red and square, bin 1. If it's only red, bin 2. If it's only square, bin 3. Otherwise, bin 4. The trick is that the "both" condition must be checked first — otherwise you'll sort combined items incorrectly. FizzBuzz has the same shape: the divisible-by-15 case must come before the individual checks.

Print numbers 1 through n. For multiples of 3, print "Fizz". For multiples of 5, print "Buzz". For multiples of both, print "FizzBuzz".

Common mistake: Checking the individual conditions first. If you check n % 3 === 0 before n % 15 === 0, you'll print "Fizz" for 15 and never reach the "FizzBuzz" branch.

// Standard solution — O(n) time, O(1) space
function fizzBuzz(n) {
  for (let i = 1; i <= n; i++) {
    if (i % 15 === 0) console.log('FizzBuzz'); // Must come first!
    else if (i % 3 === 0) console.log('Fizz');
    else if (i % 5 === 0) console.log('Buzz');
    else console.log(i);
  }
}

// Elegant variant: build the string, avoid repeating the modulo checks
function fizzBuzzClean(n) {
  for (let i = 1; i <= n; i++) {
    let output = '';
    if (i % 3 === 0) output += 'Fizz';
    if (i % 5 === 0) output += 'Buzz';
    console.log(output || i);
  }
}

// One-liner for the follow-up "can you do it without if/else?"
// Uses short-circuit evaluation and string concatenation
const fizzBuzzOneLiner = n =>
  Array.from({ length: n }, (_, i) => i + 1)
    .map(i => (i % 3 ? '' : 'Fizz') + (i % 5 ? '' : 'Buzz') || String(i));

console.log(fizzBuzzOneLiner(15));
// ['1','2','Fizz','4','Buzz','Fizz','7','8','Fizz','Buzz','11','Fizz','13','14','FizzBuzz']
Junior Answer

Writes the three conditions in the wrong order, prints "Fizz" for 15 instead of "FizzBuzz". When asked to extend for a new rule (e.g., multiples of 7 → "Bazz"), manually adds a new condition and hopes for the best.

Senior Answer

Writes the clean string-building version. When asked to extend for multiples of 7, adds if (i % 7 === 0) output += 'Bazz' — no reordering required, scales to any number of rules. Notes that the "combined condition" variant avoids the ordering problem entirely, making it more maintainable.

Complexity: O(n) time, O(1) space. There's no optimization possible here — you have to visit every number.

2. Two Sum

💡 Analogy: Finding a Dance Partner

You need two people whose heights add up to exactly 10 feet. Brute force: pair every person with every other person and measure. Optimized: as each person enters the room, announce "I need someone who is X feet tall" and check if that person is already there. You only need to look at each person once.

Given an array of integers nums and a target integer, return the indices of the two numbers that add up to the target. Assume exactly one solution exists and no element is used twice.

// ─── Brute Force: O(n²) time, O(1) space ─────────────────────────────────────
// Check every possible pair
function twoSumBrute(nums, target) {
  for (let i = 0; i < nums.length; i++) {
    for (let j = i + 1; j < nums.length; j++) {
      if (nums[i] + nums[j] === target) {
        return [i, j];
      }
    }
  }
  return null;
}

// Why it's slow: for each element (n iterations), we scan the rest of the array
// (up to n iterations). n * n = n².

// ─── Optimized: O(n) time, O(n) space ────────────────────────────────────────
// The bottleneck in the brute force is the inner loop — finding the complement.
// Replace it with a hash map lookup.
function twoSum(nums, target) {
  const seen = new Map(); // maps value → index

  for (let i = 0; i < nums.length; i++) {
    const complement = target - nums[i];

    if (seen.has(complement)) {
      return [seen.get(complement), i];
    }

    seen.set(nums[i], i);
  }

  return null;
}

// Trace through [2, 7, 11, 15], target = 9:
// i=0: complement = 7, seen = {}  → not found → seen = {2: 0}
// i=1: complement = 2, seen = {2: 0} → FOUND → return [0, 1]

console.log(twoSum([2, 7, 11, 15], 9));  // [0, 1]
console.log(twoSum([3, 2, 4], 6));       // [1, 2]
console.log(twoSum([3, 3], 6));          // [0, 1]

Common mistake: Using an object instead of a Map and not handling the case where a value is used twice. With nums = [3, 3] and target = 6, you need to make sure you're returning different indices, not the same index twice. The hash map approach handles this naturally because you check the map before inserting.

The key insight to verbalize: "The inner loop in the brute force is just asking 'does the complement exist?' — that's a lookup, and lookups are what hash maps are built for."

3. Reverse a String

💡 Analogy: Reading a Book Backwards

The obvious approach: make a photocopy of every page, then stack them in reverse order. That takes O(n) space for all the copies. The in-place approach: go to the middle of the book, swap page 1 with page n, page 2 with page n-1, and so on. Same result, no photocopier needed.

// ─── Naive: using built-in methods ────────────────────────────────────────────
// O(n) time, O(n) space — creates intermediate array
function reverseStringNaive(s) {
  return s.split('').reverse().join('');
}

// ─── In-place with two pointers ───────────────────────────────────────────────
// O(n) time, O(n) space for the char array (strings are immutable in JS)
// In languages with mutable strings (C++, Java char[]), this is truly O(1) space
function reverseString(s) {
  const chars = s.split(''); // JavaScript strings are immutable, must use array
  let left = 0;
  let right = chars.length - 1;

  while (left < right) {
    // Swap characters at left and right pointers
    [chars[left], chars[right]] = [chars[right], chars[left]];
    left++;
    right--;
  }

  return chars.join('');
}

console.log(reverseString('hello'));    // 'olleh'
console.log(reverseString('racecar')); // 'racecar' (palindrome)
console.log(reverseString(''));         // '' (edge case: empty string)
console.log(reverseString('a'));        // 'a' (edge case: single char)

Common mistake: Forgetting the edge cases — empty string, single character, even vs. odd length. The two-pointer approach handles all of them naturally because left < right exits immediately when they meet or cross.

What the interviewer is really testing: Do you know about string immutability? Do you reach for the two-pointer technique, which is a pattern you'll reuse in palindrome checks, sorted array problems, and more?

4. Check Palindrome

💡 Analogy: Checking if a Word Reads the Same in a Mirror

Naive approach: write the word down, hold it up to a mirror, compare the reflection to the original. Optimized: just compare the first letter to the last, the second to the second-to-last, and stop as soon as you hit the middle. You don't need the reflection at all.

// ─── Naive: reverse and compare ───────────────────────────────────────────────
// O(n) time, O(n) space
function isPalindromeNaive(s) {
  const cleaned = s.toLowerCase().replace(/[^a-z0-9]/g, '');
  return cleaned === cleaned.split('').reverse().join('');
}

// ─── Optimized: two pointers ──────────────────────────────────────────────────
// O(n) time, O(1) space — no intermediate string needed
function isPalindrome(s) {
  // Normalize: lowercase, alphanumeric only (common follow-up requirement)
  const cleaned = s.toLowerCase().replace(/[^a-z0-9]/g, '');

  let left = 0;
  let right = cleaned.length - 1;

  while (left < right) {
    if (cleaned[left] !== cleaned[right]) return false;
    left++;
    right--;
  }

  return true;
}

console.log(isPalindrome('racecar'));          // true
console.log(isPalindrome('A man a plan a canal Panama')); // true
console.log(isPalindrome('hello'));            // false
console.log(isPalindrome(''));                 // true (empty string is a palindrome)

// Follow-up: Valid Palindrome II — can you remove at most one character?
function validPalindromeII(s) {
  function isPalin(str, l, r) {
    while (l < r) {
      if (str[l] !== str[r]) return false;
      l++; r--;
    }
    return true;
  }

  let left = 0, right = s.length - 1;
  while (left < right) {
    if (s[left] !== s[right]) {
      // Try skipping either the left or the right character
      return isPalin(s, left + 1, right) || isPalin(s, left, right - 1);
    }
    left++; right--;
  }
  return true;
}

Common mistake: Forgetting to normalize the string. "A man, a plan, a canal: Panama" is a palindrome if you strip spaces, punctuation, and case — but not if you compare the raw string character-by-character. Always ask the interviewer: "Should I consider only alphanumeric characters?"

5. Find Max / Min in an Array

💡 Analogy: Finding the Tallest Person in a Room

Walk through the room once. Keep a mental note of the tallest person you've seen so far. When you find someone taller, update your note. At the end of the room, your note has the answer. You only needed one pass.

// ─── One pass: O(n) time, O(1) space ─────────────────────────────────────────
function findMax(arr) {
  if (arr.length === 0) throw new Error('Array is empty');

  let max = arr[0]; // initialize with the first element, not -Infinity (unless negatives possible)
  for (let i = 1; i < arr.length; i++) {
    if (arr[i] > max) max = arr[i];
  }
  return max;
}

function findMin(arr) {
  if (arr.length === 0) throw new Error('Array is empty');
  let min = arr[0];
  for (let i = 1; i < arr.length; i++) {
    if (arr[i] < min) min = arr[i];
  }
  return min;
}

// ─── Built-in shortcut ────────────────────────────────────────────────────────
// Fine for small arrays. Risky for very large arrays (stack overflow with spread)
const arr = [3, 1, 4, 1, 5, 9, 2, 6];
console.log(Math.max(...arr));    // 9 — uses spread, causes stack overflow if arr is huge
console.log(Math.min(...arr));    // 1

// Safe for large arrays:
console.log(arr.reduce((max, val) => Math.max(max, val), -Infinity)); // 9
console.log(findMax([3, 1, 4, 1, 5, 9, 2, 6])); // 9

// Find both max and min in a single pass — O(n) time, only n-1 comparisons
function findMaxMin(arr) {
  if (arr.length === 0) throw new Error('Array is empty');
  let max = arr[0], min = arr[0];
  for (let i = 1; i < arr.length; i++) {
    if (arr[i] > max) max = arr[i];
    else if (arr[i] < min) min = arr[i];
  }
  return { max, min };
}

console.log(findMaxMin([3, 1, 4, 1, 5, 9, 2, 6])); // { max: 9, min: 1 }

Common mistake: Initializing max to 0 instead of arr[0]. If all values are negative, you'll return 0, which is wrong. Always initialize to the first element of the array.

When NOT to write a loop: When your array is small and you're in application code (not an algorithm interview), Math.max(...arr) is fine and more readable. In interviews, always mention both options and their tradeoffs.

6. Count Character Frequency

💡 Analogy: Taking Inventory

You have a pile of mixed Lego bricks. You want to know how many of each color you have. You go through the pile once, tallying each color in a notebook. At the end you have a complete count without ever sorting the pile. That notebook is your hash map.

// ─── Character frequency with a hash map ──────────────────────────────────────
// O(n) time, O(k) space where k = number of unique characters
function charFrequency(str) {
  const freq = new Map();
  for (const ch of str) {
    freq.set(ch, (freq.get(ch) ?? 0) + 1);
  }
  return freq;
}

console.log(charFrequency('hello'));
// Map { 'h' => 1, 'e' => 1, 'l' => 2, 'o' => 1 }

// ─── Application: Anagram Detection ──────────────────────────────────────────
// Two strings are anagrams if they have the same characters with the same frequencies.
function isAnagram(s, t) {
  if (s.length !== t.length) return false;

  const freq = new Map();

  // Count characters in s
  for (const ch of s) freq.set(ch, (freq.get(ch) ?? 0) + 1);

  // Subtract characters in t
  for (const ch of t) {
    if (!freq.has(ch)) return false;
    freq.set(ch, freq.get(ch) - 1);
    if (freq.get(ch) === 0) freq.delete(ch);
  }

  // If all counts canceled out, the map should be empty
  return freq.size === 0;
}

console.log(isAnagram('anagram', 'nagaram')); // true
console.log(isAnagram('rat', 'car'));         // false
console.log(isAnagram('listen', 'silent'));   // true

// ─── Application: Group Anagrams ──────────────────────────────────────────────
// Given an array of words, group the anagrams together.
function groupAnagrams(words) {
  const groups = new Map();

  for (const word of words) {
    // The sorted word is the canonical key for a group of anagrams
    const key = word.split('').sort().join('');
    if (!groups.has(key)) groups.set(key, []);
    groups.get(key).push(word);
  }

  return [...groups.values()];
}

console.log(groupAnagrams(['eat', 'tea', 'tan', 'ate', 'nat', 'bat']));
// [['eat','tea','ate'], ['tan','nat'], ['bat']]

Common mistake: Using freq[ch]++ on a plain object without initializing to 0. undefined++ gives NaN. Always initialize: freq[ch] = (freq[ch] || 0) + 1 or use the Map version with ?? 0.

Senior insight: The sorted-string key trick for group anagrams is worth memorizing. Any time you need a canonical representation of a set of characters, sort them. It turns a complex equality problem into a simple string comparison.

7. Fibonacci

💡 Analogy: Recomputing Your Commute Time Every Day from Scratch

Imagine you calculate your commute time by re-adding every road segment, every signal, every turn — from scratch — every single morning. That's naive recursive Fibonacci. But if you wrote down yesterday's total on a sticky note and just adjusted for any new changes, you'd be done in seconds. Memoization is the sticky note. Dynamic programming is the discipline of always using the sticky note.

Fibonacci is the gateway drug to dynamic programming. It looks trivial, but it perfectly demonstrates why naive recursion is dangerous, what memoization does, and how iterative solutions can be more efficient than recursive ones.

// ─── Recursive (Naive): O(2ⁿ) time, O(n) space (call stack) ──────────────────
// Why exponential? Each call branches into two more calls.
// fib(5) calls fib(4) and fib(3).
// fib(4) calls fib(3) and fib(2).
// fib(3) is computed TWICE. fib(2) is computed THREE times.
// The total calls form a binary tree of depth n → 2^n nodes.
function fibRecursive(n) {
  if (n <= 1) return n;
  return fibRecursive(n - 1) + fibRecursive(n - 2);
}

// fibRecursive(40) takes seconds. fibRecursive(50) would take minutes.

// ─── Memoized Recursion: O(n) time, O(n) space ────────────────────────────────
// Cache results so each subproblem is solved exactly once.
function fibMemo(n, memo = new Map()) {
  if (n <= 1) return n;
  if (memo.has(n)) return memo.get(n); // already computed? return immediately

  const result = fibMemo(n - 1, memo) + fibMemo(n - 2, memo);
  memo.set(n, result);
  return result;
}

// ─── Iterative (Bottom-Up DP): O(n) time, O(1) space ─────────────────────────
// Build from the bottom up. No call stack. No memoization overhead.
// This is the most efficient approach in production.
function fibIterative(n) {
  if (n <= 1) return n;

  let prev2 = 0; // fib(0)
  let prev1 = 1; // fib(1)

  for (let i = 2; i <= n; i++) {
    const current = prev1 + prev2;
    prev2 = prev1;
    prev1 = current;
  }

  return prev1;
}

// Test all three
console.log(fibRecursive(10));  // 55 (works, but slow for large n)
console.log(fibMemo(50));       // 12586269025 (fast)
console.log(fibIterative(50));  // 12586269025 (fast and constant space)

// Complexity comparison:
// fibRecursive: Time O(2^n), Space O(n) — the call stack grows to depth n
// fibMemo:      Time O(n),   Space O(n) — n entries in the memo map
// fibIterative: Time O(n),   Space O(1) — only two variables, no memo needed
💡 Tip

When an interviewer asks for Fibonacci, start with the naive recursive solution and immediately call out that it's exponential and why. Then show the memoized version. Then show the iterative. This three-step progression demonstrates that you understand the problem at multiple levels — not just that you know the optimized answer.

Junior Answer

Writes the recursive solution. When asked "what's the time complexity?" says "O(n) because there are n recursive calls." Does not know about the branching — misses that each call spawns two more.

Senior Answer

"The naive recursive solution is O(2^n) because the call tree branches at every node. I can draw that out if useful. Memoization brings it to O(n) by caching subresults. The iterative approach achieves O(n) time with O(1) space — no call stack, just two rolling variables. In production I'd use the iterative version. For a DP explanation, I'd start with the memoized version since it maps directly to the recursive intuition."

Practice Questions

Beginner Write FizzBuzz without using any if or else keywords. There are multiple approaches — the string-concatenation method shown above is one. Can you think of another using a lookup array or object?

Intermediate Given a string, find the first non-repeating character and return its index. If none exists, return -1. Example: "leetcode" → 0 (the 'l' appears only once). Example: "aabb" → -1. Approach: two passes — first pass builds a frequency map, second pass finds the first character with frequency 1. Time O(n), space O(1) since there are at most 26 lowercase letters.

Advanced Full worked solution: Longest Subarray with Equal Number of 0s and 1s. Given a binary array (only 0s and 1s), find the maximum length of a contiguous subarray with equal numbers of 0s and 1s. Example: [0, 1, 0] → 2. Example: [0, 1, 0, 1, 1, 0, 0] → 6.

The insight: Convert 0s to -1s. Now "equal 0s and 1s" becomes "subarray sum equals 0." This is a classic prefix sum + hash map problem.

Key observation: If the prefix sum at index i equals the prefix sum at index j (where i < j), then the subarray from i+1 to j has sum 0 — meaning equal 0s and 1s. Track the first time each prefix sum appears, then maximize the distance when you see it again.

function findMaxLength(nums) {
  // Step 1: convert 0s to -1s so the problem becomes "subarray sum = 0"
  // Step 2: use prefix sum + hash map to find longest zero-sum subarray

  const map = new Map();
  map.set(0, -1); // prefix sum of 0 exists before the array starts (index -1)

  let prefixSum = 0;
  let maxLen = 0;

  for (let i = 0; i < nums.length; i++) {
    prefixSum += nums[i] === 0 ? -1 : 1; // treat 0 as -1

    if (map.has(prefixSum)) {
      // Same prefix sum seen before → subarray between has sum 0
      maxLen = Math.max(maxLen, i - map.get(prefixSum));
    } else {
      // First time seeing this prefix sum — record the index
      // (we want the earliest occurrence to maximize length)
      map.set(prefixSum, i);
    }
  }

  return maxLen;
}

// Trace through [0, 1, 0, 1, 1, 0, 0]:
// i=0: num=0→-1, sum=-1, map={0:-1,-1:0}
// i=1: num=1,    sum=0,  seen at -1 → len = 1-(-1) = 2, maxLen=2
// i=2: num=0→-1, sum=-1, seen at 0  → len = 2-0 = 2, maxLen=2
// i=3: num=1,    sum=0,  seen at -1 → len = 3-(-1) = 4, maxLen=4
// i=4: num=1,    sum=1,  map={..., 1:4}
// i=5: num=0→-1, sum=0,  seen at -1 → len = 5-(-1) = 6, maxLen=6
// i=6: num=0→-1, sum=-1, seen at 0  → len = 6-0 = 6, maxLen=6

console.log(findMaxLength([0, 1, 0]));              // 2
console.log(findMaxLength([0, 1, 0, 1, 1, 0, 0])); // 6
console.log(findMaxLength([0, 0, 0, 1, 1, 1]));    // 6

Complexity: O(n) time — one pass. O(n) space — hash map stores at most n+1 entries. The brute force would be O(n²) — check all subarrays. The key technique — prefix sum + hash map — is a pattern worth memorizing. It appears in dozens of subarray problems.

Chapter Summary

  • Beginner algorithms are calibration problems — get them wrong and the interview is effectively over before it starts. Know these cold.
  • Always show brute force first, then optimize. The jump from O(n²) to O(n) via hash map is the most common optimization pattern in beginner-level problems.
  • Fibonacci is not a trivia question — it's an introduction to memoization and bottom-up DP thinking. The three-version progression (recursive → memoized → iterative) demonstrates mastery of the concept.
  • Character frequency via hash map is the foundation of anagram detection, group anagrams, and many string problems — it's one of the most reusable patterns in the beginner toolkit.
  • The prefix sum + hash map pattern unlocks a whole class of subarray problems that would otherwise require O(n²) brute force. Recognizing when to apply it is a significant skill jump.
