System Design Building Blocks Reference

The 10 components that appear in 90% of system design interviews, and the tradeoffs that make or break your answer.

Building Blocks Quick Reference

Component What It Does Use When Don't Use When Interview Red Flag
Load Balancer Distributes incoming traffic across multiple server instances Multiple application instances; need high availability or horizontal scale Single server, low traffic, or stateful sessions without sticky routing Adding one without explaining sticky sessions / health checks
PostgreSQL ACID-compliant relational database with advanced query support Structured data, complex JOINs, strong consistency requirements, default choice Massive horizontal write scale needs or true schema-less documents Choosing NoSQL by default "for scale" without reasoning
MongoDB Document database with flexible schema and horizontal sharding Varied/nested document structures, rapid schema evolution, denormalized reads Multi-document transactions at high volume, complex relational queries Using it because "it's faster" — it isn't, by default
Redis In-memory key-value store with persistence options and pub/sub Caching hot reads, session storage, rate limiting, leaderboards, pub/sub fanout Primary data store for large datasets; data you cannot afford to lose without persistence config Treating it as a primary DB without acknowledging data loss risk
Kafka Distributed, durable, high-throughput event streaming log High-volume event pipelines, audit logs, event sourcing, fan-out to many consumers, replay Simple task queues, low-volume messaging, when operational complexity isn't justified Adding Kafka to every design regardless of volume requirements
RabbitMQ Message broker with flexible routing, acknowledgments, and dead-letter queues Task queues, work distribution, complex routing logic, guaranteed delivery Event replay / reprocessing history; high-throughput streaming Confusing it with Kafka — they solve different problems
CDN Globally distributed edge cache serving static and dynamic content close to users Static assets, media, geographically distributed users, high-traffic reads Highly personalized or real-time content that cannot be cached Not considering cache invalidation strategy
API Gateway Single entry point for routing, auth, rate limiting, and protocol translation Microservices with many clients, cross-cutting concerns (auth, rate limit, logging) Simple monolith or single-client architecture where added latency isn't justified Making it a single point of failure without HA configuration
Elasticsearch Distributed full-text search and analytics engine built on Lucene Full-text search, faceted filtering, log aggregation, autocomplete Primary transactional store; data requiring strong consistency Using it as the source of truth — always a secondary read store synced via CDC
S3 / Object Store Infinitely scalable blob storage for arbitrary binary objects User uploads, media files, backups, large binary data, static website hosting Structured queryable data, sub-millisecond access patterns, small transactional records Storing application state or using it as a database

Tradeoffs Reference Card

SQL vs NoSQL

Use SQL when... Use NoSQL when...
Data has well-defined, stable schema Schema varies per record or evolves rapidly
You need multi-table JOINs and complex queries Access patterns are simple (lookup by key/range)
ACID transactions across multiple entities Need horizontal write scale across many nodes
Starting a new project (default to PostgreSQL) Data naturally fits a document, graph, or wide-column model

REST vs GraphQL vs gRPC

Use REST when... Use GraphQL when... Use gRPC when...
Public APIs, widely consumed, HTTP cache matters Multiple client types need different data shapes Internal service-to-service communication
Simple CRUD resources, standard tooling Clients suffer from over-fetching / under-fetching High-performance, low-latency, binary protocol
Team unfamiliar with schema-first design Rapid frontend iteration without backend changes Strongly typed contracts between services

Sync vs Async Communication

Use Synchronous when... Use Asynchronous when...
The caller needs the result immediately to proceed Work can be decoupled from the request/response cycle
Simple request/response fits the use case Operations are slow (email, video encoding, notifications)
Failure handling is straightforward (retry, error return) Need to absorb traffic spikes or rate-limit downstream calls

Monolith vs Microservices

Use Monolith when... Use Microservices when...
Early-stage product; domain boundaries not yet clear Teams need to deploy independently of each other
Small team (fewer than 10 engineers) Services have vastly different scaling requirements
Operational simplicity is a priority Clear domain boundaries exist (DDD bounded contexts)
Latency between services would be unacceptable Org structure already reflects independent teams (Conway's Law)

Cache Strategies

Strategy How It Works Best For Drawback
Cache-aside App checks cache; on miss, reads DB and populates cache Read-heavy workloads, lazy population Cache miss storm on cold start; stale data risk
Write-through App writes to cache and DB simultaneously on every write Read-after-write consistency required Write penalty; cache fills with rarely-read data
Write-behind App writes to cache; DB is updated asynchronously later Write-heavy; tolerate eventual persistence Data loss risk if cache node fails before flush

Common System Design Patterns

Pattern Problem It Solves Key Mechanism Chapter
CQRS Read and write models with different shape/scale requirements Separate command (write) and query (read) models; sync via events Ch 32
Saga Distributed transactions across multiple services Sequence of local transactions; compensating transactions on failure Ch 19
Outbox Dual-write problem: DB update + event publish must be atomic Write event to outbox table in same DB transaction; relay publishes it Ch 19
Circuit Breaker Prevent cascading failures when a downstream service is degraded Track failure rate; open circuit (fail fast) when threshold exceeded Ch 19
Strangler Fig Incrementally migrate a monolith to microservices without big-bang rewrite Route traffic for new features to new service; strangle old code over time Ch 32
BFF (Backend for Frontend) Different clients need different API shapes from the same backend Dedicated aggregator service per client type (web, mobile, partner) Ch 32
Rate Limiting (Token Bucket) Protect services from traffic spikes and abusive clients Bucket fills at fixed rate; each request consumes a token; empty = reject Ch 30
Consistent Hashing Minimize key remapping when nodes are added/removed from a cluster Hash ring; each node owns a range; add/remove only remaps adjacent keys Ch 30
Event Sourcing Audit trail, time travel, and rebuilding state from history Store all state changes as immutable events; derive state by replaying Ch 32
Hexagonal Architecture Decouple business logic from infrastructure (DB, HTTP, queues) Core domain surrounded by ports; adapters plug in at the edges Ch 32

Database Choice Decision Tree

  Start: What is your primary data access pattern?
  │
  ├── Need complex multi-table JOINs or ACID transactions?
  │     └── YES → PostgreSQL (or MySQL)
  │
  ├── Need horizontal write scale with flexible schema?
  │     └── YES → MongoDB (or DynamoDB for AWS)
  │
  ├── Need full-text search and faceted filtering?
  │     └── YES → Elasticsearch (as a secondary read store, synced via CDC)
  │
  ├── Need sub-millisecond reads / caching / session store?
  │     └── YES → Redis
  │
  ├── Need time-series data at high write volume?
  │     └── YES → Cassandra or TimescaleDB
  │
  ├── Need a wide-column store for massive scale reads?
  │     └── YES → Cassandra or HBase
  │
  └── Not sure / starting fresh?
        └── DEFAULT → PostgreSQL
              (Mature, reliable, extensible. Migrate later if you hit its limits.)
Interview tip: Always say "I'd start with PostgreSQL unless there's a specific access pattern that rules it out." This shows you understand premature optimization is a real cost — and that SQL is not a performance liability for most systems.
