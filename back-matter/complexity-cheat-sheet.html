Algorithm Best Average Worst Space Stable? Notes

Selection Sort O(n²) O(n²) O(n²) O(1) No Minimizes swaps; never adaptive
Insertion Sort O(n) O(n²) O(n²) O(1) Yes Excellent on small or nearly sorted data

Recursion & Space Complexity

Scenario Space Complexity Explanation
Linear recursion (depth n) O(n) Each call frame lives on the stack until the base case is reached
Balanced divide-and-conquer O(log n) Tree depth is log n (e.g., binary search, merge sort call stack)
In-place algorithms O(1) No extra memory proportional to input (e.g., heap sort, two-pointer)
Memoization added to O(2^n) O(n) added Cache stores n unique subproblems; time drops from O(2^n) to O(n)
Tail recursion (optimized) O(1) Compiler reuses stack frame; not guaranteed in all languages

Pattern → Complexity Guide

Code Pattern Typical Time Complexity Example Problem
Single loop over n elements O(n) Find max, linear search, prefix sum
Nested loops over same array O(n²) Bubble sort, all-pairs comparison, brute-force two-sum
Binary search on sorted input O(log n) Search in rotated array, find insertion point
Divide and conquer (halving) O(n log n) Merge sort, closest pair of points
Hash map lookup / insert O(1) amortized Two-sum, anagram detection, frequency count
BFS / DFS on graph O(V + E) Connected components, shortest path (unweighted), cycle detection
DP with 1D state array O(n) time, O(n) or O(1) space Fibonacci, coin change, climbing stairs
DP with 2D state table O(n×m) time, O(n×m) space Longest common subsequence, edit distance, 0/1 knapsack
Heap / priority queue operations O(log n) per op Kth largest, merge k sorted lists, Dijkstra's algorithm
Sliding window (fixed or variable) O(n) Max subarray, longest substring without repeats

Quick Estimation Tips

Rule of thumb: ~108 simple operations per second in most compiled/JIT languages. Use 107–108 as your budget and work backwards from input size.

Input Size (n) Acceptable Complexity Unacceptable
n ≤ 20 O(2^n) or even O(n!) acceptable —
n ≤ 100 O(n³) borderline (106) O(2^n)
n ≤ 1,000 O(n²) = 106 (fast) O(n³) = 109 (too slow)
n ≤ 105 O(n log n) = ~1.7 × 106 O(n²) = 1010
n ≤ 106 O(n) = 106, O(n log n) ≈ 2 × 107 O(n²) = 1012 (never)
n ≤ 109 O(log n) or O(1) only Anything linear or worse
Interview shortcut: State the complexity before you code. Say: "This will be O(n log n) time and O(n) space — is that acceptable?" It signals you think about tradeoffs, not just solutions.
