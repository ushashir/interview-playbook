How to Approach System Design Questions

A repeatable 4-step framework that replaces anxiety with process — and signals senior thinking from the first minute.

What you'll learn

  • The 4-step framework and exactly how long to spend on each phase
  • How to do back-of-envelope estimation that impresses interviewers
  • How to surface tradeoffs rather than pretend they don't exist
  • What interviewers are actually scoring — and what destroys candidates
  • A full URL shortener walkthrough from naive to production-grade

"System design questions have no perfect answer. The interviewer knows this. What they're scoring is your ability to reason systematically under uncertainty, surface tradeoffs, and defend decisions — the exact skills you'll use every day as a senior engineer."

The Architecture Analogy

Analogy: Designing a Building

An architect doesn't start sketching floor plans the moment a client walks in. They ask: how many people will occupy this building? What's the budget? Will it expand in five years? Is it a hospital or a hotel? Only after those answers do they touch pencil to paper.

System design interviews follow the same logic. The software you're asked to sketch is an architecture — and like all architecture, the constraints come before the blueprints. Clarify first, estimate second, sketch third, detail fourth.

The 4-Step Framework

Every system design interview, regardless of the prompt, follows the same structure. The difference between a passing and failing answer is almost always about spending the right amount of time in each phase — not about knowing more technology.

Timeline (40-minute interview):
┌──────────────┬──────────────┬─────────────────────┬──────────────────┐
│  CLARIFY     │   ESTIMATE   │   HIGH-LEVEL DESIGN  │   DEEP DIVE      │
│   5 min      │   5 min      │      15 min          │    15 min        │
│              │              │                      │                  │
│ Requirements │ Back-of-env. │ Happy path + diagram │ 1-2 components   │
│ Scale        │ math         │ Core components      │ at impl. level   │
│ Constraints  │ Storage est. │ Data flow            │ Tradeoffs named  │
└──────────────┴──────────────┴─────────────────────┴──────────────────┘

Step 1 — Clarify (5 minutes)

This is the phase most candidates skip. They hear "design Twitter" and immediately start talking about databases. That's a mistake. You need to understand what you're actually building before you build anything.

Ask these categories of questions, in this order:

Functional requirements — what does the system do?

  • "What are the core features in scope for this interview? Should I focus on posting, feeds, or DMs?"
  • "Are we designing for read-heavy or write-heavy behavior?"
  • "Is search in scope?"

Non-functional requirements — what qualities must it have?

  • "What's the acceptable latency for the most common operation?"
  • "Do we need strong consistency, or is eventual consistency acceptable?"
  • "What's the availability requirement? 99.9%? 99.99%?"

Scale — how big is this thing?

  • "How many monthly active users are we targeting?"
  • "What's the daily active user count?"
  • "Is this global, or single-region?"
Tip: Signal senior thinking immediately. Asking "is this read-heavy or write-heavy?" in the first 60 seconds immediately signals you understand that system architecture changes based on this answer. Most junior candidates never ask it.

Step 2 — Estimate Scale (5 minutes)

Back-of-envelope math is not about precision. It is about demonstrating that you understand the order of magnitude of the problem — and that your design is sized correctly for it.

Walk through this chain out loud. Say the numbers. The interviewer wants to watch you reason, not hear a conclusion.

Scale Estimation Chain:
─────────────────────────────────────────────────────────────
1M MAU
  → Assume 10% DAU = 100,000 DAU
  → Assume 10 requests/user/day = 1,000,000 req/day
  → 1,000,000 / 86,400 seconds ≈ 12 req/sec (avg)
  → Peak = avg × 10 = 120 req/sec

Storage:
  1M users × 1 KB per record = 1 GB   (comfortable for one DB)
  1B users × 1 MB per user   = 1 TB   (need distributed storage)
  1B events/day × 1 KB each  = 1 TB/day (need Kafka + cold storage)
─────────────────────────────────────────────────────────────

Useful constants to have memorized:

  • 1 million seconds ≈ 11.5 days
  • 1 day = 86,400 seconds
  • Average web request: ~1 KB payload
  • Average image: 100 KB–1 MB
  • Average video: 100 MB–1 GB per hour
  • Redis: handles ~100,000 ops/sec per instance
  • PostgreSQL: handles ~10,000 writes/sec, ~50,000 reads/sec comfortably

Step 3 — High-Level Design (15 minutes)

Draw the happy path first. Don't start with edge cases. Don't start with failure scenarios. Show the most common operation end to end, then add complexity.

Every system design should include at minimum these components unless you've explicitly decided not to and said why:

  • Load balancer — distributes traffic across server instances
  • API servers — stateless, horizontally scalable
  • Cache — reduces database load for reads
  • Database — persistent source of truth
Minimal viable system design skeleton:

        [Client]
           |
     [Load Balancer]
      /    |    \
  [API] [API] [API]   (stateless, horizontally scalable)
      \    |    /
       [Cache (Redis)]
           |
       [Database]
      /          \
 [Primary]   [Read Replica]

While drawing, narrate. "I'm putting a load balancer at the edge because our API servers need to be stateless and horizontally scalable. I'm using a cache between the API tier and the database because our read-to-write ratio is 100:1 — most traffic is reads."

Step 4 — Deep Dive (15 minutes)

The interviewer will guide this phase. They will pick one or two components and ask you to go deeper. Be ready to descend to implementation level on any component you've drawn.

If they point at your cache and ask "how do you handle cache invalidation?", you need a real answer — not "it depends." You need: "I'd use a TTL-based approach with a 1-hour TTL for product data, and event-based invalidation triggered by write events for user session data, because session data correctness is more critical than product data."

Warning: Don't draw components you can't explain. If you put Kafka in your diagram and can't explain how consumer groups work, you've just handed the interviewer a free attack surface. Only draw what you can defend.

The Tradeoffs You Must Always Surface

The Senior Rule: Never say "I'd use X." Always say "I'd use X because Y, though the tradeoff is Z."

These four tradeoffs appear in virtually every system design question. Know how to articulate all of them:

Consistency vs. Availability (CAP Theorem)

In the presence of a network partition, a distributed system must choose between consistency (every read sees the most recent write) and availability (every request receives a response, even if stale). You cannot guarantee both.

  • CP systems (Consistency + Partition tolerance): HBase, ZooKeeper. Correct but can be unavailable during network issues.
  • AP systems (Availability + Partition tolerance): Cassandra, DynamoDB. Always responds, but may return stale data.

In interviews: "For this payment system, I need strong consistency, so I'll accept some availability cost during network partitions. For the notification feed, eventual consistency is fine — stale by a few seconds is acceptable."

Latency vs. Throughput

Optimizing for one often hurts the other. Batching increases throughput but increases latency per operation. Single-record writes are low-latency but hit the database harder.

Simplicity vs. Flexibility

A monolith is simpler to develop and debug. Microservices give flexibility but add network complexity, observability overhead, and deployment complexity. For a 3-engineer startup, a monolith is the right answer. For a 500-engineer organization with 30 teams, microservices decompose ownership.

Cost vs. Performance

More cache = faster reads but higher cost. Multi-region deployment = lower latency globally but 3× the infrastructure bill. Always acknowledge this dimension.

Case Study: Design a URL Shortener

This is the most common warm-up system design question. It looks simple. It hides a lot of nuance. Work through it completely before your interview.

Requirements

  • Shorten 100 million URLs (write volume)
  • Serve 1 billion redirect requests per day (read volume)
  • Custom short codes optional
  • Analytics (click counts, geo breakdown)
  • Short URLs must not be guessable or reversible

Scale estimate: 1B redirects/day = 1,000,000,000 / 86,400 = ~11,574 redirects/sec at peak.

Naive Design and Why It Fails

NAIVE DESIGN:
─────────────
[Client] → POST /shorten → [API Server] → MD5(url)[0:7] → [PostgreSQL]
[Client] → GET /abc123  → [API Server] → SELECT url → [PostgreSQL] → 302 Redirect

Problems:
  - MD5 collision risk: two different URLs could produce the same short code
  - MD5 is reversible: the original URL can be reconstructed from the hash
  - No caching: every redirect hits PostgreSQL at 11,574 req/sec
  - PostgreSQL single instance: fails around 50,000 reads/sec, but still wasteful
  - Analytics is synchronous: slows down the redirect response

Better Short Code Generation

Don't hash the URL. Instead, use an auto-incrementing integer ID from the database and encode it in Base62 (characters a-z, A-Z, 0-9). A 7-character Base62 string can represent 62^7 = 3.5 trillion unique URLs. No collisions. No reversibility. Simple and fast.

Base62 Encoding:
  ID: 12345678  →  Base62: "5BXxz"  (always unique, not reversible)
  Steps: divide by 62 repeatedly, map remainders to character set
  62^6 = 56 billion codes (6 chars is sufficient for 100M URLs with headroom)

Production Design

PRODUCTION DESIGN:
──────────────────

WRITE PATH (create short URL):
  [Client]
     |
  [Load Balancer]
     |
  [API Server]
     |
  [Rate Limiter] ← Redis (sliding window per IP)
     |
  [Primary PostgreSQL] ← auto-increment ID, Base62 encode
     |
  [Async: push to Redis cache] ← warm cache on write

READ PATH (redirect):
  [Client]
     |
  [CDN (CloudFront)] ← cache 302 responses with Cache-Control
     |  (cache HIT: 90%+ of traffic)
     |
  [Load Balancer]
     |
  [API Server]
     |
  [Redis Cache] ← cache HIT: 99%+ of remaining traffic
     |  (TTL: 24h for popular URLs)
     |
  [Read Replica PostgreSQL] ← only true cache misses

ANALYTICS PATH:
  [API Server] → Kafka topic "url.clicked" → Analytics Consumer
                                            → ClickHouse / BigQuery
                                            (async, never blocks redirect)

Validating the Design

Estimate: 11,574 redirects/sec at peak. Redis handles 100,000+ ops/sec. With a 99%+ cache hit rate, fewer than 116 requests/sec reach the database. A single read replica handles this comfortably. The design is validated against the scale requirements.

What the Interviewer Is Actually Scoring

Senior engineers at top companies use a scorecard. Understanding what's on it changes how you approach the interview.

DimensionWhat It Looks Like
Requirement GatheringDid you ask the right questions before drawing anything?
Scale AwarenessDo you know what 1M req/sec actually means for your components?
Component JustificationNot just "I'd use Kafka" — WHY Kafka, and what's the alternative?
Tradeoff SurfacingThis is the thing that signals senior thinking most clearly.
CommunicationCan you draw this and explain it clearly in real-time?
Depth on DemandWhen pushed on a component, can you go implementation-level deep?

What Kills System Design Interviews

These are the patterns that end interviews early. All of them are avoidable.

  • Starting before clarifying requirements. You may build the wrong thing in perfect detail.
  • Designing the "perfect" system instead of the right system for the stated constraints. A 3-engineer startup does not need Kafka.
  • Never mentioning tradeoffs. A senior engineer who claims their design has no downsides is either lying or hasn't thought it through.
  • Going too deep too fast. Spending 20 minutes on database sharding before you've addressed how requests get routed is a failure to prioritize.
  • Designing for scale you don't need. If your estimate shows 100 req/sec, you don't need Cassandra. PostgreSQL and Redis are more than sufficient — and saying that signals maturity.

Senior vs. Junior: URL Shortener Design

Junior

"I'd store the URL in a database and use MD5 to generate the short code. Then when someone visits the short URL, look it up in the database and redirect them."

What's missing: no discussion of collision risk, no caching strategy, no estimation of read load, no mention of analytics as a separate concern, no rate limiting on creation.

Senior

"First let me clarify scale — are we targeting 100M URLs and 1B redirects/day, or is this a smaller deployment? Assuming those numbers: reads are roughly 11,500/sec at peak, which means our database cannot handle redirect traffic directly. I'd use Redis as a cache in front of a read replica, with CDN caching redirect responses at the edge for maximum URLs. For short code generation, I'd auto-increment an ID in PostgreSQL and encode it as Base62 — no collision risk, not reversible, deterministic. I'd separate analytics into an async Kafka pipeline so it never adds latency to the redirect path. The main tradeoff here is that this adds operational complexity — two additional infrastructure components. For a small deployment, a single PostgreSQL with an index and a connection pool is perfectly sufficient."

Chapter Summary

  • Use the 4-step framework in every system design: Clarify (5 min), Estimate (5 min), High-Level Design (15 min), Deep Dive (15 min).
  • Always clarify functional requirements, non-functional requirements, and scale before drawing anything.
  • Back-of-envelope math is about reasoning out loud, not precision — walk through the chain and state your assumptions.
  • Never say "I'd use X." Always say "I'd use X because Y, though the tradeoff is Z." Tradeoff surfacing is the clearest signal of senior thinking.
  • The URL shortener teaches core patterns: Base62 encoding, cache-aside, async analytics, CDN for cacheable responses, and rate limiting — patterns that recur across dozens of other system design problems.

Practice Questions

Intermediate

Design a paste service (like Pastebin). Apply the 4-step framework. What changes if pastes expire after 30 days? How do you handle expiration at scale?

Intermediate

You've estimated your URL shortener will receive 500 req/sec on day one. Walk through which components you would remove from the production design because they're premature. Justify each removal.

Advanced

Your URL shortener needs to support custom short codes (e.g., bit.ly/my-brand). How does this change your short code generation strategy? What race conditions emerge and how do you prevent them?

Advanced

An interviewer pushes back: "Your Redis cache is a single point of failure. How do you handle it?" Walk through Redis Sentinel vs. Redis Cluster vs. client-side consistent hashing and the tradeoffs of each.
