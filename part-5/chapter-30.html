Core Building Blocks

Ten components that assemble into almost any system — master not just what they are, but when to reach for them and when they're the wrong tool.

What you'll learn

  • The 10 standard building blocks used in virtually every system design interview
  • For each: when to use it, when NOT to use it, and the interview-critical fact
  • The common candidate mistake for each block — and how to avoid it
  • A master comparison table to use as a study reference
  • How to justify component selection the way senior engineers do

"Every system design interview answer is built from the same 10 building blocks. Master these — not what they are, but when to reach for them and when NOT to — and you can design almost any system."

Analogy: LEGO Bricks

A master LEGO builder doesn't design a castle from scratch on every build. They know which bricks exist, how they connect, and which combinations are structurally sound. The creativity is in the assembly — not in reinventing the bricks.

System design is the same. The building blocks are fixed. Your skill is knowing which ones to assemble for the problem in front of you, and being able to explain why you chose each piece over the alternatives.

Building Block 1: Load Balancer

Analogy: A traffic cop at a busy intersection — directing cars to whichever lane is moving fastest, pulling cars out of the lane if a crash occurs.

A load balancer distributes incoming traffic across multiple server instances. Without one, scaling horizontally is impossible — all traffic hits one server.

L4 vs L7:

  • Layer 4 (transport): routes by IP and TCP port. Fast, but can't inspect HTTP content. Use for raw TCP performance.
  • Layer 7 (application): routes by HTTP headers, URL path, cookies. More flexible — can route /api/* to one pool and /static/* to another. Use for web applications.

Routing algorithms:

  • Round-robin: requests distributed evenly in sequence. Simple, no state required.
  • Least-connections: send to the server with the fewest active connections. Better for long-lived connections.
  • Consistent hashing: routes the same client consistently to the same server. Required for sticky sessions without server-side session storage.

When to use: Any stateless, horizontally scaled application.
When NOT to use: There is no scenario where you don't want a load balancer if you have more than one server instance. The question is which type.
Interview-critical fact: Sticky sessions (session affinity) routes a client to the same server every time. It sounds good but creates hot spots and complicates deploys. The better solution is to store sessions in Redis so any server can serve any client.

Common candidate mistake: Drawing a load balancer but saying "the servers can share session state" without explaining how. If your servers are stateless, say so explicitly and explain where session state lives (Redis, JWT tokens, etc.).

Building Block 2: Relational Database (PostgreSQL / MySQL)

Analogy: An organized filing cabinet with strict cross-references — every drawer has labeled folders, and a folder in one drawer can officially reference a folder in another.

Relational databases enforce schema, support ACID transactions, and allow complex JOINs across related data. PostgreSQL is the current industry preference for new systems — it supports JSON columns, full-text search, and advanced indexing (GIN, GiST) beyond basic B-tree.

Use when: data has complex relationships (orders → line items → products → inventory), you need transactions (debit account A, credit account B — atomically), or your schema is stable and well-understood.

Don't use when: you need horizontal write scaling beyond a few terabytes. PostgreSQL scales reads easily with read replicas but write throughput is bounded by the single primary. At write-heavy TB-scale, you need a different tool.

Interview-critical fact: B-tree indexes make range queries and equality queries fast (O(log n)). A missing index on a foreign key is the most common production performance bug in relational databases. Always mention indexes when discussing database design.

Tip: When an interviewer asks "how would you scale your database?", the correct first answer is almost always "read replicas for read scaling, then connection pooling (PgBouncer), and only then sharding if the data truly requires it." Jumping to sharding before exhausting simpler options signals you're pattern-matching, not reasoning.

Building Block 3: NoSQL — Document Store (MongoDB)

Analogy: A flexible corkboard where you can pin anything in any shape — sticky notes, index cards, photos — without needing every item to look the same.

Document databases store JSON-like documents. Schema is flexible per-document. Embedded documents allow you to store related data together (a product with embedded reviews) rather than joining across tables.

Use when: schema changes frequently during development, content is hierarchical (CMS, product catalogs with varied attributes), or you need to embed related data for read performance without JOINs.

Don't use when: you have complex many-to-many relationships that require joins, or you need ACID transactions across multiple documents. MongoDB added multi-document transactions in 4.0, but they are slower than PostgreSQL equivalents and the data model fights them.

Interview-critical fact: The most common MongoDB mistake is embedding unbounded arrays in documents (e.g., embedding all comments in a post document). MongoDB has a 16 MB document size limit. For data that grows unboundedly, use a separate collection with a reference.

Common candidate mistake: Choosing MongoDB "for flexibility" without identifying what specifically needs to be flexible. Interviewers hear this as "I don't know what my data looks like." Be specific: "I'd use a document store here because the product attribute schema varies by category — electronics have voltage specs, clothing has sizes and colors — and a relational schema would require an EAV pattern which is painful."

Building Block 4: NoSQL — Key-Value Store (Redis / DynamoDB)

Analogy: A set of labeled drawers — you know exactly which drawer holds what, open it directly, and get sub-millisecond access. No searching, no sorting.

Key-value stores trade query flexibility for raw speed. You can only look up by key — no range scans, no full-text search, no complex filters.

Redis: in-memory, optional disk persistence (RDB snapshots, AOF log), rich data structures beyond simple string values:

  • Sorted sets: leaderboards, rate limiting with sliding windows
  • Lists: activity feeds, queues
  • Pub/Sub: lightweight real-time messaging between services
  • Streams: persistent log with consumer groups (lightweight Kafka alternative)

DynamoDB: managed AWS service, serverless pricing, single-digit millisecond latency at any scale. No tuning required. Global tables for multi-region active-active. The tradeoff: you're locked into AWS and the access pattern must be designed upfront.

Use when: caching, session storage, rate limiting, leaderboards, feature flags, distributed locks.

Don't use when: you need complex queries, reporting, or relationships. Key-value is a specialized accelerator, not a general-purpose database.

Interview-critical fact: Redis is single-threaded for command execution, which is why it can make 100,000+ operations/sec guarantees — no lock contention. Redis Cluster shards across nodes for horizontal scale when a single instance isn't enough.

Building Block 5: NoSQL — Wide-Column Store (Cassandra)

Analogy: A spreadsheet on steroids — optimized for appending rows incredibly fast, with columns that can vary by row and queries that are optimized for reading time-ordered slices.

Cassandra is a distributed database designed for write-heavy, time-series workloads. It writes to an in-memory structure (memtable) and an append-only log, then flushes to disk — making writes extremely fast. Reads are slightly more expensive.

Use when: IoT sensor data, event logs, time-series metrics, activity streams — anything where you write continuously and read time-ordered slices.

Don't use when: you need complex queries, transactions, or flexible access patterns. Cassandra's data model must be designed around your query patterns — changing queries later often means restructuring tables.

Interview-critical fact: Cassandra uses eventual consistency by default. You can tune consistency level per query (QUORUM reads guarantee reading the latest write, at the cost of latency). In interviews, acknowledge this when recommending Cassandra: "We'd accept eventual consistency on metrics dashboards, but order history would require QUORUM reads."

Common candidate mistake: Recommending Cassandra because "it's distributed and scalable." PostgreSQL with read replicas and partitioning handles most workloads to hundreds of millions of rows. Cassandra adds significant operational complexity — justify it with write volume (millions of writes/sec) or data volume (tens of TB+) that genuinely exceeds what PostgreSQL can handle.

Building Block 6: Cache (Redis)

Analogy: A sticky note on your monitor with the phone number you call every day — faster than looking it up in your contacts every time.

Caching stores frequently accessed data in fast memory (Redis) to avoid repeated expensive computation or database queries.

Cache-aside pattern (the most common): application checks cache first. On miss, reads from database and writes to cache. On write, invalidates or updates the cache.

Cache-Aside Read:
  [App] → GET user:1234 → [Redis]
              MISS? → [PostgreSQL] → return data → [App]
                                               → [Redis] SET user:1234 (TTL: 1h)
              HIT? → return data → [App]

Cache Key Design:
  user:1234:profile        ← entity:id:property
  product:5678:inventory   ← namespaced with colons
  rate:192.168.1.1:minute  ← rate limit by IP + window

Cache invalidation strategies:

  • TTL-based: set an expiry time. Simple. Accepts stale data up to TTL duration. Good for product catalog, public data.
  • Event-based: when data changes, immediately delete or update the cache key. Consistent but adds coupling. Good for user sessions, profile data.
  • Write-through: on every write to the database, also write to cache. Cache always warm. Tradeoff: cache fills with infrequently-read data.

Eviction policies:

  • LRU (Least Recently Used): evicts the least recently accessed keys first. Good for general-purpose caches.
  • LFU (Least Frequently Used): evicts the least accessed over time. Better for keeping "popular" items — hot products, trending content.
  • TTL: evicts keys based purely on expiry time. Predictable.

Interview-critical fact: Cache keys without TTL are memory leaks. If your application sets keys but never expires them, Redis will fill up and start evicting based on eviction policy — which may evict the wrong things. Always set TTLs.

Building Block 7: Message Queue (Kafka / RabbitMQ / SQS)

Analogy: A postal sorting facility. The sender drops off a package (message). The facility holds it until the recipient is ready. The sender and recipient never have to be available at the same time.

Message queues decouple producers (services that generate events) from consumers (services that react to them). This enables async processing, independent scaling, and resilience to consumer failures.

Kafka

A distributed, log-based message system. Messages are stored durably on disk with configurable retention (days, weeks, forever). Consumers read from an offset — they can replay past messages. Consumer groups allow parallelism within a topic.

Use Kafka when: event sourcing, audit logs, stream processing, high-throughput pipelines (millions of events/sec), or when you need replay capability (re-process events after a bug fix).

Kafka's tradeoff: operationally complex. Requires ZooKeeper (or KRaft in newer versions), partition management, offset management. Worth it at scale. Overkill for simple task queues.

RabbitMQ

A traditional message broker. Messages are routed via exchanges and bindings to queues. Supports complex routing topologies (fanout, direct, topic). Messages are deleted after consumption.

Use RabbitMQ when: task queues (background jobs), RPC-over-queue patterns, routing complexity, or when your team already runs it. The AMQP protocol is widely understood.

SQS (Amazon Simple Queue Service)

Managed AWS queue. No infrastructure to run. At-least-once delivery (design for idempotency). FIFO queues for guaranteed ordering at lower throughput.

Use SQS when: AWS-native stack, you don't need Kafka's replay capability, and you want zero operational overhead. The right choice for most AWS workloads that don't need Kafka features.

Message Queue Decision Tree:
  Need replay/event-sourcing?  → Kafka
  Need complex routing?        → RabbitMQ
  AWS-native, simple task?     → SQS
  Need ordered FIFO?           → SQS FIFO or Kafka (with single partition)

Building Block 8: CDN (Content Delivery Network)

Analogy: A warehouse network that stores your products near your customers — instead of shipping everything from one factory across the country, you stock regional fulfillment centers.

A CDN caches content at geographically distributed edge nodes. When a user in Tokyo requests an asset, they get it from Tokyo's edge node, not a server in Virginia.

Use for: static assets (JS, CSS, images, fonts) always. Also: video streams, software downloads, and — often overlooked — API responses that are cacheable (public product catalog, search results with cache-control headers).

Pull vs Push model:

  • Pull: CDN fetches from origin on first request, caches based on Cache-Control headers. Simple. Use for most web assets.
  • Push: you proactively upload content to CDN edge nodes. Use for large files (video) where you want to pre-populate before users request.

Edge computing: Cloudflare Workers, Lambda@Edge allow running code at CDN edge nodes. Use for: auth token validation without round-trip to origin, A/B testing, bot detection.

Interview-critical fact: Cache-Control headers control CDN behavior. Cache-Control: public, max-age=86400 means cache this for 24 hours. Cache-Control: private, no-store means never cache (for authenticated user-specific responses).

Common candidate mistake: Only applying CDN to static files. In interviews, explicitly mention that cacheable API responses (product listings, public content, search results) should also be cached at the CDN. This demonstrates you understand CDN as a general caching layer, not just a file server.

Building Block 9: Search Engine (Elasticsearch)

Analogy: A librarian who has memorized every word on every page of every book in the library — and can tell you instantly which books contain "machine learning" within 10 words of "Python," ranked by relevance.

Elasticsearch (and OpenSearch, its open-source fork) provides full-text search, faceted filtering, fuzzy matching, and relevance scoring. It inverts the data — instead of "book has words," it stores "word appears in these books" — enabling sub-second full-text queries across billions of documents.

Use when: full-text search (e-commerce product search, blog search), faceted filtering (filter by brand, price range, rating simultaneously), fuzzy matching (typo tolerance), or complex aggregations for analytics dashboards.

Don't use as a primary database. Elasticsearch is eventually consistent, doesn't support ACID transactions, and loses data more easily than PostgreSQL. Always sync data FROM your primary database INTO Elasticsearch.

Correct Elasticsearch Architecture:
  [Writes] → [PostgreSQL] ← source of truth
                  |
          [Debezium CDC]  ← reads PostgreSQL WAL
                  |
              [Kafka]
                  |
        [Elasticsearch Consumer]
                  |
           [Elasticsearch] ← search reads only

Interview-critical fact: Elasticsearch relevance scoring uses TF-IDF and BM25 algorithms. Documents that mention a search term more frequently (TF) and in smaller documents (IDF) score higher. In practice: acknowledge relevance tuning exists and mention that boosting fields (title > description > body) is a common production concern.

Building Block 10: Object Storage (S3 / GCS)

Analogy: An infinitely expandable filing cabinet in a warehouse — you can put any size file in any drawer, the cabinet never fills up, and you pay only for what you store.

Object storage is the right answer for any binary large object: user-uploaded images, videos, audio, PDF documents, database backups, ML model artifacts, log archives.

Use when: storing files of any size. Never store BLOBs in your relational database — it bloats the database, defeats indexes, and makes backups enormous. Store the file in S3, store the S3 key (URL) in the database.

Always pair with a CDN for serving. Direct S3 access is slower and costs more in data transfer than serving through CloudFront. S3 is optimized for durability and storage cost, not for serving high-frequency requests.

Key features to mention in interviews:

  • Versioning: S3 can keep every version of every file. Critical for user documents.
  • Lifecycle policies: automatically move objects to cheaper storage tiers (S3 Glacier) after 90 days. Move to deep archive after 1 year.
  • Pre-signed URLs: generate a time-limited URL allowing a client to upload directly to S3, bypassing your server. Removes your server from the upload path — better performance, lower egress cost.

Interview-critical fact: Pre-signed upload URLs are the correct pattern for user file uploads. Client requests a pre-signed URL from your API, uploads directly to S3, then calls your API to confirm the upload. Your server never handles the binary data.

Load Balancer Horizontal scaling, HA Essentially unlimited (route to more servers) Mentioning sticky sessions without explaining where sessions actually live
PostgreSQL / MySQL Relational data, transactions, complex queries ~10K writes/sec, ~50K reads/sec (single node) Jumping to sharding before read replicas
MongoDB Flexible schema, hierarchical content High with sharding Embedding unbounded arrays in documents
Redis (KV) Cache, sessions, rate limiting, leaderboards ~100K ops/sec single node; cluster for more Not setting TTLs on keys
Cassandra Time-series, write-heavy, IoT Linearly with nodes Using it for relational data or complex queries
Kafka Event sourcing, high-throughput streams, replay Millions of events/sec with partitions Using it where SQS would suffice
CDN Static assets, cacheable content, global latency Effectively unlimited Only mentioning it for static files, ignoring cacheable APIs
Elasticsearch Full-text search, faceting, fuzzy matching Billions of documents with sharding Using as the primary database
Object Storage (S3) Files, images, videos, backups Unlimited Serving directly from S3 without CDN
RabbitMQ / SQS Task queues, background jobs, routing Tens of thousands of messages/sec Not designing for at-least-once (idempotency)

Senior vs. Junior: Component Selection

Junior

"I'd store user avatars in the database as a BLOB. That keeps everything in one place. For search, I'd just use a LIKE query on the PostgreSQL table. For caching, I'd add Redis."

Problems: BLOBs in relational DB bloat backups and defeat indexes. LIKE queries don't scale to millions of rows. "Add Redis" without explaining what to cache, TTLs, or invalidation strategy is not a design decision.

Senior

"Avatars go in S3 with CloudFront in front for serving — never in the database. The database stores only the S3 key. For search, at 10M+ products a LIKE query is unusable — I'd sync from PostgreSQL to Elasticsearch via Debezium CDC and serve all search queries from Elasticsearch. PostgreSQL remains the source of truth; Elasticsearch is a read projection. For the Redis cache, I'd cache product detail pages with a 1-hour TTL and invalidate on update events. Sessions I'd cache with a 24-hour TTL and invalidate on logout. The tradeoff on Elasticsearch is operational complexity — syncing and managing two stores — but at our scale there's no alternative for full-text search quality."

Chapter Summary

  • The 10 building blocks — load balancer, relational DB, document store, key-value store, wide-column store, cache, message queue, CDN, search engine, object storage — assemble into virtually any system design answer.
  • The senior signal is not knowing all 10 exist — it is knowing when NOT to use each one and being able to articulate the tradeoffs of your choice.
  • Redis without TTLs is a memory leak. Elasticsearch as a primary database is a data loss risk. BLOBs in PostgreSQL are a backup disaster. Know the failure modes.
  • Always justify component selection: "I'd use Kafka because we need event replay for audit compliance, though the operational overhead vs SQS is real — at our team size, SQS would be acceptable if replay weren't required."
  • Pre-signed S3 URLs, Debezium CDC into Elasticsearch, and Redis sorted sets for leaderboards are three patterns that immediately signal experience when mentioned correctly.

Practice Questions

Beginner

A candidate says "I'd use Redis for everything because it's fast." Identify three scenarios where Redis would be the wrong choice and explain why.

Intermediate

You need to add full-text search to an existing PostgreSQL-backed application. Walk through the complete architecture for syncing data to Elasticsearch, handling out-of-order updates, and keeping the systems consistent if the Elasticsearch indexer has downtime.

Intermediate

An interviewer asks: "Your design uses both Kafka and Redis. That seems like a lot. Could you simplify?" How do you defend keeping both, or how do you decide which to drop?

Advanced

Design the data layer for a multi-tenant SaaS application. Each tenant has between 100 and 10 million records. How do you decide between: a shared database with tenant_id columns, a database-per-tenant, and a schema-per-tenant? Walk through the tradeoffs for each.
