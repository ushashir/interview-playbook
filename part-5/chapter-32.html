Architecture Patterns for Senior Engineers

The patterns that appear in Staff and Senior Engineer discussions — not textbook definitions, but how you'd argue for or against them in a real interview.

What you'll learn

  • Event-driven architecture: loose coupling, independent scaling, and the hard parts
  • CQRS in depth: read model optimization and event sourcing as a projection
  • Hexagonal Architecture: why it matters for testability and long-term maintainability
  • Strangler Fig: how to migrate a monolith without a big-bang rewrite
  • API Gateway patterns: BFF, aggregation, and centralized auth
  • Data consistency patterns: outbox, CDC, and when to use two-phase commit

"Knowing design patterns is junior. Knowing when to apply them and when they're overkill is senior. This chapter covers the patterns that appear in real Staff and Senior Engineer discussions — not textbook definitions, but how you'd actually argue for or against them."

Analogy: Medical Diagnosis

A doctor who has memorized 100 treatments but cannot diagnose is dangerous. They might prescribe the right drug for the wrong condition. A skilled doctor diagnoses first, then matches treatment to patient and context.

Architecture patterns are treatments. Knowing event-driven architecture, CQRS, and the strangler fig pattern is table stakes at senior level. What separates senior from staff is knowing when to prescribe each — and being able to argue against them when the context doesn't warrant the complexity.

Pattern 1: Event-Driven Architecture

Analogy: A Newsroom

The editor publishes a story. The print department, the website team, the social media team, and the archive team each independently decide how to respond to it. The editor doesn't know who subscribes, doesn't wait for them to respond, and doesn't care if the social media post takes 30 seconds to go out. Publishers and subscribers are completely decoupled.

In an event-driven system, services communicate by publishing events to a shared bus (Kafka, EventBridge, SNS) rather than calling each other directly. Publishers emit facts — "order placed," "payment succeeded," "user registered" — without knowing who cares or what they'll do with the information.

Synchronous (tight coupling):
  [Order Service] ──call──→ [Inventory Service]
                 ──call──→ [Notification Service]
                 ──call──→ [Analytics Service]
  Problem: Order Service must know about all three. If any is down, order fails.

Event-Driven (loose coupling):
  [Order Service] → publish "order.placed" → [Kafka]
                                                 |
                              ┌──────────────────┼──────────────────┐
                    [Inventory Consumer]  [Notification Consumer]  [Analytics Consumer]
                    (reduce stock)        (send email)             (record event)

  Order Service knows nothing about downstream services.
  Each consumer independently scales, deploys, and fails.

When to use event-driven architecture:

  • Large systems with many teams: teams own their consumers independently
  • Independent scaling: notification service spikes during marketing campaigns, not during checkout
  • Audit trail required: every event is persisted in Kafka with replay capability
  • Long-running workflows: saga pattern coordinates multi-step processes via events

When NOT to use event-driven architecture:

  • Small teams: the operational overhead of Kafka + consumer management outweighs benefits for <5 engineers
  • Synchronous responses required: if the user needs an answer before they can proceed, async events don't fit
  • Simple CRUD apps: adding Kafka to a to-do list app is pure overhead

The hard parts that interviewers probe:

  • Eventual consistency: downstream services update after a delay. The order service says "paid" but the inventory service hasn't yet decremented the stock. This is acceptable for many use cases and unacceptable for others — be explicit about which.
  • Debugging: tracing a request through 5 async consumers requires distributed tracing (Jaeger, Zipkin) with a shared correlation ID propagated through every event.
  • Event schema evolution: if the "order.placed" event schema changes, all consumers must be updated. Backward-compatible changes (additive fields) are safe. Removing fields breaks consumers. Use Avro or Protobuf with a schema registry to enforce compatibility.
Interview red flag: Describing event-driven architecture as "just use Kafka and everything is decoupled." The coupling moves from service-to-service calls to the event schema. Schema evolution is a hidden, long-term coupling that you must address.

Pattern 2: CQRS in Depth

Analogy: A Restaurant with Separate Kitchen Staff

The head chef handles complex dish preparation (writes). A separate expediter handles plating and delivery (reads). Different people, different tools, optimized for different tasks. Trying to use the same person for both creates a bottleneck — and a mess.

Command Query Responsibility Segregation (CQRS) separates the write model (commands that change state) from the read model (queries that return data). This allows each to be independently optimized.

Read Model Optimization

The write model normalizes data for integrity (3NF: users table, orders table, line_items table, products table). The read model denormalizes for query performance — a single "order summary" document containing everything a UI needs, pre-joined.

Write Model (normalized):
  users(id, name, email)
  orders(id, user_id, status, created_at)
  order_items(id, order_id, product_id, qty, price)
  products(id, name, image_url)

  Write: INSERT/UPDATE one table at a time. Transactionally consistent.
  JOIN query for order page: 4 table join — expensive at scale.

Read Model (denormalized, CQRS):
  order_summaries(
    order_id, user_name, user_email,
    status, total,
    items: [{ name, image_url, qty, price }],
    created_at
  )

  Read: single document lookup by order_id. O(1).
  No joins. Pre-computed. Served from Redis or read replica.

  Updated by: event consumer that listens for "order.*" events
              and projects the write model onto the read model.

Event Sourcing + CQRS

Event sourcing takes CQRS further: the write model stores only events (immutable facts), not current state. Current state is a projection — computed by replaying all events.

Event Sourcing:
  Events table (append-only):
    { type: "OrderPlaced",   orderId: 123, userId: 456, ts: ... }
    { type: "PaymentMade",   orderId: 123, amount: 99.00, ts: ... }
    { type: "OrderShipped",  orderId: 123, trackingCode: "UPS123", ts: ... }

  Current state of order 123: replay all events in sequence
    → status: SHIPPED, total: 99.00, tracking: UPS123

  Advantages:
    - Complete audit trail (immutable)
    - Replay events to rebuild read models after bugs
    - Time-travel: reconstruct state at any point in history

  Disadvantages:
    - Current state requires replaying all events (mitigate with snapshots)
    - Schema evolution of events is hard
    - Overkill for most domains — only use where audit/replay is genuinely required

Code sketch — projecting to Redis read model:

// Event handler: when 'order.shipped' event arrives
async function onOrderShipped(event: OrderShippedEvent) {
  const key = `order:${event.orderId}:summary`;
  const current = await redis.get(key);
  const summary = JSON.parse(current);

  summary.status = 'SHIPPED';
  summary.trackingCode = event.trackingCode;
  summary.shippedAt = event.timestamp;

  await redis.set(key, JSON.stringify(summary));
  // Read model is now up to date — no JOIN query required
}

Eventual consistency on reads: between a write event and the read model being updated, there's a window (typically milliseconds to seconds) where the read model is stale. This is acceptable for most read operations. For critical reads (payment confirmation), read directly from the write model. Design explicitly: "reads of payment status come from PostgreSQL directly — we don't accept staleness there. Order history reads come from the denormalized read model — 30-second staleness is acceptable."

Pattern 3: Hexagonal Architecture (Ports and Adapters)

Analogy: A Universal Power Adapter

Your laptop doesn't care if you plug it into a UK socket, a US socket, or a European socket. The adapter handles the translation. The laptop's core functionality — running your code — is completely isolated from the specifics of electrical standards in different countries.

Hexagonal architecture does this for your business logic. The core doesn't care if it's talking to PostgreSQL or MongoDB, a REST endpoint or a message queue. Adapters handle the translation.

In hexagonal architecture, the application is divided into:

  • Core (Domain): pure business logic with no infrastructure dependencies. Can run with no database, no HTTP, no Kafka.
  • Ports: interfaces the core defines. The core says "I need something that can save a user" — that's a port (UserRepository interface). The core says "I expose this operation" — that's also a port (CreateOrderUseCase interface).
  • Adapters: implementations of ports. PostgresUserRepository implements UserRepository. InMemoryUserRepository also implements UserRepository — used in tests.
Hexagonal Architecture:

          [REST Controller]   [Kafka Consumer]   [CLI]
               |                    |              |
           (Primary Adapters — drive the application)
               |                    |              |
               └────────────────────┴──────────────┘
                                    |
                          ┌─────────▼──────────┐
                          │    CORE / DOMAIN   │
                          │  Business Logic    │
                          │  (no imports of   │
                          │   Postgres, Redis, │
                          │   HTTP, etc.)      │
                          └─────────┬──────────┘
                                    |
               ┌────────────────────┴──────────────┐
               |                    |              |
           (Secondary Adapters — driven by the application)
               |                    |              |
    [PostgresUserRepo]   [RedisCache]   [SendGridEmailAdapter]
    implements            implements      implements
    UserRepository        CachePort       EmailPort

Why it matters for testability: In tests, swap PostgresUserRepository for InMemoryUserRepository. Your business logic tests run in milliseconds with no database required. This is the most practical benefit — not theoretical purity, but genuinely fast, reliable tests.

NestJS directory structure example:

src/
  orders/
    domain/                    ← Core: no infrastructure
      order.entity.ts
      order.repository.ts      ← Port (interface)
      create-order.use-case.ts ← Business logic

    adapters/
      inbound/                 ← Primary adapters (drive the app)
        orders.controller.ts   ← REST
        order-events.consumer.ts ← Kafka

      outbound/                ← Secondary adapters (driven by app)
        postgres-order.repository.ts  ← implements order.repository.ts
        redis-order-cache.ts
        sendgrid-notification.ts

    orders.module.ts           ← Wire up: bind interface to implementation
Interview tip: When asked "how do you structure your NestJS services?" describing hexagonal architecture with this directory structure immediately signals you think about long-term maintainability, not just getting features out. Most candidates describe a flat controllers/services/repos structure — hexagonal differentiates you.

Pattern 4: Strangler Fig (Migrating a Monolith)

Analogy: The Strangler Fig Vine

A strangler fig starts as a vine growing around an existing tree. Over years, it grows around the host tree, eventually replacing it. The original tree decomposes from within, leaving the fig standing in the same shape. At no point is the forest without a tree. You never clear-cut and replant.

This is exactly how you should migrate a monolith to microservices. Never clear-cut. Never stop the forest.

The strangler fig migration strategy allows you to incrementally extract functionality from a monolith into services, with the monolith continuing to run throughout the process. Users experience no disruption. Rollback is always available.

Strangler Fig Migration Steps:

PHASE 0: Status Quo
  [Client] → [Monolith] (handles everything)

PHASE 1: Add Facade (proxy layer)
  [Client] → [Facade / API Gateway]
                    |
              [Monolith] (still handles everything)
  The facade adds zero value at first — just a passthrough.
  But now you control routing.

PHASE 2: Extract first service (new features go here)
  [Client] → [Facade / API Gateway]
               |              |
        [Notification      [Monolith]
          Service]         (still handles most things)
  New "notification preferences" feature → built in the new service
  Old notification code in monolith still runs old features

PHASE 3: Migrate existing features incrementally
  For each bounded context (user management, orders, payments):
    1. Build the service with the same behavior
    2. Route 1% of traffic to new service (canary)
    3. Compare outputs (shadow mode)
    4. Ramp to 100% when confident
    5. Remove that module from the monolith

PHASE 4 (eventual): Monolith retired
  [Client] → [API Gateway]
               |    |    |
           [User] [Order] [Payment]
               Services

  The monolith process can now be shut down.

What NOT to do — the Big Bang Rewrite: "Let's stop feature development for 6 months, rewrite everything from scratch, and then switch over." This almost always fails or causes chaos. The rewrite is never complete when promised, the old system accumulates new features during the rewrite that must be re-replicated, and the switch-over day is a crisis. The strangler fig's value is that you never have a switch-over day.

Key principles for strangler fig execution:

  • Identify bounded contexts before extracting — don't split arbitrarily by technical layer (services layer, repos layer), split by business domain
  • Start with new features, not rewrites of existing features — reduce risk early
  • Measure before and after each extraction: latency, error rate, throughput
  • Maintain data compatibility during transition — both monolith and service may need to read/write shared data temporarily

Pattern 5: API Gateway Patterns

Analogy: A Hotel Concierge

A hotel guest doesn't call the kitchen, the housekeeping department, and the valet separately. They call the concierge, who coordinates all of it. The guest makes one call, the concierge handles the rest. The API gateway is your concierge.

Backend for Frontend (BFF)

A single API gateway for all clients is a compromise — the mobile app needs different data shapes than the web app, and both need different things than a third-party partner API. The BFF pattern creates separate gateway instances per client type, each optimized for its client's needs.

Without BFF (one API fits all — poorly):
  [Web App]    ─┐
  [Mobile App] ─┤→ [Single API Gateway] → [Services]
  [Partner API]─┘
  Problem: API becomes a bloated compromise

With BFF:
  [Web App]     → [Web BFF]     → aggregates data optimized for desktop UI
  [Mobile App]  → [Mobile BFF]  → minimal payload, battery-efficient responses
  [Partner API] → [Partner BFF] → versioned, stable, documented API surface
  Each BFF independently deployed, independently versioned.

Aggregation at the Gateway

Rather than the client making 5 separate API calls to render a page (user profile + recent orders + notifications + recommendations + cart), the gateway aggregates: calls all 5 services in parallel, merges the responses, and returns one payload. This reduces client latency (one round-trip vs. five) and reduces the number of connections clients must manage.

Gateway Aggregation (parallel fan-out):
  Client → GET /dashboard
               |
         [API Gateway]
          /   |   |   \
    [User] [Orders] [Notif] [Cart]  ← parallel calls
          \   |   |   /
           merged response
               |
          Client renders

  Latency: max(user, orders, notif, cart) instead of sum
  Timeout: if one service exceeds 500ms, return partial data + default

Rate Limiting and Auth at the Gateway

Authentication (JWT validation) and rate limiting should live at the gateway, not duplicated in every downstream service. If every service validates JWT independently, a security bug in the validation logic must be patched in 20 places. Centralized at the gateway: patch once.

Rate limiting at the gateway: 1000 requests/minute per API key. Services never see more than their designed load. Downstream services trust that requests passing the gateway are authenticated and within rate limits.

Pattern 6: Data Consistency Patterns

Analogy: Moving Houses

When you move, you need your furniture out of the old house and into the new one without ever having no furniture at all. Data consistency patterns are strategies for moving data between systems while keeping everything coherent — no data in neither place, no data in both places at once.

Outbox Pattern (recap)

Write to the database and insert an "outbox" event row in the same transaction. A relay service polls the outbox and publishes to Kafka. Guarantees at-least-once event delivery without distributed transactions. Covered in detail in Chapter 31.

Change Data Capture (CDC)

Debezium reads the PostgreSQL write-ahead log (WAL) — the database's own durability mechanism — and publishes change events to Kafka. Every INSERT, UPDATE, DELETE becomes a Kafka event. No changes to application code required.

CDC Architecture:
  [Application] → writes to [PostgreSQL]
                                |
                    PostgreSQL WAL (write-ahead log)
                                |
                         [Debezium Connector]
                         (reads WAL as a Postgres client)
                                |
                           [Kafka topics]
                           "postgres.public.orders"
                                |
                  ┌─────────────┴──────────────┐
        [Elasticsearch Indexer]      [Analytics Consumer]
        [Read Model Updater]         [Audit Log Writer]

  Advantages over dual-write:
    - Application writes to ONE place (Postgres only)
    - No risk of partial write (DB wrote, Kafka didn't)
    - Captures all changes including bulk updates and admin edits
    - Zero application code change to add a new consumer

Two-Phase Commit (2PC)

Two-phase commit achieves strong consistency across multiple databases or services without an outbox pattern. In phase 1 (prepare), all participants vote whether they can commit. In phase 2 (commit or abort), the coordinator either commits all or aborts all based on votes.

Two-Phase Commit:
  Phase 1 (Prepare):
    Coordinator → "Can you commit?" → Database A → "YES (vote)"
    Coordinator → "Can you commit?" → Database B → "YES (vote)"

  Phase 2 (Commit):
    All voted YES → Coordinator → "Commit" → Database A + B commit

  Phase 2 (Abort):
    Any voted NO  → Coordinator → "Abort"  → Database A + B roll back

  Problem: if coordinator crashes after Phase 1 but before Phase 2,
  participants are stuck in limbo — blocking protocol.
  This is why 2PC is rarely used in practice for distributed systems.
  Know the theory. Explain why you'd use Saga + Outbox instead.

When 2PC is used in practice: within a single database engine (PostgreSQL supports 2PC for distributed queries with its foreign data wrapper). Across entirely separate systems, the blocking nature and coordinator single-point-of-failure make it impractical. The outbox pattern + saga pattern provides equivalent consistency guarantees without the blocking.

Senior vs. Junior: Monolith Migration

"How would you migrate our monolith to microservices?"

Junior

"I'd break it up into smaller services. Maybe take each module and make it a separate service with its own database. Probably need Docker and Kubernetes."

Problems: no migration strategy, no risk mitigation, no mention of bounded contexts, no discussion of data sharing during transition, no plan for what happens when the new services have bugs, no rollback strategy.

Senior

"I'd use the Strangler Fig pattern — never a big-bang rewrite. First step: add a facade (API gateway) in front of the monolith with a passthrough configuration. This costs nothing and immediately gives us routing control.

Next, I'd identify bounded contexts within the monolith — not technical layers, but business domains. For a typical e-commerce monolith: user management, catalog, orders, payments, notifications. Each becomes a candidate service.

I'd start with new features rather than rewrites — any new capability goes into a new service from day one, routed through the facade. This builds team confidence and production experience with the new infrastructure without risking existing functionality.

For migrating existing features, I'd do canary releases: route 1% of traffic to the new service, compare outputs to the monolith in shadow mode, ramp up when parity is confirmed. Rollback is always one routing change away.

The key risk is shared data during migration. If both the monolith and the orders service write to the same database table temporarily, that's acceptable — it's a migration, not a permanent state. Once the monolith's orders module is retired, the orders service owns its data exclusively.

Timeline: expect 6-18 months for a meaningful monolith depending on size. Anyone promising 3 months hasn't done this before."

Chapter Summary

  • Event-driven architecture decouples services and enables independent scaling, but introduces eventual consistency and moves coupling from service calls to event schemas — schema evolution is the long-term challenge.
  • CQRS separates write and read models for independent optimization; event sourcing takes this further by storing events as the write model, with current state computed as a projection. Use event sourcing only where audit trail and replay are genuine requirements.
  • Hexagonal architecture (ports and adapters) isolates business logic from infrastructure, enabling fast tests with in-memory adapters and painless swaps of infrastructure components — the practical benefit is test speed, not theoretical elegance.
  • The Strangler Fig is the only safe monolith migration strategy at production scale: facade first, new features in services second, incremental migration with canary releases third, monolith retirement last — never a big-bang rewrite.
  • Data consistency across services: use the outbox pattern for guaranteed event publishing, CDC (Debezium) for zero-code-change event generation from database changes, and sagas for multi-step workflow coordination. Two-phase commit is theoretically interesting but practically abandoned due to its blocking nature.

Practice Questions

Intermediate

You're asked to add CQRS to an existing e-commerce application. Walk through: what becomes the write model, what becomes the read model, how does the read model stay in sync, and what is the consistency guarantee for reads immediately after a write?

Intermediate

An interviewer asks: "Should we add an API gateway to our system?" Your system currently has 5 backend services and 2 client types (web, mobile). Walk through the tradeoffs of adding a gateway vs. having clients call services directly. When does the gateway become worth the complexity?

Advanced

You're leading a strangler fig migration of a 500K line Rails monolith. The monolith uses ActiveRecord, which means all models share one database. How do you handle data ownership during the migration? At what point does the orders service get its own database, and what does the transition look like?

Advanced

Your event-driven system has 12 consumers of the "order.placed" event. You need to add a new required field (fulfillmentWarehouseId) to the event schema. Walk through the migration strategy that doesn't break existing consumers while they're updated. What role does a schema registry play?
