C# Deep Dive

C# is the language Microsoft spent 20 years quietly perfecting. If you're interviewing at a company with Azure infrastructure, a Windows-native product, or a .NET ecosystem, C# questions will come. The candidates who know it deeply stand out because most devs treat it as "Java but verbose."

What you'll learn

  • How C# meaningfully differs from Java and why those differences matter in interviews
  • LINQ — the most powerful data-querying feature in any mainstream language
  • async/await, Task<T>, ConfigureAwait(false), and cancellation tokens
  • ASP.NET Core patterns: controllers, DI lifetimes, Entity Framework Core
  • Value types vs reference types — structs, boxing, and when each matters
  • Nullable reference types and the compiler's null-safety system
  • C# 9+ records, extension methods, sealed classes, and IDisposable

C# vs Java — What the Interview Actually Tests

Analogy: C# is like a watch built by copying a Swiss watch (Java), then spending 25 years adding their own innovations until it became its own thing. Both tell time, but the internal mechanisms are increasingly different. A watchmaker who says "they're basically the same" hasn't looked inside one in years.

Both languages share a garbage-collected runtime, strong static typing, and class-based OOP. But the design decisions diverge significantly after that. Here are the differences interviewers specifically probe:

FeatureJavaC#
PropertiesManual getters/settersBuilt-in get/set syntax
Data queryingStream API (Java 8+)LINQ — first-class, richer syntax
async modelCompletableFutureasync/await with Task<T>
Null safetyOptional<T> (optional use)Nullable reference types (compiler-enforced)
Value typesprimitives onlystruct — user-defined value types
EventsObserver pattern manuallydelegate + event — built into the language
Immutable datarecord (Java 16+)record (C# 9+) with with-expressions

Properties are the first thing that trips Java developers. In Java you write getName() and setName(). In C# the property syntax is a first-class citizen:

// Java style — still works in C# but nobody does this
public string GetName() { return _name; }
public void SetName(string value) { _name = value; }

// C# properties — what you should write
public string Name { get; set; }           // auto-property
public string Name { get; private set; }   // read-only from outside
public string Name { get; init; }          // set only during object initialization (C# 9+)

// Computed property — no backing field needed
public string FullName => $"{FirstName} {LastName}";

Events and delegates encode the observer pattern directly into the type system. A delegate is a type-safe function pointer. An event is a delegate with restricted access (only the declaring class can invoke it).

public delegate void OrderShipped(string orderId);

public class OrderService
{
    public event OrderShipped? OnOrderShipped;

    public void Ship(string orderId)
    {
        // ... ship logic
        OnOrderShipped?.Invoke(orderId);  // null-conditional invoke
    }
}

// Subscriber
orderService.OnOrderShipped += id => Console.WriteLine($"Shipped: {id}");

LINQ — SQL for Objects

Analogy: LINQ is SQL, but your data lives in memory instead of a database. The mental model is identical: you describe what you want (filter, transform, sort, group), not how to iterate.

LINQ (Language Integrated Query) is one of C#'s genuine design masterpieces. It works on any type that implements IEnumerable<T>, which means arrays, lists, dictionaries, database queries, and even infinite sequences.

You have two syntaxes — method syntax (preferred in professional code) and query syntax (looks like SQL):

var orders = new List<Order>
{
    new Order { Id = 1, CustomerId = 10, Amount = 250m, Status = "shipped" },
    new Order { Id = 2, CustomerId = 10, Amount = 80m,  Status = "pending" },
    new Order { Id = 3, CustomerId = 11, Amount = 500m, Status = "shipped" },
};

// Method syntax — used in 90% of professional code
var result = orders
    .Where(o => o.Status == "shipped")
    .OrderByDescending(o => o.Amount)
    .Select(o => new { o.Id, o.Amount });

// Query syntax — occasionally clearer for joins and grouping
var result2 = from o in orders
              where o.Status == "shipped"
              orderby o.Amount descending
              select new { o.Id, o.Amount };

The key operators you must know cold:

// Filtering
orders.Where(o => o.Amount > 100)

// Projection
orders.Select(o => o.CustomerId)          // map
orders.SelectMany(o => o.Items)           // flatMap

// Aggregation
orders.Any(o => o.Status == "pending")    // exists?
orders.All(o => o.Amount > 0)            // all match?
orders.Count(o => o.Status == "shipped")
orders.Sum(o => o.Amount)
orders.Average(o => o.Amount)

// Single element
orders.First()                            // throws if empty
orders.FirstOrDefault()                   // returns null/default if empty
orders.Single()                           // throws if != 1 result
orders.SingleOrDefault()

// Sorting
orders.OrderBy(o => o.Amount)
orders.OrderByDescending(o => o.Amount)
    .ThenBy(o => o.Id)

// Grouping
orders.GroupBy(o => o.CustomerId)
      .Select(g => new {
          CustomerId = g.Key,
          Total = g.Sum(o => o.Amount)
      })
Deferred execution — the most misunderstood LINQ concept. LINQ queries are lazy. They don't execute until you enumerate them (with foreach, ToList(), ToArray(), Count(), etc.). This is powerful but creates a trap: 
// DANGER: ToList() inside a loop — executes query N times
foreach (var customerId in customerIds)
{
    // This hits the database (or iterates the collection) once per customer
    var customerOrders = allOrders.Where(o => o.CustomerId == customerId).ToList();
}

// CORRECT: materialize once, then group in memory
var ordersByCustomer = allOrders
    .GroupBy(o => o.CustomerId)
    .ToDictionary(g => g.Key, g => g.ToList());

foreach (var customerId in customerIds)
{
    var customerOrders = ordersByCustomer.GetValueOrDefault(customerId)
                         ?? new List<Order>();
}

async/await in C#

Analogy: Task<T> is C#'s equivalent of JavaScript's Promise<T>. If you understand async/await in JS, you already understand the concept in C# — the mechanism is nearly identical. The syntax differences are minor. 
// JavaScript Promise comparison
// JS:  async function fetchUser(id: string): Promise<User>
// C#:  async Task<User> FetchUserAsync(string id)

// Full C# async example with HttpClient and CancellationToken
public class UserService
{
    private readonly HttpClient _httpClient;

    public UserService(HttpClient httpClient)
    {
        _httpClient = httpClient;
    }

    public async Task<User?> FetchUserAsync(
        string userId,
        CancellationToken cancellationToken = default)
    {
        try
        {
            var response = await _httpClient
                .GetAsync($"/api/users/{userId}", cancellationToken)
                .ConfigureAwait(false);  // important for library code

            response.EnsureSuccessStatusCode();

            var user = await response.Content
                .ReadFromJsonAsync<User>(cancellationToken: cancellationToken)
                .ConfigureAwait(false);

            return user;
        }
        catch (HttpRequestException ex)
        {
            // log ex
            return null;
        }
        catch (OperationCanceledException)
        {
            // caller cancelled — bubble up or return null
            throw;
        }
    }
}
ConfigureAwait(false) — When you await a task in a UI application or ASP.NET (older), the continuation resumes on the original SynchronizationContext (the UI thread or the request context). This is expensive overhead for library code that doesn't need it. Calling .ConfigureAwait(false) tells the runtime: "I don't care which thread continues this — use the thread pool." Library code should always use it. Application-level code (controllers, view models) usually doesn't need to, because you do want to return to the right context.

Parallel async operations with Task.WhenAll:

// Sequential — slow, each awaits before starting the next
var user  = await FetchUserAsync(userId);
var prefs = await FetchPrefsAsync(userId);

// Parallel — both start immediately, wait for both to finish
var (user, prefs) = await Task.WhenAll(
    FetchUserAsync(userId),
    FetchPrefsAsync(userId)
);  // actually need tuple deconstruction via:
var userTask  = FetchUserAsync(userId);
var prefsTask = FetchPrefsAsync(userId);
await Task.WhenAll(userTask, prefsTask);
var user  = await userTask;
var prefs = await prefsTask;

ASP.NET Core — Controllers, DI, and Entity Framework

Analogy: ASP.NET Core's Dependency Injection container is like a hotel concierge. You tell the concierge what you need (an IOrderService), and it figures out how to get it to you — you don't know or care whether it's the same instance everyone shares, a fresh one per request, or a unique one just for you. That's DI lifetimes.

The three DI lifetimes are a guaranteed interview question:

LifetimeHow long it livesUse for
AddSingletonEntire app lifetimeConfig, caches, thread-safe stateless services
AddScopedOne HTTP requestDbContext, services that hold per-request state
AddTransientEvery injectionLightweight stateless services, utilities
Captive dependency bug: Never inject a Scoped service into a Singleton. The Singleton lives forever and captures the Scoped instance at startup — it will never be replaced, leaking stale state across all requests. ASP.NET Core will throw an exception at startup if it detects this. 
// Program.cs — service registration
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseNpgsql(builder.Configuration.GetConnectionString("Default")));

builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.AddSingleton<ICacheService, RedisCacheService>();
builder.Services.AddTransient<IEmailFormatter, HtmlEmailFormatter>();

// Full controller example
[ApiController]
[Route("api/[controller]")]
public class OrdersController : ControllerBase
{
    private readonly IOrderService _orderService;
    private readonly AppDbContext _db;

    public OrdersController(IOrderService orderService, AppDbContext db)
    {
        _orderService = orderService;
        _db = db;
    }

    [HttpGet("{id:int}")]
    public async Task<ActionResult<OrderDto>> GetOrder(
        int id,
        CancellationToken ct)
    {
        var order = await _db.Orders
            .Include(o => o.Items)
            .FirstOrDefaultAsync(o => o.Id == id, ct);

        if (order is null) return NotFound();

        return Ok(new OrderDto(order));
    }

    [HttpPost]
    public async Task<ActionResult<OrderDto>> CreateOrder(
        CreateOrderRequest request,
        CancellationToken ct)
    {
        var order = await _orderService.CreateAsync(request, ct);
        return CreatedAtAction(nameof(GetOrder), new { id = order.Id }, new OrderDto(order));
    }
}

// Entity Framework Core DbContext
public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    public DbSet<Order> Orders => Set<Order>();
    public DbSet<OrderItem> OrderItems => Set<OrderItem>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Order>(entity =>
        {
            entity.HasKey(o => o.Id);
            entity.Property(o => o.Amount).HasPrecision(18, 2);
            entity.HasMany(o => o.Items).WithOne(i => i.Order);
        });
    }
}

Value Types vs Reference Types

Analogy: Reference types are like a shared Google Doc — when you pass it around, everyone edits the same document. Value types are like printing out the doc and handing someone a copy — they can scribble on their copy without affecting yours.

Classes are reference types. Structs are value types. This distinction affects memory layout, performance, and correctness.

// Reference type — class
public class Point3DClass
{
    public double X { get; set; }
    public double Y { get; set; }
    public double Z { get; set; }
}

// Value type — struct
public readonly struct Point3D
{
    public double X { get; }
    public double Y { get; }
    public double Z { get; }

    public Point3D(double x, double y, double z)
        => (X, Y, Z) = (x, y, z);

    public double DistanceTo(Point3D other)
        => Math.Sqrt(
            Math.Pow(X - other.X, 2) +
            Math.Pow(Y - other.Y, 2) +
            Math.Pow(Z - other.Z, 2));
}

// Mutation difference
void ModifyClass(Point3DClass p) { p.X = 99; }  // mutates the original
void ModifyStruct(Point3D p)     { /* can't — readonly struct */ }

var cls = new Point3DClass { X = 1 };
ModifyClass(cls);
Console.WriteLine(cls.X);  // 99 — mutated

var val = new Point3D(1, 2, 3);
ModifyStruct(val);           // copy passed in — original untouched

Use struct when: the type is small (rule of thumb: < 16 bytes), conceptually a value (not an identity), short-lived, and ideally immutable. DateTime, Guid, TimeSpan, Vector2 are all structs for this reason.

Boxing is the silent performance killer: when a value type is stored in an object variable, it gets wrapped in a heap-allocated object. In hot paths, this causes GC pressure.

// Boxing — creates heap allocation
object boxed = 42;          // int is boxed into an object
int unboxed = (int)boxed;   // unboxing — explicit cast required

// Avoid boxing with generics
List<int> ints = new();   // no boxing — generic preserves value type
ArrayList legacy = new();   // boxes every int — avoid this

Nullable Reference Types (C# 8+)

Analogy: Nullable reference types give the compiler a checklist. When you declare string name, the compiler adds it to the "must not be null" list and warns you if you put something that might be null into it. string? name means "this one is allowed to be null — I'll handle it." It's like opt-in null safety.

Enable in your .csproj with <Nullable>enable</Nullable> (on by default in .NET 6+ templates).

// Without nullable enabled, both compile fine — both are traps
string name = null;    // runtime NullReferenceException waiting to happen
string? name = null;   // same behavior, different semantics

// With nullable enabled
string name = null;    // COMPILER WARNING: cannot assign null to non-nullable
string? name = null;   // fine — you've acknowledged it can be null

// The compiler tracks flow:
public void Greet(string? name)
{
    Console.WriteLine(name.Length);      // WARNING: possible null dereference

    if (name is not null)
    {
        Console.WriteLine(name.Length);  // OK — compiler knows name != null here
    }

    Console.WriteLine(name?.Length ?? 0); // null-conditional + null-coalescing: safe
}

// The null-forgiving operator ! — use sparingly
string definitelyNotNull = GetName()!;  // tells compiler "trust me, not null"
// Use only when you know something the compiler doesn't (e.g., after external validation)

Common C# Interview Questions

IEnumerable vs IList vs IQueryable

IEnumerable<T> — forward-only, lazy iteration. No indexing, no count. Works on anything iterable. LINQ operations on it execute in memory.

IList<T> — extends ICollection, adds indexing (list[i]) and Insert/RemoveAt. Already materialized in memory.

IQueryable<T> — extends IEnumerable, but LINQ operations build an expression tree that gets translated to SQL (or another query language) when enumerated. Use for database queries via Entity Framework. 
// IQueryable — query is translated to SQL, runs on the database
IQueryable<Order> query = _db.Orders.Where(o => o.Status == "shipped");
query = query.OrderByDescending(o => o.Amount);  // still SQL, no data fetched yet
var results = await query.ToListAsync();          // NOW it hits the database

// IEnumerable — once you do AsEnumerable(), you're in memory
var memoryQuery = _db.Orders.AsEnumerable()
    .Where(o => o.Status == "shipped");  // fetches ALL orders, filters in C#
    // VERY DIFFERENT — don't do this accidentally

using Statement and IDisposable

// IDisposable pattern — deterministic cleanup of unmanaged resources
public class FileProcessor : IDisposable
{
    private readonly FileStream _stream;
    private bool _disposed;

    public FileProcessor(string path)
    {
        _stream = File.Open(path, FileMode.Open);
    }

    public void Dispose()
    {
        if (!_disposed)
        {
            _stream.Dispose();
            _disposed = true;
        }
    }
}

// using statement — Dispose() called automatically, even on exception
using var processor = new FileProcessor("/data/orders.csv");
// processor.Dispose() called when scope exits

// Classic using block — identical behavior
using (var processor = new FileProcessor("/data/orders.csv"))
{
    // use it
}  // Dispose() called here

Records (C# 9+)

// Record — immutable, value-based equality, with-expressions
public record OrderSummary(int Id, decimal Amount, string Status);

var original = new OrderSummary(1, 250m, "pending");
var updated  = original with { Status = "shipped" }; // creates new instance

Console.WriteLine(original == updated);  // false — different Status
Console.WriteLine(original.Id == updated.Id);  // true

Extension Methods

// Extension methods — add methods to types you don't own
public static class StringExtensions
{
    public static bool IsValidEmail(this string value)
        => !string.IsNullOrWhiteSpace(value) && value.Contains('@');

    public static string Truncate(this string value, int maxLength)
        => value.Length <= maxLength ? value : value[..maxLength] + "...";
}

// Usage — reads naturally
"user@example.com".IsValidEmail();  // true
"This is a very long string".Truncate(10);  // "This is a ..."

Senior vs Junior Comparison

Junior answer — "Explain async/await in C#"

"async/await lets you write asynchronous code without callbacks. You put async on a method and await on the thing that takes time. It makes the code easier to read."

Senior answer

"async/await is syntactic sugar over the Task Parallel Library. The compiler rewrites an async method as a state machine — each await point is a state. When the awaited Task completes, execution resumes from that state on a thread pool thread (unless a SynchronizationContext is captured).

In library code I always use ConfigureAwait(false) to avoid capturing the caller's SynchronizationContext — this prevents deadlocks in environments with synchronization contexts like WinForms or classic ASP.NET, and reduces overhead in ASP.NET Core even though it doesn't deadlock there.

I use CancellationToken throughout the call chain so that if the HTTP request is cancelled — say the user navigated away — we stop all downstream work immediately rather than burning CPU on a response nobody will read. Task.WhenAll lets me fire parallel async operations and collect all results in one await rather than awaiting sequentially."

Practice Questions

Beginner

Q: What is the difference between First() and FirstOrDefault() in LINQ?

A: First() throws an InvalidOperationException if the sequence is empty. FirstOrDefault() returns default(T) — which is null for reference types and 0 for value types — if the sequence is empty. Use First() when an empty result is a programming error. Use FirstOrDefault() when empty is a valid expected case and you'll null-check the result.

Intermediate

Q: What is the difference between AddScoped, AddTransient, and AddSingleton in ASP.NET Core DI? Give a real example of when each is appropriate.

A: Singleton lives for the entire application lifetime — use for IConfiguration, an IMemoryCache, or a connection pool wrapper that is thread-safe and stateless across requests. Scoped lives for one HTTP request — use for DbContext (Entity Framework) because it tracks entity state per request and is not thread-safe. Transient lives for the duration of a single injection — use for a lightweight formatter or validator that carries no state. The key danger is injecting a Scoped service into a Singleton, which captures a stale instance forever. ASP.NET Core detects and throws on this at startup in development.

Advanced

Q: Implement a generic in-memory cache with expiry using C# that is thread-safe. Use ConcurrentDictionary and explain why Dictionary alone is insufficient.

using System.Collections.Concurrent;

public class ExpiringCache<TKey, TValue> where TKey : notnull
{
    private record CacheEntry(TValue Value, DateTimeOffset ExpiresAt);

    private readonly ConcurrentDictionary<TKey, CacheEntry> _store = new();

    public void Set(TKey key, TValue value, TimeSpan ttl)
    {
        var entry = new CacheEntry(value, DateTimeOffset.UtcNow + ttl);
        _store[key] = entry;
    }

    public bool TryGet(TKey key, out TValue? value)
    {
        if (_store.TryGetValue(key, out var entry) &&
            entry.ExpiresAt > DateTimeOffset.UtcNow)
        {
            value = entry.Value;
            return true;
        }

        // Lazy eviction — remove stale entry
        _store.TryRemove(key, out _);
        value = default;
        return false;
    }

    public TValue GetOrAdd(TKey key, Func<TKey, TValue> factory, TimeSpan ttl)
    {
        if (TryGet(key, out var cached))
            return cached!;

        var value = factory(key);
        Set(key, value, ttl);
        return value;
    }

    public void Evict()
    {
        var now = DateTimeOffset.UtcNow;
        foreach (var (key, entry) in _store)
        {
            if (entry.ExpiresAt <= now)
                _store.TryRemove(key, out _);
        }
    }
}

// Usage
var cache = new ExpiringCache<string, UserProfile>();
var profile = cache.GetOrAdd(
    "user-42",
    id => userService.FetchProfile(id),  // only called on cache miss
    TimeSpan.FromMinutes(5)
);

Why not Dictionary? Dictionary<K,V> is not thread-safe. Concurrent reads are usually fine, but a read concurrent with a write can corrupt internal state and throw InvalidOperationException or loop forever. ConcurrentDictionary uses fine-grained locking per bucket, making reads lock-free and writes lock only the affected bucket. For a true high-throughput cache with atomic get-or-add semantics, ConcurrentDictionary.GetOrAdd with a factory overload is the right primitive — though the factory may be called more than once under contention, so it must be idempotent.

Chapter 26 Summary

  • C# diverges from Java significantly: properties, LINQ, records, nullable reference types, and events are first-class language features, not library add-ons.
  • LINQ's deferred execution means queries don't run until enumerated — calling ToList() inside a loop is a classic performance bug.
  • async/await in C# is a state machine over the Task Parallel Library; use ConfigureAwait(false) in library code and pass CancellationToken throughout your call chains.
  • DI lifetimes are a common interview question: Singleton (app), Scoped (request), Transient (injection) — never inject a shorter lifetime into a longer one.
  • Nullable reference types (C# 8+) make null safety compiler-enforced; IQueryable vs IEnumerable determines whether filtering happens in the database or in memory — a critical performance distinction.
