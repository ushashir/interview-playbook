Go Deep Dive

Go was designed by engineers frustrated with C++ and Java. Its philosophy: if a feature makes the language harder to learn or understand, don't add it. The result is a language that's surprisingly powerful for how small it is.

What you'll learn

  • Goroutines and channels — Go's concurrency model and why it scales to hundreds of thousands
  • Error handling philosophy — why Go's explicit errors beat hidden exceptions in production
  • Interfaces with implicit satisfaction — the most elegant interface system in mainstream languages
  • Go modules — go.mod, go.sum, and dependency management
  • Building REST APIs with net/http and Gin
  • Goroutine leaks, defer, panic, make vs new — the interview gotchas
  • Goroutines vs threads — the M:N scheduling model explained clearly

Goroutines and Channels

Analogy: Think of a factory. Workers are goroutines — cheap to hire, lightweight. The conveyor belt between them is a channel — it moves products (data) from one worker to the next. An unbuffered channel is a belt with no room to queue: the sender must wait until the receiver picks up. A buffered channel is a belt with a small staging area: the sender can drop off a few items without waiting.

Go is like a perfectly balanced kitchen knife. No 37 attachments. One blade, perfectly balanced, faster and cleaner than the Swiss Army knife. Go prioritizes clarity over cleverness — and in production systems, clarity wins.

A goroutine costs roughly 2–8 KB of stack memory (it grows dynamically) versus ~1 MB for an OS thread. You can run hundreds of thousands of goroutines on a single machine that would be impossible with OS threads.

package main

import (
    "fmt"
    "sync"
)

func worker(id int, jobs <-chan int, results chan<- int, wg *sync.WaitGroup) {
    defer wg.Done()
    for job := range jobs {
        result := job * job  // expensive computation simulated
        results <- result
    }
}

func main() {
    const numWorkers = 5
    const numJobs    = 20

    jobs    := make(chan int, numJobs)
    results := make(chan int, numJobs)

    var wg sync.WaitGroup

    // Spawn worker pool
    for i := 1; i <= numWorkers; i++ {
        wg.Add(1)
        go worker(i, jobs, results, &wg)
    }

    // Send jobs
    for j := 1; j <= numJobs; j++ {
        jobs <- j
    }
    close(jobs)  // signals workers: no more jobs coming

    // Close results once all workers finish
    go func() {
        wg.Wait()
        close(results)
    }()

    // Collect results
    for r := range results {
        fmt.Println(r)
    }
}

The select Statement

Analogy: select is like a traffic controller at a multi-road intersection — it waits for whichever road (channel) has a car (data) and lets that one through. If multiple are ready at once, it picks one at random. If none are ready and there's a default case, it takes that immediately (non-blocking). 
func fanIn(ch1, ch2 <-chan string) <-chan string {
    merged := make(chan string)
    go func() {
        defer close(merged)
        for {
            select {
            case msg, ok := <-ch1:
                if !ok { ch1 = nil }  // nil channel blocks forever — effectively remove it
                merged <- msg
            case msg, ok := <-ch2:
                if !ok { ch2 = nil }
                merged <- msg
            }
            if ch1 == nil && ch2 == nil {
                return
            }
        }
    }()
    return merged
}

Goroutine Leaks and Context Cancellation

Goroutine leak: A goroutine blocked on a channel nobody will ever write to or read from runs forever, consuming memory. This is the most common Go production bug. Always use context.WithCancel or context.WithTimeout to give goroutines an escape hatch. 
package main

import (
    "context"
    "fmt"
    "time"
)

// LEAK: this goroutine runs forever if nobody reads from done
func leakySearch(query string) <-chan string {
    done := make(chan string)
    go func() {
        result := expensiveSearch(query)  // takes a long time
        done <- result                   // if nobody reads this, goroutine hangs
    }()
    return done
}

// FIX: accept context, return early if cancelled
func safeSearch(ctx context.Context, query string) <-chan string {
    results := make(chan string, 1)  // buffered: sender won't block even if nobody reads
    go func() {
        defer close(results)
        result := expensiveSearch(query)
        select {
        case results <- result:  // send if receiver is ready
        case <-ctx.Done():       // bail out if context cancelled
            return
        }
    }()
    return results
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    defer cancel()  // always defer cancel to release resources

    results := safeSearch(ctx, "go concurrency patterns")
    select {
    case result := <-results:
        fmt.Println("Got:", result)
    case <-ctx.Done():
        fmt.Println("Search timed out:", ctx.Err())
    }
}

Error Handling in Go

Analogy: Go's error handling is like a cashier giving you a receipt with every purchase. The receipt either says "success" or lists exactly what went wrong. There's no hidden alarm that triggers somewhere else in the building — the problem is right in your hand, right now.

Go has no exceptions. Every function that can fail returns (result, error) as its last two values. This is a deliberate design decision: it makes all fallible operations explicit. You cannot accidentally ignore an error if you're paying attention to the return values.

package main

import (
    "errors"
    "fmt"
    "strconv"
)

// Custom error type — implements the error interface
type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation failed on %s: %s", e.Field, e.Message)
}

// Sentinel error — compare with errors.Is()
var ErrNotFound = errors.New("resource not found")

func parsePositiveInt(s string) (int, error) {
    n, err := strconv.Atoi(s)
    if err != nil {
        // Wrap with context using %w — preserves the original error for errors.Is/As
        return 0, fmt.Errorf("parsePositiveInt: %w", err)
    }
    if n <= 0 {
        return 0, &ValidationError{Field: "value", Message: "must be positive"}
    }
    return n, nil
}

func processInput(s string) error {
    n, err := parsePositiveInt(s)
    if err != nil {
        return fmt.Errorf("processInput: %w", err)  // wrap again, adding context
    }
    fmt.Println("Processed:", n)
    return nil
}

func main() {
    err := processInput("-5")
    if err != nil {
        // errors.Is checks the entire error chain (unwraps via %w)
        var valErr *ValidationError
        if errors.As(err, &valErr) {
            fmt.Println("Validation error on field:", valErr.Field)
        }
        fmt.Println("Full error:", err)
        // Output: processInput: parsePositiveInt: validation failed on value: must be positive
    }

    // errors.Is for sentinel errors
    err2 := fmt.Errorf("lookup: %w", ErrNotFound)
    fmt.Println(errors.Is(err2, ErrNotFound))  // true — unwraps the chain
}
Error wrapping convention: Always wrap errors with context using fmt.Errorf("operationName: %w", err). This builds a call-stack-like chain. At the top level you log the full error string and get a complete picture of what happened without a stack trace. Never swallow errors silently — at minimum, log them.

Interfaces — Implicit Satisfaction

Analogy: In Java or C#, implementing an interface is like signing a contract before doing the job. In Go, it's like a job interview — if you can demonstrate the skills on the spot, you're hired, no contract needed. A type satisfies an interface simply by having the right methods.

This has a profound consequence: Go interfaces are defined by the consumer, not the producer. You can define an interface for any type you don't own and use it polymorphically without modifying that type's package.

package main

import (
    "bytes"
    "fmt"
    "os"
)

// io.Writer is defined in the standard library as:
// type Writer interface {
//     Write(p []byte) (n int, err error)
// }

// Our custom logger accepts any io.Writer
type Logger struct {
    out    io.Writer
    prefix string
}

func NewLogger(out io.Writer, prefix string) *Logger {
    return &Logger{out: out, prefix: prefix}
}

func (l *Logger) Log(message string) {
    fmt.Fprintf(l.out, "[%s] %s\n", l.prefix, message)
}

// All of these satisfy io.Writer without any declaration:
//   os.Stdout       (*os.File has Write method)
//   os.Stderr
//   bytes.Buffer    (*bytes.Buffer has Write method)
//   http.ResponseWriter

func main() {
    // Log to stdout
    stdLogger := NewLogger(os.Stdout, "INFO")
    stdLogger.Log("server started")

    // Log to a buffer (useful in tests)
    var buf bytes.Buffer
    bufLogger := NewLogger(&buf, "TEST")
    bufLogger.Log("test message")
    fmt.Println("Captured:", buf.String())
}

// Define a small, focused interface yourself
type Storer interface {
    Get(key string) ([]byte, error)
    Set(key string, value []byte) error
    Delete(key string) error
}

// Now any struct with these three methods satisfies Storer
// — whether it's a Redis client, a Postgres implementation,
//   an in-memory map for tests, or a file system adapter
Interface size principle: The Go standard library's most used interfaces are tiny — io.Reader has one method, io.Writer has one method, error has one method. Prefer small interfaces. A large interface is a sign you're designing around implementation rather than behavior.

Building REST APIs — net/http and Gin

Analogy: The standard library's net/http is a powerful but bare-bones toolkit — like building a desk from lumber. Gin is a pre-cut kit with hardware included. Both produce the same desk; Gin just requires less measuring. 
package main

import (
    "net/http"
    "strconv"

    "github.com/gin-gonic/gin"
    "go.uber.org/zap"
)

type OrderService interface {
    GetOrder(id int) (*Order, error)
    CreateOrder(req CreateOrderRequest) (*Order, error)
}

type Order struct {
    ID     int     `json:"id"`
    Amount float64 `json:"amount"`
    Status string  `json:"status"`
}

type CreateOrderRequest struct {
    Amount float64 `json:"amount" binding:"required,gt=0"`
    Note   string  `json:"note"`
}

type OrderHandler struct {
    svc    OrderService
    logger *zap.Logger
}

func NewRouter(svc OrderService, logger *zap.Logger) http.Handler {
    r := gin.New()

    // Middleware
    r.Use(gin.Recovery())  // recover from panics, return 500
    r.Use(func(c *gin.Context) {
        logger.Info("request",
            zap.String("method", c.Request.Method),
            zap.String("path", c.Request.URL.Path),
        )
        c.Next()
    })

    h := &OrderHandler{svc: svc, logger: logger}

    // Route groups
    api := r.Group("/api/v1")
    {
        orders := api.Group("/orders")
        orders.GET("/:id", h.GetOrder)
        orders.POST("", h.CreateOrder)
    }

    return r
}

func (h *OrderHandler) GetOrder(c *gin.Context) {
    idStr := c.Param("id")
    id, err := strconv.Atoi(idStr)
    if err != nil {
        c.JSON(http.StatusBadRequest, gin.H{"error": "invalid order id"})
        return
    }

    order, err := h.svc.GetOrder(id)
    if err != nil {
        h.logger.Error("GetOrder failed", zap.Int("id", id), zap.Error(err))
        c.JSON(http.StatusNotFound, gin.H{"error": "order not found"})
        return
    }

    c.JSON(http.StatusOK, order)
}

func (h *OrderHandler) CreateOrder(c *gin.Context) {
    var req CreateOrderRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        // ShouldBindJSON validates struct tags (binding:"required,gt=0")
        c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
        return
    }

    order, err := h.svc.CreateOrder(req)
    if err != nil {
        h.logger.Error("CreateOrder failed", zap.Error(err))
        c.JSON(http.StatusInternalServerError, gin.H{"error": "failed to create order"})
        return
    }

    c.JSON(http.StatusCreated, order)
}

Common Interview Gotchas

defer — LIFO Execution

Analogy: defer is like putting something on a stack of sticky notes for "when I leave the room." The last note you stick on is the first you read on the way out. 
func cleanup() {
    defer fmt.Println("first defer — runs LAST")
    defer fmt.Println("second defer — runs SECOND")
    defer fmt.Println("third defer — runs FIRST")
    fmt.Println("doing work")
}
// Output:
// doing work
// third defer — runs FIRST
// second defer — runs SECOND
// first defer — runs LAST

// Classic defer for resource cleanup
func readFile(path string) ([]byte, error) {
    f, err := os.Open(path)
    if err != nil {
        return nil, fmt.Errorf("readFile: %w", err)
    }
    defer f.Close()  // always closes, even if error below
    return io.ReadAll(f)
}

Maps Are Not Safe for Concurrent Use

// RACE CONDITION — concurrent reads AND writes on the same map = undefined behavior
m := make(map[string]int)
go func() { m["key"] = 1 }()
go func() { _ = m["key"] }()

// FIX 1: sync.Mutex
type SafeMap struct {
    mu sync.RWMutex
    m  map[string]int
}
func (s *SafeMap) Set(k string, v int) {
    s.mu.Lock(); defer s.mu.Unlock()
    s.m[k] = v
}
func (s *SafeMap) Get(k string) (int, bool) {
    s.mu.RLock(); defer s.mu.RUnlock()
    v, ok := s.m[k]; return v, ok
}

// FIX 2: sync.Map (good for mostly-reads or many-keys workloads)
var sm sync.Map
sm.Store("key", 42)
if v, ok := sm.Load("key"); ok {
    fmt.Println(v.(int))
}

make vs new

// new(T) allocates zeroed memory for T, returns *T
p := new(int)   // *int pointing to 0
*p = 42

// make(T, args) initializes slices, maps, channels — returns T (not *T)
s := make([]int, 0, 10)    // slice: len=0, cap=10
m := make(map[string]int)  // initialized map — ready to write
ch := make(chan int, 5)     // buffered channel

// WRONG — common mistake
m2 := new(map[string]int)  // *map[string]int — the map itself is nil!
// (*m2)["key"] = 1  // panic: assignment to nil map

panic vs error

// panic: for programmer errors — things that should never happen
func mustPositive(n int) int {
    if n <= 0 {
        panic(fmt.Sprintf("mustPositive: got %d", n))
    }
    return n
}

// recover: catch a panic inside a deferred function
func safeDiv(a, b int) (result int, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("recovered panic: %v", r)
        }
    }()
    return a / b, nil
}

// RULE: return error for expected failure conditions
//        use panic only for invariant violations (nil pointer you guaranteed wouldn't be nil, etc.)
//        HTTP servers should recover from panics in middleware so one bad request doesn't kill the server

Goroutines vs Threads — The Scheduling Model

Analogy: OS threads are like full-time employees — expensive to hire, have a fixed desk, and go through HR. Goroutines are like freelancers — cheap, flexible, and the Go scheduler (the agency) decides which freelancer sits at which desk (OS thread) at any moment.
OS ThreadGoroutine
Stack size~1 MB fixed2 KB, grows dynamically to ~1 GB
Creation costSyscall, ~1ms~1 microsecond
Practical limitThousandsHundreds of thousands
SchedulingOS preemptiveGo runtime cooperative + preemptive
Context switchKernel space, expensiveUser space, cheap

Go uses an M:N threading model: M goroutines are multiplexed onto N OS threads by the Go runtime scheduler (GOMAXPROCS controls N, defaulting to CPU count). When a goroutine blocks on I/O or a channel, the scheduler parks it and runs another goroutine on that OS thread — the OS thread is never idle waiting for I/O.

// GOMAXPROCS — number of OS threads running Go code simultaneously
import "runtime"

func main() {
    // By default = number of logical CPUs
    fmt.Println("Logical CPUs:", runtime.NumCPU())
    fmt.Println("GOMAXPROCS:", runtime.GOMAXPROCS(0))  // 0 = query, don't change

    // Goroutine count — useful for leak detection
    fmt.Println("Goroutines:", runtime.NumGoroutine())
}

Senior vs Junior Comparison

Junior answer — "Explain goroutines vs OS threads"

"Goroutines are like threads but lighter. You can have a lot more of them. They're cheap to create."

Senior answer

"Goroutines use an M:N threading model — the Go runtime scheduler maps M goroutines onto N OS threads, where N is typically GOMAXPROCS (defaults to CPU count). Each goroutine starts with a 2 KB stack that grows dynamically, versus a fixed 1 MB for an OS thread. You can realistically run hundreds of thousands of goroutines.

The critical difference is how blocking works. When a goroutine blocks on a channel or network I/O, the Go scheduler detects this and runs another goroutine on that OS thread — the OS thread itself is never blocked on I/O. The runtime uses non-blocking syscalls with epoll/kqueue under the hood. This is why Go services handle massive concurrency without the C10k problem.

The biggest production pitfall is goroutine leaks — a goroutine blocked on a channel nobody will write to or read from runs forever. We prevent this by propagating context.Context through all long-running operations and selecting on ctx.Done() alongside any channel operation. You can detect leaks with runtime.NumGoroutine() in tests or with pprof's /debug/pprof/goroutine endpoint in production."

Practice Questions

Beginner

Q: What is the difference between a buffered and an unbuffered channel in Go?

A: An unbuffered channel (make(chan int)) requires the sender and receiver to be ready at exactly the same time. The send blocks until a receiver is ready, and the receive blocks until a sender sends. Think of it as a handshake — both hands must be extended simultaneously. A buffered channel (make(chan int, 5)) has an internal queue. The sender can deposit up to 5 items without a receiver being ready; it only blocks when the buffer is full. The receiver only blocks when the buffer is empty. Use unbuffered channels for synchronization and rendezvous; use buffered channels to decouple sender and receiver speeds.

Intermediate

Q: What is a goroutine leak and how do you prevent it? Give a specific code example.

A: A goroutine leak occurs when a goroutine is stuck waiting on a channel that will never receive data or is never drained — it continues consuming memory and scheduler resources indefinitely. Prevention: (1) Always pass a context.Context to long-running goroutines and select on ctx.Done(). (2) Use buffered channels (size 1 is often enough) so that a goroutine can send its result even if the receiver has moved on. (3) Use context.WithTimeout for all operations with an expected duration. In tests, assert runtime.NumGoroutine() before and after to detect leaks. In production, expose /debug/pprof/goroutine?debug=1 and monitor the goroutine count metric.

Advanced

Q: Implement a concurrent rate limiter in Go that allows at most N requests per second using the token bucket algorithm. The implementation must be safe for concurrent use.

package ratelimit

import (
    "context"
    "sync"
    "time"
)

// TokenBucket implements a thread-safe token bucket rate limiter.
// The bucket holds up to Capacity tokens. Tokens are added at Rate per second.
// Each call to Allow() consumes one token. If no token is available, it either
// waits (via Wait) or returns false (via Allow).
type TokenBucket struct {
    capacity  float64
    tokens    float64
    rate      float64     // tokens per second
    lastFill  time.Time
    mu        sync.Mutex
}

// New creates a rate limiter allowing `rate` events per second with a
// burst capacity of `capacity`.
func New(rate float64, capacity float64) *TokenBucket {
    return &TokenBucket{
        capacity: capacity,
        tokens:   capacity, // start full
        rate:     rate,
        lastFill: time.Now(),
    }
}

// refill adds tokens proportional to elapsed time since last call.
// Must be called with mu held.
func (tb *TokenBucket) refill() {
    now := time.Now()
    elapsed := now.Sub(tb.lastFill).Seconds()
    tb.tokens = min(tb.capacity, tb.tokens+elapsed*tb.rate)
    tb.lastFill = now
}

// Allow returns true if a token is available and consumes it.
// Returns false immediately if the bucket is empty (non-blocking).
func (tb *TokenBucket) Allow() bool {
    tb.mu.Lock()
    defer tb.mu.Unlock()

    tb.refill()
    if tb.tokens >= 1 {
        tb.tokens--
        return true
    }
    return false
}

// Wait blocks until a token is available or the context is cancelled.
func (tb *TokenBucket) Wait(ctx context.Context) error {
    for {
        tb.mu.Lock()
        tb.refill()
        if tb.tokens >= 1 {
            tb.tokens--
            tb.mu.Unlock()
            return nil
        }
        // Calculate how long until the next token arrives
        waitFor := time.Duration((1-tb.tokens)/tb.rate * float64(time.Second))
        tb.mu.Unlock()

        select {
        case <-ctx.Done():
            return ctx.Err()
        case <-time.After(waitFor):
            // retry
        }
    }
}

func min(a, b float64) float64 {
    if a < b { return a }
    return b
}

// Usage example
// limiter := ratelimit.New(100, 200)  // 100 req/s, burst up to 200
//
// func handler(ctx context.Context) error {
//     if err := limiter.Wait(ctx); err != nil {
//         return err  // context cancelled
//     }
//     return doWork()
// }

Key design decisions: The mutex protects both the token count and the last-fill timestamp — they must be updated atomically or you'll double-count elapsed time on concurrent calls. Refilling on every check (lazy refill) is simpler than a background ticker and avoids a goroutine. The Wait method calculates how long to sleep rather than busy-polling, reducing CPU usage. Passing a context.Context allows the caller to cancel waiting (e.g., if the HTTP request is cancelled). A burst capacity larger than the rate allows absorbing short spikes without dropping traffic — critical for real-world workloads.

Chapter 27 Summary

  • Goroutines are multiplexed onto OS threads by the Go scheduler (M:N model) — 2 KB stack, microsecond creation, suitable for hundreds of thousands concurrent; always propagate context to avoid goroutine leaks.
  • Error handling is explicit by design: every fallible function returns (result, error); wrap errors with fmt.Errorf("%w", err) to build a context chain; use errors.Is and errors.As for type checking.
  • Interfaces are satisfied implicitly — if a type has the required methods it satisfies the interface, enabling powerful composition and easy testing without modifying third-party packages.
  • Maps are not safe for concurrent use — use sync.Mutex with a wrapper or sync.Map for concurrent access; defer executes in LIFO order at function return, not at the point of declaration.
  • Go prioritizes clarity: small interfaces, explicit errors, and no generics bloat (pre-Go 1.18) make the codebase readable by anyone on the team — a key production advantage over feature-heavy languages.
