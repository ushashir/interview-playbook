TypeScript Deep Dive

TypeScript is not a different language. It's JavaScript with a spell-checker installed and the ability to read your mind. The spell-checker catches errors before the browser does. Both together mean fewer 2am production surprises.

What you'll learn

  • Structural typing and why TypeScript's type system is fundamentally different from Java or C#
  • Type narrowing techniques: typeof, instanceof, in, type guards, and discriminated unions
  • Generics from first principles through generic constraints and repository patterns
  • The critical difference between unknown and any — and why any is a code smell
  • All major utility types with real NestJS production examples
  • Discriminated unions for making impossible states unrepresentable
  • interface vs type: practical rules and when each is required
  • NestJS decorators, DI, class-validator, and full DTO validation

1. Structural Typing vs Nominal Typing

Analogy — The Dress Code: A restaurant has a dress code: "collared shirt and dress shoes." They don't care whether you bought your shirt from Ralph Lauren or a market stall. If it has a collar, it qualifies. TypeScript works the same way. It doesn't care about the label on the tin — only the shape of what's inside.

Most statically typed languages use nominal typing: a class is only compatible with a type if it explicitly declares that it implements it. Java requires implements Runnable. TypeScript uses structural typing: if an object has the right shape (the right properties and method signatures), it satisfies the type — regardless of what it's called or whether it was explicitly declared to implement anything.

interface Serializable {
  serialize(): string;
}

// No "implements Serializable" — yet this works perfectly
class UserSession {
  constructor(
    public userId: string,
    public token: string,
    public expiresAt: Date
  ) {}

  serialize(): string {
    return JSON.stringify({
      userId: this.userId,
      token: this.token,
      expiresAt: this.expiresAt.toISOString(),
    });
  }
}

function saveToCache(item: Serializable): void {
  const data = item.serialize();
  console.log('Saving to cache:', data);
}

const session = new UserSession('u123', 'tok_abc', new Date());
saveToCache(session); // TypeScript is perfectly happy — shape matches

This is powerful for composition. You can define interfaces after the fact, test code with mock objects that just happen to have the right shape, and use third-party classes without touching their source code.

2. Type Narrowing

Analogy — The Detective: Sherlock Holmes doesn't say "I know nothing about this person." He starts with broad information (a human being) and narrows it down with each clue: military posture (soldier), tan line (served abroad), calluses (manual work). TypeScript's type narrowing works identically — you start with a broad type and use checks to narrow it to something specific.

typeof narrowing

function formatId(id: string | number): string {
  if (typeof id === 'string') {
    // TypeScript knows: id is string here
    return id.toUpperCase();
  }
  // TypeScript knows: id is number here
  return id.toFixed(0);
}

instanceof narrowing

function handleError(err: unknown): string {
  if (err instanceof Error) {
    // TypeScript knows: err has .message, .stack, etc.
    return err.message;
  }
  return String(err);
}

in operator narrowing

interface Cat { meow(): void }
interface Dog { bark(): void }

function makeNoise(animal: Cat | Dog): void {
  if ('meow' in animal) {
    animal.meow(); // TypeScript: Cat
  } else {
    animal.bark(); // TypeScript: Dog
  }
}

User-defined type guards (is operator)

interface AdminUser {
  role: 'admin';
  permissions: string[];
}

interface RegularUser {
  role: 'user';
}

type User = AdminUser | RegularUser;

// The "is" operator creates a type predicate — tells TS what narrowing means
function isAdmin(user: User): user is AdminUser {
  return user.role === 'admin';
}

function getPermissions(user: User): string[] {
  if (isAdmin(user)) {
    return user.permissions; // TypeScript knows: AdminUser
  }
  return [];
}

Discriminated unions with switch/case

type Shape =
  | { kind: 'circle'; radius: number }
  | { kind: 'rectangle'; width: number; height: number }
  | { kind: 'triangle'; base: number; height: number };

function area(shape: Shape): number {
  switch (shape.kind) {
    case 'circle':
      return Math.PI * shape.radius ** 2;
    case 'rectangle':
      return shape.width * shape.height;
    case 'triangle':
      return 0.5 * shape.base * shape.height;
    default:
      // Exhaustiveness check — TypeScript will error if a case is unhandled
      const _exhaustive: never = shape;
      throw new Error(`Unhandled shape: ${_exhaustive}`);
  }
}

3. Generics

Analogy — Labeled Storage Boxes: Imagine a warehouse with generic boxes. A plain box holds anything. But once you label it "BOOKS ONLY," the warehouse system enforces it — you can't accidentally shove a lamp inside. Generics let you write one piece of code that works for any type, while the caller chooses what label goes on the box.

Basic generic function

function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

const name = first(['Alice', 'Bob']); // TypeScript infers: string | undefined
const score = first([98, 87, 76]);    // TypeScript infers: number | undefined

Generic constraints with extends

function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { id: 1, name: 'Alice', email: 'alice@example.com' };
const email = getProperty(user, 'email'); // string — type-safe key access
// getProperty(user, 'password');         // TypeScript ERROR — key doesn't exist

Generic repository pattern (NestJS-ready)

interface Repository<T, ID = number> {
  findById(id: ID): Promise<T | null>;
  findAll(): Promise<T[]>;
  save(entity: T): Promise<T>;
  delete(id: ID): Promise<void>;
}

// A concrete implementation for users
@Injectable()
class UserRepository implements Repository<User, string> {
  constructor(
    @InjectRepository(UserEntity)
    private readonly repo: TypeOrmRepository<UserEntity>
  ) {}

  async findById(id: string): Promise<User | null> {
    const entity = await this.repo.findOne({ where: { id } });
    return entity ? this.toDomain(entity) : null;
  }

  async findAll(): Promise<User[]> {
    const entities = await this.repo.find();
    return entities.map(this.toDomain);
  }

  async save(user: User): Promise<User> {
    const saved = await this.repo.save(this.toEntity(user));
    return this.toDomain(saved);
  }

  async delete(id: string): Promise<void> {
    await this.repo.delete(id);
  }

  private toDomain(e: UserEntity): User { /* mapping */ return e as unknown as User; }
  private toEntity(u: User): UserEntity { /* mapping */ return u as unknown as UserEntity; }
}

4. unknown vs any — The Critical Section

Analogy — Package Labels: You receive two packages. The first is labeled "STUFF" — you reach in, grab whatever, use it however, and hope for the best. That's any: the compiler steps aside and trusts you completely. The second is labeled "UNKNOWN CONTENTS" — before you use anything, you open it, check what's inside, and handle it appropriately. That's unknown: safe by default.
Warning: any silently disables TypeScript's type checking. Every any in your codebase is a lie you're telling the compiler — and lies compound at runtime.

any disables type checking — leading to runtime crashes

function processUserInput_UNSAFE(input: any): string {
  // TypeScript won't complain — even though this crashes at runtime
  // if input is a number, boolean, null, or object
  return input.toUpperCase(); // Runtime TypeError: input.toUpperCase is not a function
}

processUserInput_UNSAFE(42);    // CRASH at runtime
processUserInput_UNSAFE(null);  // CRASH at runtime

unknown forces narrowing — type safe

function processUserInput_SAFE(input: unknown): string {
  // TypeScript FORCES you to check before using
  if (typeof input === 'string') {
    return input.toUpperCase(); // Safe — narrowed to string
  }
  if (typeof input === 'number') {
    return input.toFixed(2);    // Safe — narrowed to number
  }
  throw new Error(`Unsupported input type: ${typeof input}`);
}

processUserInput_SAFE('hello'); // "HELLO"
processUserInput_SAFE(42);      // "42.00"
// processUserInput_SAFE(null); // throws descriptive error — predictable

When to use unknown

// 1. JSON.parse always returns any — cast to unknown immediately
function safeParse(json: string): unknown {
  return JSON.parse(json) as unknown;
}

// 2. API responses from external sources
async function fetchExternalData(url: string): Promise<unknown> {
  const res = await fetch(url);
  return res.json() as unknown; // Treat as unknown, validate before using
}

// 3. Error catch blocks — errors can be anything
function handleCaughtError(err: unknown): string {
  if (err instanceof Error) return err.message;
  if (typeof err === 'string') return err;
  return 'An unknown error occurred';
}
Rule: Use unknown everywhere you'd be tempted to use any. If you find yourself reaching for any in production code, stop — it's a code smell. The only legitimate use of any is during incremental JavaScript-to-TypeScript migration, and even then, mark those spots with // TODO: remove any comments.

5. Utility Types — Real-World Examples

Analogy — Power Tools for Types: You could build a table with hand tools. Or you could use a circular saw, a sander, and a drill press. Utility types are TypeScript's power tools — they derive new types from existing ones automatically, so you don't write the same shape twice with slight variations. 
// Base entity — the source of truth
interface User {
  id: string;
  email: string;
  password: string;
  name: string;
  role: 'admin' | 'user';
  createdAt: Date;
}

// Partial<T> — for PATCH requests: all fields optional
type UpdateUserDto = Partial<Pick<User, 'email' | 'name' | 'role'>>;
// Result: { email?: string; name?: string; role?: 'admin' | 'user' }

// Required<T> — enforce all fields (useful for post-validation shapes)
type ValidatedUser = Required<User>;

// Pick<T, Keys> — select subset for API response (exclude internals)
type UserProfile = Pick<User, 'id' | 'name' | 'email' | 'role'>;
// Result: { id: string; name: string; email: string; role: ... }

// Omit<T, Keys> — exclude sensitive fields from public response
type PublicUser = Omit<User, 'password' | 'createdAt'>;
// Result: { id: string; email: string; name: string; role: ... }

// Record<K, V> — typed dictionary for config objects and cache maps
type RolePermissions = Record<User['role'], string[]>;
const permissions: RolePermissions = {
  admin: ['read', 'write', 'delete', 'manage-users'],
  user: ['read'],
};

// ReturnType<F> — extract what a function returns (useful for mocks)
async function findUser(id: string): Promise<PublicUser | null> {
  // Implementation...
  return null;
}
type FindUserResult = ReturnType<typeof findUser>;
// Resolves to: Promise<PublicUser | null>

Full NestJS DTO hierarchy using utility types

import { IsString, IsEmail, IsEnum, IsOptional, MinLength } from 'class-validator';

// Base class captures the full shape with validators
class BaseUserDto {
  @IsEmail()
  email!: string;

  @IsString()
  @MinLength(8)
  password!: string;

  @IsString()
  name!: string;

  @IsEnum(['admin', 'user'])
  role!: 'admin' | 'user';
}

// CreateUserDto — all fields required (inherits from base)
export class CreateUserDto extends BaseUserDto {}

// UpdateUserDto — all fields optional for PATCH
// Use PartialType from @nestjs/mapped-types (preserves validators!)
import { PartialType, OmitType } from '@nestjs/mapped-types';

export class UpdateUserDto extends PartialType(
  OmitType(BaseUserDto, ['password'] as const)
) {}
// Result: { email?: string; name?: string; role?: ... } — no password on update

6. Discriminated Unions — Making Impossible States Unrepresentable

Analogy — Shipping Labels: A package can be "in transit," "delivered," or "failed." Each state has different fields: in-transit has an ETA, delivered has a timestamp, failed has a reason. You'd never have both a deliveryTimestamp AND a failureReason on the same package — that's a contradiction. Discriminated unions enforce these rules at compile time. 
// The discriminant is the "success" field — it determines the shape
type Result<T, E = string> =
  | { success: true; data: T }
  | { success: false; error: E };

// A function that returns Result instead of throwing
async function findUserById(id: string): Promise<Result<PublicUser>> {
  try {
    const user = await userRepository.findById(id);
    if (!user) {
      return { success: false, error: `User ${id} not found` };
    }
    return { success: true, data: user };
  } catch (err) {
    return { success: false, error: 'Database error' };
  }
}

// Consuming the Result — TypeScript narrows automatically
async function handleGetUser(id: string): Promise<void> {
  const result = await findUserById(id);

  if (result.success) {
    // TypeScript knows: result.data is PublicUser
    console.log(`Found user: ${result.data.name}`);
  } else {
    // TypeScript knows: result.error is string
    console.error(`Error: ${result.error}`);
    // result.data would be a TYPE ERROR here — impossible state blocked!
  }
}
Tip: The Result pattern eliminates the try/catch pyramid of doom from your controllers. Every async operation returns a Result; controllers check success and respond accordingly. No uncaught exceptions, no 500s from forgotten try/catch blocks.

7. interface vs type — The Question Everyone Asks

Analogy — Corporate Contracts vs Algebraic Equations: An interface is like a signed contract between parties — open, formal, can be amended (extended) later by anyone. A type alias is like a mathematical equation — precise, can express complex relationships, but once written it's exactly what it says.

They're mostly the same. The key differences are:

  • Interfaces are open (declaration merging): You can add properties to an existing interface by declaring it again. Types cannot be reopened.
  • Types can represent anything: Union types, intersection types, tuples, primitives — these require type. Interfaces can only describe object shapes.
  • Error messages: Interfaces sometimes give cleaner errors in complex scenarios.
// interface — open, can be merged (essential for augmenting third-party types)
interface UserRequest {
  userId: string;
}
// Later, in another file or declaration file:
interface UserRequest {
  tenantId: string; // Merges with above — now has both userId and tenantId
}

// Augmenting Express Request (requires interface — type cannot merge)
declare global {
  namespace Express {
    interface Request {
      user?: AuthenticatedUser;
    }
  }
}

// type — required for unions, tuples, and complex type algebra
type StringOrNumber = string | number;        // Can't do with interface
type Pair<T> = [T, T];                        // Tuple — can't do with interface
type Status = 'pending' | 'active' | 'inactive'; // Literal union

// Rule of thumb:
// - interface for object shapes (signals "this is a contract")
// - type for unions, intersections, tuples, or complex compositions

8. NestJS with TypeScript: Decorators and DI

Analogy — Name Badges at a Conference: A decorator is like putting a name badge on someone. The badge doesn't change who they are — it adds metadata that the system (in this case, NestJS's IoC container) reads to decide what role they play, who needs them, and how to wire them together. 
// Full NestJS controller + DTO with nested validation + custom validator

import {
  Controller, Post, Body, Param, Get, Query, HttpCode, HttpStatus
} from '@nestjs/common';
import {
  IsString, IsEmail, IsUUID, IsOptional, IsInt, Min, Max,
  ValidatorConstraint, ValidatorConstraintInterface,
  ValidationArguments, Validate, ValidateNested, Type as TransformType
} from 'class-validator';
import { Type } from 'class-transformer';
import { Injectable } from '@nestjs/common';

// Custom validator
@ValidatorConstraint({ name: 'isStrongPassword', async: false })
class IsStrongPassword implements ValidatorConstraintInterface {
  validate(password: string, _args: ValidationArguments): boolean {
    return (
      password.length >= 8 &&
      /[A-Z]/.test(password) &&
      /[0-9]/.test(password) &&
      /[^A-Za-z0-9]/.test(password)
    );
  }
  defaultMessage(): string {
    return 'Password must be 8+ chars with uppercase, number, and special character';
  }
}

// Nested DTO
class AddressDto {
  @IsString()
  street!: string;

  @IsString()
  city!: string;

  @IsString()
  country!: string;
}

// Main create DTO
class CreateUserDto {
  @IsEmail()
  email!: string;

  @IsString()
  @Validate(IsStrongPassword)
  password!: string;

  @IsString()
  name!: string;

  @ValidateNested()
  @TransformType(() => AddressDto)
  address!: AddressDto;
}

// Pagination DTO
class PaginationDto {
  @IsOptional()
  @IsInt()
  @Min(1)
  @Type(() => Number)
  page?: number = 1;

  @IsOptional()
  @IsInt()
  @Min(1)
  @Max(100)
  @Type(() => Number)
  limit?: number = 20;
}

// Service
@Injectable()
class UserService {
  async create(dto: CreateUserDto): Promise<Result<PublicUser>> {
    // Implementation
    return { success: true, data: { id: 'u1', ...dto, role: 'user' } as PublicUser };
  }
  async findAll(page: number, limit: number): Promise<PublicUser[]> {
    return []; // Implementation
  }
  async findOne(id: string): Promise<Result<PublicUser>> {
    return { success: false, error: 'Not found' };
  }
}

// Controller
@Controller('users')
class UserController {
  constructor(private readonly userService: UserService) {}

  @Post()
  @HttpCode(HttpStatus.CREATED)
  async create(@Body() dto: CreateUserDto) {
    const result = await this.userService.create(dto);
    if (!result.success) throw new Error(result.error);
    return result.data;
  }

  @Get()
  async findAll(@Query() pagination: PaginationDto) {
    return this.userService.findAll(pagination.page!, pagination.limit!);
  }

  @Get(':id')
  async findOne(@Param('id') id: string) {
    const result = await this.userService.findOne(id);
    if (!result.success) throw new Error(result.error);
    return result.data;
  }
}

9. Monorepo Type Versioning

Analogy — Shared Parts Catalog: Imagine a car factory where the design team updates the spec for a door hinge, but the assembly line keeps using the old spec. Parts that were supposed to fit together don't. A shared types package is the single source of truth — everyone reads from the same spec.

When frontend and backend share a monorepo, the risk is silent type drift: the backend updates the User type, the frontend still compiles (it has the old cached type), but at runtime the shape is wrong and things break in subtle ways.

Solution 1 — Shared types package (for tRPC or internal APIs):

// packages/shared-types/src/user.ts
export interface User {
  id: string;
  email: string;
  name: string;
  role: 'admin' | 'user';
}

// apps/backend/src/users/users.service.ts
import type { User } from '@myapp/shared-types';

// apps/frontend/src/api/users.ts
import type { User } from '@myapp/shared-types';
// When the backend changes User, TypeScript errors appear in the FRONTEND too

Solution 2 — Generated types from OpenAPI schema:

// In package.json
// "generate:types": "openapi-typescript openapi.yaml -o src/api-types.ts"
// Run after any backend API change — types are always in sync with the contract

Senior vs Junior Answer

Interview question: "What's the difference between interface and type in TypeScript?"
Junior answer

"Interface is for objects and type is for primitives. You use interface when you want to describe an object shape."

Senior answer

"They're largely interchangeable for object shapes, but the key differences matter in practice. Interfaces support declaration merging — you can re-declare an interface in multiple places and TypeScript merges the declarations. This is essential when augmenting third-party types, like adding a user property to Express's Request interface. Types are closed — you can't reopen them.

On the other side, type is more powerful for type algebra: it can represent union types (string | number), intersection types, tuples, and mapped types — things interface simply cannot express.

My rule of thumb: use interface for object contracts (it signals 'this is a shape something must conform to') and type for unions, intersections, and complex derived types. TypeScript's structural type system means both work for simple object shapes, so don't overthink it — pick a consistent convention and stick to it."

Practice Questions

Beginner

Question 1: Fix the any — Use unknown with type narrowing

The following code uses any and will crash at runtime. Identify the problem and rewrite it using unknown with proper type narrowing.

// BROKEN CODE
function parseApiResponse(response: any): string {
  return response.data.name.toUpperCase();
}

// This crashes when called with:
parseApiResponse({ error: 'Not found' });  // runtime crash
parseApiResponse(null);                     // runtime crash
Show solution 
interface SuccessResponse {
  data: { name: string };
}

interface ErrorResponse {
  error: string;
}

type ApiResponse = SuccessResponse | ErrorResponse;

function isSuccessResponse(r: unknown): r is SuccessResponse {
  return (
    typeof r === 'object' &&
    r !== null &&
    'data' in r &&
    typeof (r as SuccessResponse).data?.name === 'string'
  );
}

function parseApiResponse(response: unknown): string {
  if (isSuccessResponse(response)) {
    // TypeScript knows: response.data.name is a string
    return response.data.name.toUpperCase();
  }
  if (
    typeof response === 'object' &&
    response !== null &&
    'error' in response &&
    typeof (response as ErrorResponse).error === 'string'
  ) {
    throw new Error((response as ErrorResponse).error);
  }
  throw new Error('Unexpected response shape');
}

// Now safe:
parseApiResponse({ data: { name: 'alice' } }); // "ALICE"
// parseApiResponse({ error: 'Not found' });    // throws Error: Not found

Question 2: Generic Result<T, E> type with fetchUser

Write a generic Result<T, E = string> discriminated union type. Then implement a fetchUser(id: string) function that returns Promise<Result<User>>. Demonstrate proper narrowing at the call site.

Show solution 
// The generic Result type
type Result<T, E = string> =
  | { success: true; data: T }
  | { success: false; error: E };

// Helper constructors for readability
const ok = <T>(data: T): Result<T, never> => ({ success: true, data });
const err = <E = string>(error: E): Result<never, E> => ({ success: false, error });

interface User {
  id: string;
  name: string;
  email: string;
}

// Simulated DB
const users: Record<string, User> = {
  'u1': { id: 'u1', name: 'Alice', email: 'alice@example.com' },
};

async function fetchUser(id: string): Promise<Result<User>> {
  if (!id || id.trim() === '') {
    return err('User ID cannot be empty');
  }
  const user = users[id];
  if (!user) {
    return err(`User with id "${id}" not found`);
  }
  return ok(user);
}

// Consuming with narrowing
async function displayUser(id: string): Promise<void> {
  const result = await fetchUser(id);

  if (result.success) {
    // TypeScript: result.data is User
    console.log(`Name: ${result.data.name}, Email: ${result.data.email}`);
  } else {
    // TypeScript: result.error is string
    console.error(`Failed: ${result.error}`);
    // result.data; // TYPE ERROR — impossible state blocked
  }
}

displayUser('u1');    // Name: Alice, Email: alice@example.com
displayUser('u999');  // Failed: User with id "u999" not found

Question 3: Full NestJS service with generic repository, utility types, and error handling

Write a complete, runnable TypeScript module containing: a generic Repository<T, ID> interface, a User entity, CreateUserDto, UpdateUserDto (using Partial + Omit), an in-memory UserRepository implementation, a UserService using the Result pattern, and demonstrate usage. No frameworks required — pure TypeScript that can run in Node.

Show solution 
// ============================================================
// types.ts — shared types
// ============================================================
type Result<T, E = string> =
  | { success: true; data: T }
  | { success: false; error: E };

const ok = <T>(data: T): Result<T> => ({ success: true, data });
const fail = (error: string): Result<never> => ({ success: false, error });

// ============================================================
// user.entity.ts
// ============================================================
interface User {
  id: string;
  email: string;
  name: string;
  role: 'admin' | 'user';
  passwordHash: string;
  createdAt: Date;
  updatedAt: Date;
}

// PublicUser — never expose passwordHash
type PublicUser = Omit<User, 'passwordHash'>;

// CreateUserDto — all required except id, timestamps (those are generated)
type CreateUserDto = Pick<User, 'email' | 'name' | 'role'> & { password: string };

// UpdateUserDto — all fields optional, cannot update email or password here
type UpdateUserDto = Partial<Pick<User, 'name' | 'role'>>;

// ============================================================
// repository.interface.ts
// ============================================================
interface Repository<T, ID = string> {
  findById(id: ID): Promise<T | null>;
  findAll(): Promise<T[]>;
  findByEmail?(email: string): Promise<T | null>;
  save(entity: Omit<T, 'id' | 'createdAt' | 'updatedAt'> & Partial<Pick<T, 'id'>>): Promise<T>;
  update(id: ID, partial: Partial<Omit<T, 'id' | 'createdAt'>>): Promise<T | null>;
  delete(id: ID): Promise<boolean>;
}

// ============================================================
// user.repository.ts — in-memory implementation
// ============================================================
class UserRepository implements Repository<User> {
  private store: Map<string, User> = new Map();
  private nextId = 1;

  async findById(id: string): Promise<User | null> {
    return this.store.get(id) ?? null;
  }

  async findAll(): Promise<User[]> {
    return Array.from(this.store.values());
  }

  async findByEmail(email: string): Promise<User | null> {
    for (const user of this.store.values()) {
      if (user.email === email) return user;
    }
    return null;
  }

  async save(
    data: Omit<User, 'id' | 'createdAt' | 'updatedAt'>
  ): Promise<User> {
    const user: User = {
      ...data,
      id: `u${this.nextId++}`,
      createdAt: new Date(),
      updatedAt: new Date(),
    };
    this.store.set(user.id, user);
    return user;
  }

  async update(id: string, partial: Partial<Omit<User, 'id' | 'createdAt'>>): Promise<User | null> {
    const existing = this.store.get(id);
    if (!existing) return null;
    const updated: User = { ...existing, ...partial, updatedAt: new Date() };
    this.store.set(id, updated);
    return updated;
  }

  async delete(id: string): Promise<boolean> {
    return this.store.delete(id);
  }
}

// ============================================================
// user.service.ts
// ============================================================
class UserService {
  constructor(private readonly repo: UserRepository) {}

  private toPublic(user: User): PublicUser {
    const { passwordHash, ...publicUser } = user;
    return publicUser;
  }

  // Simple hash simulation — in production use bcrypt
  private hashPassword(password: string): string {
    return `hashed_${password}_${Date.now()}`;
  }

  async createUser(dto: CreateUserDto): Promise<Result<PublicUser>> {
    // Check for duplicate email
    const existing = await this.repo.findByEmail(dto.email);
    if (existing) {
      return fail(`Email ${dto.email} is already registered`);
    }

    // Validate password length
    if (dto.password.length < 8) {
      return fail('Password must be at least 8 characters');
    }

    const user = await this.repo.save({
      email: dto.email,
      name: dto.name,
      role: dto.role,
      passwordHash: this.hashPassword(dto.password),
    });

    return ok(this.toPublic(user));
  }

  async getUser(id: string): Promise<Result<PublicUser>> {
    const user = await this.repo.findById(id);
    if (!user) return fail(`User ${id} not found`);
    return ok(this.toPublic(user));
  }

  async updateUser(id: string, dto: UpdateUserDto): Promise<Result<PublicUser>> {
    if (Object.keys(dto).length === 0) {
      return fail('No fields provided for update');
    }
    const updated = await this.repo.update(id, dto);
    if (!updated) return fail(`User ${id} not found`);
    return ok(this.toPublic(updated));
  }

  async deleteUser(id: string): Promise<Result<{ deleted: true }>> {
    const deleted = await this.repo.delete(id);
    if (!deleted) return fail(`User ${id} not found`);
    return ok({ deleted: true });
  }

  async listUsers(): Promise<PublicUser[]> {
    const users = await this.repo.findAll();
    return users.map(this.toPublic.bind(this));
  }
}

async function main(): Promise<void> {
  const repo = new UserRepository();
  const service = new UserService(repo);

  // Create users
  const r1 = await service.createUser({
    email: 'alice@example.com', name: 'Alice', role: 'admin', password: 'SecurePass1!'
  });
  const r2 = await service.createUser({
    email: 'bob@example.com', name: 'Bob', role: 'user', password: 'SecurePass2!'
  });

  console.log('Created:', r1.success ? r1.data.name : r1.error); // Alice
  console.log('Created:', r2.success ? r2.data.name : r2.error); // Bob

  // Duplicate email
  const r3 = await service.createUser({
    email: 'alice@example.com', name: 'Alice2', role: 'user', password: 'AnotherPass1!'
  });
  console.log('Duplicate:', r3.success ? 'ok' : r3.error);
  // "Email alice@example.com is already registered"

  // Get user
  if (r1.success) {
    const user = await service.getUser(r1.data.id);
    if (user.success) {
      console.log('Fetched:', user.data.name, '— has passwordHash?', 'passwordHash' in user.data); // false
    }
  }

  // Update user
  if (r1.success) {
    const updated = await service.updateUser(r1.data.id, { name: 'Alice Senior' });
    console.log('Updated name:', updated.success ? updated.data.name : updated.error);
  }

  // List all
  const all = await service.listUsers();
  console.log('All users:', all.map(u => u.name)); // ['Alice Senior', 'Bob']
}

Chapter Summary

  • Structural typing means TypeScript checks shape, not name — enabling powerful composition and duck-typed interfaces without explicit declarations.
  • Type narrowing (typeof, instanceof, in, type guards, discriminated unions) is how TypeScript progresses from broad to specific types safely — use the exhaustiveness check (never) in switch statements to catch unhandled cases at compile time.
  • unknown over any — always. any disables the type checker; unknown forces you to prove what you have before using it. Every any in production code is a crash waiting to happen.
  • Utility types (Partial, Omit, Pick, Record, ReturnType) let you derive DTOs and response shapes from a single source-of-truth type — keeping your NestJS DTO hierarchy DRY and consistent.
  • Discriminated unions with the Result pattern make impossible states unrepresentable — replacing try/catch spaghetti with explicit, type-safe success/failure handling that compilers can verify.
