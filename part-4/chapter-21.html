CI/CD, Git, and DevOps

Shipping software is as important as writing it. The team that ships ten times a day catches bugs earlier, delivers value faster, and has a smaller blast radius when something goes wrong.

What you'll learn

  • GitFlow vs Trunk-Based Development vs GitHub Flow — when each strategy wins
  • Designing a CI/CD pipeline with a full GitHub Actions example
  • Docker fundamentals: multi-stage builds, layer caching, and image optimization
  • Kubernetes essentials: Pods, Deployments, rolling updates, and health probes
  • Blue/Green vs Canary deployments, TDD, and Feature Flags

"Shipping software is as important as writing it. The team that ships 10 times a day catches bugs earlier, delivers value faster, and has a smaller blast radius when something goes wrong."

This chapter covers the tooling and practices that separate a developer who writes code from a senior engineer who ships production software reliably. Interviewers who hire for backend roles almost always ask about CI/CD, deployment strategies, and Docker/Kubernetes basics. Know them cold.

1. The Big Idea: Custom Workshop vs Assembly Line

Analogy: A custom workshop quality-checks everything at the end — the craftsperson builds the entire chair, then inspects it. The factory assembly line has a quality check at every single station. If a defect appears at station 3, it's caught immediately. It never reaches station 10. CI/CD is the assembly line for software. Every commit triggers a sequence of automated checks. Defects are caught at the earliest possible moment, when they're cheapest to fix.

2. Git Branching Strategies

Analogy: Think of git branches as parallel universe timelines. The main timeline (main/master) is the stable universe everyone can see. A feature branch is a private universe where you experiment. When your experiment succeeds, you merge your timeline back into the stable one. The question is: how long do parallel universes live before merging, and how many parallel universes are you managing at once?

GitFlow

Two long-lived branches: main (always deployable) and develop (integration branch). Short-lived branches: feature/*, hotfix/*, release/*.

Good for: versioned software with explicit release cycles (mobile apps, packaged software, libraries). Clear semantics for hotfixes and release preparation.

Downsides: complex ceremony. Long-lived feature branches create merge hell. Develop and main diverging creates confusion. Teams new to git often get lost in the flow.

Trunk-Based Development

Everyone merges to main at least daily. No long-lived branches. Incomplete features are hidden behind feature flags, not isolated branches.

Good for: high-frequency deployment. This is how Google, Facebook, and Netflix ship. Requires mature CI that catches breakages in minutes, and discipline to use feature flags properly.

Downsides: requires strong CI and a team culture of small, safe commits. Feature flags add complexity. Not suitable for versioned releases.

GitHub Flow

A simplified middle ground: main is always deployable. Developers create short-lived feature branches, open pull requests, get code review, and merge. Deploy on merge to main.

Good for: most web application teams. Simple enough for beginners, fast enough for most teams, and enforces code review via PRs.

Decision rule: Use trunk-based development if your team deploys continuously and has mature CI. Use GitFlow if you have versioned releases (e.g., mobile apps, open-source libraries). Use GitHub Flow for most web services — it's the practical default.

3. CI/CD Pipeline Design

A CI/CD pipeline is a sequence of automated steps that run on every commit. The golden rule: if any step fails, the pipeline stops. Never deploy broken code.

Pipeline stages (in order):
Lint → Unit Tests → Build → Integration Tests → Security Scan → Deploy Staging → Smoke Tests → Deploy Production

Each step acts as a gate. Linting catches style and obvious errors cheaply. Unit tests run fast and catch logic bugs. The build step verifies the artifact compiles. Integration tests verify service boundaries. Security scans catch known vulnerabilities in dependencies. Staging deploy verifies the artifact actually runs. Smoke tests verify the deployment is healthy. Only then does production get touched.

Full GitHub Actions CI/CD pipeline for a Node.js/NestJS service:

# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  test:
    name: Lint, Test, Build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci   # ci is faster than install, respects package-lock.json exactly

      - name: Lint
        run: npm run lint

      - name: Unit tests
        run: npm run test -- --coverage
        env:
          NODE_ENV: test

      - name: Build
        run: npm run build

      - name: Security audit
        run: npm audit --audit-level=high   # Fail if high or critical vulns found

  docker:
    name: Build & Push Docker Image
    runs-on: ubuntu-latest
    needs: test   # Only runs if test job passes
    if: github.ref == 'refs/heads/main'   # Only on merge to main
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Log in to container registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          cache-from: type=gha     # Use GitHub Actions cache for Docker layer cache
          cache-to: type=gha,mode=max

  deploy:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: docker
    environment: production   # Requires manual approval gate in GitHub
    steps:
      - name: Deploy to Kubernetes
        run: |
          kubectl set image deployment/api-service \
            api=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} \
            --namespace=production

4. Docker — What It Actually Solves

Analogy: Before standardized shipping containers, loading cargo onto ships required custom handling for every item. Different shapes, sizes, packing methods — a logistical nightmare. The standardized shipping container changed everything. Any ship can carry any container. Any port can load or unload any container. Docker does the same for software: the container is a standardized unit that runs identically on a developer's laptop, in CI, on staging, and in production.
Image vs Container: An image is the blueprint — a static, immutable snapshot of the filesystem and configuration. A container is a running instance of that image. One image can spawn many containers, just as one house blueprint can build many houses.

Multi-stage Dockerfile for NestJS — this is the pattern you should know cold:

# Dockerfile

# ── Stage 1: Build ────────────────────────────────────────────────────────────
FROM node:20-alpine AS builder
WORKDIR /app

# CRITICAL: Copy package files BEFORE source code.
# Docker caches each layer. If you copy package.json first and it hasn't changed,
# Docker reuses the npm install cache even if your source code changed.
# This turns a 90-second npm install into a 2-second cache hit.
COPY package.json package-lock.json ./
RUN npm ci

# Now copy source — cache-busting only happens when source actually changes
COPY . .
RUN npm run build   # Outputs compiled JS to /app/dist


# ── Stage 2: Production ───────────────────────────────────────────────────────
FROM node:20-alpine AS production
WORKDIR /app
ENV NODE_ENV=production

# Only copy production dependencies — devDependencies stay in builder stage
COPY package.json package-lock.json ./
RUN npm ci --omit=dev

# Copy the compiled output from the builder stage — NOT the source code
COPY --from=builder /app/dist ./dist

# Run as non-root user for security
USER node

EXPOSE 3000
CMD ["node", "dist/main.js"]
Result: The production image contains only the compiled JS and production node_modules. The TypeScript compiler, source files, devDependencies, and build cache are all left in the builder stage and discarded. Image size typically drops from ~600MB to ~150MB.

5. Kubernetes Basics

Analogy: Kubernetes is a fleet manager for containers. You don't say "start container X on server 3." You say "I need 5 copies of this service running at all times." Kubernetes decides which servers they go on, restarts them if they crash, and gradually replaces them when you deploy a new version — without you needing to manage individual machines.

The key resources to know for interviews:

  • Pod: The smallest deployable unit. Usually one container. Ephemeral — pods die and are replaced, not repaired.
  • Deployment: Manages a set of identical pods. Declares desired state: "run 3 replicas of this image." Kubernetes reconciles reality to match.
  • Service: Stable network endpoint (IP + DNS name) that routes to pods. Pods come and go; the Service address stays constant.
  • Ingress: HTTP routing into the cluster. Routes /api/* to the API service, /static/* to the CDN, etc.
  • ConfigMap / Secret: Inject configuration and credentials as environment variables without baking them into the image.
  • HPA (Horizontal Pod Autoscaler): Automatically scales replica count based on CPU/memory/custom metrics.

Rolling deployment (the default): Kubernetes brings up new pods one at a time and takes down old pods only after new ones are healthy. Zero downtime. Controlled by maxSurge and maxUnavailable settings.

Readiness probe vs Liveness probe:

  • Readiness: "Is this pod ready to receive traffic?" Until ready, Kubernetes removes it from the Service's load-balancer pool. Use this to prevent requests from hitting a pod that's still warming up.
  • Liveness: "Is this pod alive?" If the liveness probe fails, Kubernetes kills and replaces the pod. Use this to detect deadlocks or infinite loops where the process is running but stuck.
# deployment.yaml (simplified)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-service
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # Allow 1 extra pod during rollout (4 total briefly)
      maxUnavailable: 0  # Never reduce below desired count (no downtime)
  template:
    spec:
      containers:
        - name: api
          image: ghcr.io/myorg/api-service:abc123
          ports:
            - containerPort: 3000
          readinessProbe:
            httpGet:
              path: /health/ready
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /health/live
              port: 3000
            initialDelaySeconds: 15
            periodSeconds: 20

6. Blue/Green vs Canary Deployments

Blue/Green: You have two identical production environments — Blue (live) and Green (idle). Deploy new version to Green. Test it thoroughly. Flip the load balancer to send 100% of traffic to Green in one instant. Blue immediately becomes your rollback — one switch and you're back. No gradual ramp, no partial exposure.
Canary: Like a canary in a coal mine. Send 5% of real traffic to the new version first. Monitor error rates, latency, and business metrics. If metrics look healthy, ramp to 20%, then 50%, then 100%. If something looks wrong, route 100% back to the old version immediately with minimal users exposed.

7. Test-Driven Development (TDD) in Interviews

Analogy: Writing a test before the code is like drawing the target before throwing the dart. You define exactly what success looks like before you try to achieve it. This forces you to think about the interface before implementation.

The red-green-refactor cycle:

  1. Red: Write a failing test that describes the behavior you want.
  2. Green: Write the minimum code to make the test pass. Don't over-engineer.
  3. Refactor: Clean up the code while keeping tests green.
// tdd-example.test.ts — TDD'ing a simple discount calculator

// RED: Write the test first. This fails because discountedPrice doesn't exist yet.
import { discountedPrice } from './pricing';

describe('discountedPrice', () => {
  it('applies percentage discount correctly', () => {
    expect(discountedPrice(100, 20)).toBe(80);
  });

  it('returns original price when discount is 0', () => {
    expect(discountedPrice(100, 0)).toBe(100);
  });

  it('throws when discount exceeds 100%', () => {
    expect(() => discountedPrice(100, 110)).toThrow('Discount cannot exceed 100%');
  });
});

// GREEN: Write minimum code to pass all tests.
// pricing.ts
export function discountedPrice(price: number, discountPercent: number): number {
  if (discountPercent > 100) {
    throw new Error('Discount cannot exceed 100%');
  }
  return price * (1 - discountPercent / 100);
}

// REFACTOR: Add input validation for negative values, extract the multiplier, etc.
// Tests stay green throughout.

8. Feature Flags

Analogy: A light switch is binary — on or off. A dimmer switch lets you gradually increase brightness. Feature flags are dimmer switches for code. Instead of deploying a feature fully on or off, you can roll it out to 1% of users, then 10%, then everyone — all without redeploying.

Use cases:

  • Gradual rollout: 5% of users see the new checkout flow. Monitor conversion rate. Ramp if it's better.
  • A/B testing: 50% see version A, 50% see version B. Measure which converts better.
  • Kill switch: Flag disables a feature instantly in production without a code deploy.
  • Trunk-based development: Incomplete features merged to main but hidden behind a flag.
// Simple feature flag via environment variable (suitable for early-stage)
const NEW_CHECKOUT_ENABLED = process.env.FF_NEW_CHECKOUT === 'true';

if (NEW_CHECKOUT_ENABLED) {
  return newCheckoutFlow(cart);
} else {
  return legacyCheckoutFlow(cart);
}

// Mature feature flag with per-user targeting (using Unleash or LaunchDarkly SDK)
import { isEnabled } from 'unleash-client';

async function checkout(userId: string, cart: Cart) {
  if (isEnabled('new-checkout', { userId })) {
    return newCheckoutFlow(cart);
  }
  return legacyCheckoutFlow(cart);
}
// Unleash/LaunchDarkly evaluates the flag per user — 10% of user IDs get true,
// consistently. The same user always sees the same experience.

Senior vs Junior

Question: "How do you deploy without downtime?"

Junior Answer "I'd use Kubernetes rolling deployments."
Senior Answer "Zero-downtime deployment is a system-level concern, not just a deployment strategy. At the infrastructure level, I'd use Kubernetes rolling deployments with maxUnavailable: 0 so the old pods stay up until new pods pass their readiness probe. I'd configure readiness probes to check that the app has finished warming up — connected to the database, loaded caches — before taking traffic. At the application level, I'd make sure database migrations are backward-compatible: never drop or rename a column in the same deploy that removes the code using it. Deploy the code change first, verify it's healthy, then clean up the old schema in a second migration. I'd also use feature flags for high-risk changes so I can ship code to production without exposing it to users until I'm confident. For highest-risk releases, I'd use a canary deployment — 5% of traffic to the new version, monitor error rate and p99 latency for 10 minutes, then ramp up. If anything looks wrong, route 100% back immediately."

Practice Questions

Beginner

What is the difference between Continuous Integration (CI) and Continuous Deployment (CD)?

CI is the practice of merging code changes frequently and automatically running tests to catch integration issues early. CD extends this by automatically deploying passing builds to production (or staging). CI catches bugs early; CD ensures that every passing build can reach users quickly. Some teams practice Continuous Delivery (code is always in a deployable state, but requires manual approval to deploy) vs Continuous Deployment (automatically deploys without manual gates).

Intermediate

Explain Docker layer caching. Why does the order of instructions in a Dockerfile matter?

Docker builds images layer by layer. Each instruction (COPY, RUN, etc.) creates a layer. If Docker detects a layer hasn't changed, it reuses the cached version and skips re-executing it — and all layers built on top of it. This means instruction order is performance-critical. If you COPY . . before RUN npm install, any source file change invalidates the COPY layer, forcing a fresh npm install even when package.json hasn't changed. The correct pattern: COPY package.json package-lock.json ./ then RUN npm ci, then COPY . .. The install layer is cached as long as dependencies don't change — typically the 90% case.

Advanced

Design a complete CI/CD pipeline for a NestJS microservice. What stages would you include, why, and how would you handle database migrations?

Pipeline stages:

  1. Lint: ESLint + Prettier check. Cheap, fast. Catches obvious issues before we waste CI time.
  2. Unit tests: Jest with coverage threshold (e.g., 80%). No external dependencies — mock everything. Run in parallel. Must complete in under 2 minutes.
  3. Build: npm run build. Verifies TypeScript compiles. Fails fast on type errors.
  4. Integration tests: Spin up a test PostgreSQL via docker-compose service in the CI environment. Run tests against real DB. Tests are slower but catch SQL bugs and query performance issues.
  5. Security scan: npm audit --audit-level=high. Optionally Snyk or Trivy to scan the Docker image for OS-level vulnerabilities.
  6. Docker build + push: Multi-stage build. Tag with commit SHA. Push to container registry. Only on merge to main.
  7. Deploy to staging: kubectl set image or Helm upgrade. Run smoke tests against staging (health endpoint, one happy-path API call).
  8. Database migration: Run npm run migration:run as a Kubernetes Job before the rolling deployment. The migration must be backward-compatible — the old app version must still work with the new schema while the rolling update is in progress. Never drop columns in the same migration as the code change that stops using them.
  9. Production deploy: Gated by a manual approval step in GitHub Environments. Rolling deployment with maxUnavailable: 0. Readiness probe ensures new pods are healthy before old ones are terminated.
  10. Post-deploy monitoring: Alert on error rate spike or p99 latency increase in the 5 minutes after deploy. Automated rollback or manual flag to roll back if thresholds are breached.

Chapter Summary

  • Choose your branching strategy based on deployment frequency: trunk-based for continuous deployment, GitFlow for versioned releases, GitHub Flow for most web services.
  • A CI/CD pipeline is a sequence of automated gates: if any step fails, the pipeline stops. Never deploy broken code.
  • Multi-stage Docker builds produce small, secure production images. Layer caching order — package files before source code — is a mandatory optimization.
  • Kubernetes rolling deployments achieve zero downtime. Readiness probes prevent traffic from reaching unhealthy pods. Liveness probes restart stuck processes.
  • Canary deployments reduce blast radius for high-risk changes. Feature flags decouple deployment from release, enabling gradual rollouts without redeployment.
