Part 4 · Chapter 20

Performance, Reliability, and Observability

You can't fix what you can't see. The difference between a system that's being maintained and a system that's being operated is observability. Operators know what's happening before users report it.

What you'll learn

  • What P50/P95/P99 latency actually means and why P99 matters more than average
  • The vocabulary of reliability: SLI, SLO, SLA, and error budgets
  • The three pillars of observability — logs, metrics, and traces — with real NestJS code
  • How to handle graceful shutdown in a Kubernetes environment
  • Server-side rate limiting with a Redis sliding window implementation
  • The structured approach senior engineers use to debug a production outage

Analogy: Running a Hospital

A hospital doesn't wait for patients to arrive to know how busy it is. The moment you walk into a major hospital, there are dashboards tracking beds occupied, ER wait times, ICU capacity, ventilators in use, and staff-to-patient ratios. When any metric crosses a threshold, alarms fire and staff take action — before the situation becomes a crisis.

A well-operated software system works exactly the same way. You define what healthy looks like (your SLOs). You measure constantly (your SLIs). You alert when thresholds are breached. And you have runbooks for exactly what to do when the alarm sounds — so you're not guessing at 2 AM when users are down.

Observability is your system's vital-signs monitor. Without it, you're not operating a system. You're just hoping.

1. P50/P95/P99 Latency — The Vocabulary of Performance

Analogy: The Marathon

Picture a marathon with 1,000 runners. The P50 (median) finisher is the person in 500th place — exactly half the field finished faster, half finished slower. The P95 finisher came in 950th — they finished faster than 950 runners but slower than 50. The P99 finisher came in 990th — faster than 990 people, slower than only 10.

Now map this to API requests. P50 latency is your typical experience. P99 is the slowest "regular" request — the one your most frustrated user just experienced. They called support. They tweeted. P99 is your problem child.

The critical insight is that average latency lies. Suppose you have 99 requests that complete in 10 ms and one request that takes 10,000 ms. The average is (99 × 10 + 10,000) / 100 = 109.9 ms — totally acceptable looking. But your P99 is 10,000 ms. One in every hundred users is waiting 10 seconds. That user is your loudest critic on Twitter.

Tail latency — reducing P99 and P99.9 — is one of the hardest engineering problems in distributed systems. Common culprits:

  • Lock contention: one slow database query blocks all concurrent requests sharing that connection
  • Garbage collection pauses: JVM GC, Node.js GC, or Go GC stops the world briefly but often
  • Network jitter: occasional dropped packets force TCP retransmits
  • Cold-start effects: caches miss after a pod restarts, causing a burst of slow database reads
  • Fan-out amplification: a single API call triggers 20 microservice calls; the slowest one determines your total response time

Key Term: Percentile Latency

Pn latency means: n% of requests completed faster than this duration. P99 = 200 ms means 99% of requests completed in under 200 ms, and 1% took longer. Always ask "at what percentile?" when someone mentions latency numbers. "Our average response time is 50 ms" is a much weaker claim than "our P99 is 120 ms."

Here is how you record latency in a NestJS application using a Prometheus histogram interceptor. The histogram automatically computes P50, P95, and P99 from the recorded observations:

// latency.interceptor.ts
import {
  Injectable,
  NestInterceptor,
  ExecutionContext,
  CallHandler,
} from '@nestjs/common';
import { Observable } from 'rxjs';
import { tap } from 'rxjs/operators';
import { Registry, Histogram } from 'prom-client';

@Injectable()
export class LatencyInterceptor implements NestInterceptor {
  private readonly histogram: Histogram<string>;

  constructor(registry: Registry) {
    this.histogram = new Histogram({
      name: 'http_request_duration_seconds',
      help: 'HTTP request latency in seconds',
      labelNames: ['method', 'route', 'status_code'],
      // Buckets cover P50 through P99.9 for typical web APIs
      buckets: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10],
      registers: [registry],
    });
  }

  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    const req = context.switchToHttp().getRequest();
    const end = this.histogram.startTimer({
      method: req.method,
      route: req.route?.path ?? req.url,
    });

    return next.handle().pipe(
      tap({
        next: () => {
          const res = context.switchToHttp().getResponse();
          end({ status_code: res.statusCode });
        },
        error: () => {
          end({ status_code: 500 });
        },
      }),
    );
  }
}

// In Prometheus/Grafana, query percentiles like this:
// P99: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))
// P95: histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))
// P50: histogram_quantile(0.50, rate(http_request_duration_seconds_bucket[5m]))

Interview tip: always specify the percentile

When an interviewer asks "what's the latency of your system?" never answer with a bare number. Say: "Our P50 is around 20 ms, P95 is 80 ms, and our P99 is 350 ms. We have an SLO that keeps P99 under 500 ms." This immediately signals you understand how production systems are actually measured.

2. SLI, SLO, and SLA — The Vocabulary of Reliability

Analogy: The Pizza Delivery Promise

A pizza company measures whether deliveries arrive within 30 minutes. The SLI (Service Level Indicator) is the raw measurement: "Today, 94% of deliveries arrived within 30 minutes." The SLO (Service Level Objective) is the internal target the team commits to: "We aim for 95% of deliveries within 30 minutes." The SLA (Service Level Agreement) is the legal contract with the customer: "If fewer than 90% of deliveries arrive on time, you get a free pizza." The SLA is deliberately looser than the SLO — the gap between them is the buffer that prevents every minor SLO breach from triggering a refund.

These three terms appear in nearly every senior backend interview. Know them cold:

  • SLI (Service Level Indicator): A quantitative measurement of service behavior. Examples: request success rate, P99 latency, availability (uptime percentage), error rate. The SLI is what you actually measure.
  • SLO (Service Level Objective): The target value for an SLI that your team commits to internally. Example: "99.9% of requests succeed" or "P99 latency is below 500 ms." The SLO lives in your runbooks and alerting rules.
  • SLA (Service Level Agreement): A contractual commitment to customers, usually expressed as a consequence (credit, refund, contract termination) if the SLO is not met. SLAs are set 10–20% more lenient than your SLO to give you a safety buffer.

The practical corollary of an SLO is the error budget. A 99.9% availability SLO means you're allowed 0.1% downtime. Over a 30-day month, that's 43.8 minutes of allowed downtime. Once you've used it, any further downtime is an SLO violation. Smart teams spend their error budget deliberately — planned maintenance, risky deployments, experiments. When the budget runs low, they freeze risky changes.

Key Term: Error Budget

Error budget = 1 − SLO. It's the amount of unreliability you're allowed before violating your SLO. If you have plenty of error budget, you can move fast and take risks. If you're near zero, you slow down, freeze risky releases, and prioritize reliability work. This is how reliability becomes a first-class engineering constraint rather than an afterthought.

3. The Three Pillars of Observability

Observability is the ability to understand the internal state of a system from its external outputs. The three pillars — logs, metrics, and traces — give you different angles on the same running system. Together they let you answer: what happened, how much, and where.

Pillar 1: Logs — What Happened

Analogy: The Pilot's Flight Recorder

A flight recorder captures every significant event during a flight: altitude changes, engine readings, cockpit voice. After an incident, investigators replay the recording to reconstruct exactly what happened and in what order. Logs are your system's flight recorder — timestamped, structured events that tell the story of what happened to specific requests.

The key word is structured. A plain-text log like "Error processing request" is nearly useless at scale. You can't query it, aggregate it, or alert on it reliably. A structured JSON log is a machine-readable event:

// logger.config.ts — Winston structured logger for NestJS
import { createLogger, format, transports } from 'winston';

export const logger = createLogger({
  format: format.combine(
    format.timestamp(),
    format.errors({ stack: true }),
    format.json(),  // Output as structured JSON, not plain text
  ),
  defaultMeta: {
    service: 'payment-service',
    version: process.env.APP_VERSION,
  },
  transports: [
    new transports.Console(),
    // In production, also ship to a log aggregator (Datadog, Loki, etc.)
  ],
});

// Usage — always include request context
logger.info('Payment processed', {
  requestId: req.headers['x-request-id'],
  userId: req.user.id,
  amount: payment.amount,
  currency: payment.currency,
  durationMs: Date.now() - startTime,
});

logger.error('Payment failed', {
  requestId: req.headers['x-request-id'],
  userId: req.user.id,
  error: err.message,
  stack: err.stack,
  paymentMethod: req.body.method,
});

// The JSON output looks like:
// {
//   "level": "info",
//   "message": "Payment processed",
//   "service": "payment-service",
//   "requestId": "abc-123",
//   "userId": "user-456",
//   "amount": 9900,
//   "timestamp": "2026-05-18T10:32:11.043Z"
// }

Structured logs let you query across millions of events: "Show me all errors for userId=user-456 in the last hour," or "What's the P99 duration for payment processing today?" Log aggregators like Datadog, Grafana Loki, and AWS CloudWatch Logs Insights make this possible at scale.

Pillar 2: Metrics — How Much, Over Time

Analogy: The Car Dashboard

Your car's dashboard doesn't log every engine event. It summarizes current state: speed, RPM, fuel level, temperature. Metrics work the same way — they aggregate numerical measurements over time so you can see trends, spot anomalies, and set thresholds. You don't need every individual reading; you need the shape of what's happening.

There are three fundamental metric types:

  • Counter: monotonically increasing value. Total requests served, total errors, total bytes sent. Always goes up.
  • Gauge: value that goes up and down. Current active connections, current queue depth, memory in use.
  • Histogram: distribution of values across configurable buckets. Used for latency and request size. Lets you compute percentiles.
// metrics.service.ts — custom Prometheus metrics in NestJS
import { Injectable, OnModuleInit } from '@nestjs/common';
import { Registry, Counter, Gauge, Histogram } from 'prom-client';

@Injectable()
export class MetricsService implements OnModuleInit {
  private readonly registry = new Registry();
  private requestCounter: Counter<string>;
  private activeConnections: Gauge<string>;
  private dbQueryDuration: Histogram<string>;

  onModuleInit() {
    this.requestCounter = new Counter({
      name: 'http_requests_total',
      help: 'Total HTTP requests received',
      labelNames: ['method', 'route', 'status'],
      registers: [this.registry],
    });

    this.activeConnections = new Gauge({
      name: 'http_active_connections',
      help: 'Number of currently active HTTP connections',
      registers: [this.registry],
    });

    this.dbQueryDuration = new Histogram({
      name: 'db_query_duration_seconds',
      help: 'Database query execution time',
      labelNames: ['operation', 'table'],
      buckets: [0.001, 0.005, 0.01, 0.05, 0.1, 0.5, 1, 5],
      registers: [this.registry],
    });
  }

  incrementRequest(method: string, route: string, status: number) {
    this.requestCounter.inc({ method, route, status: String(status) });
  }

  trackConnection(delta: 1 | -1) {
    if (delta === 1) this.activeConnections.inc();
    else this.activeConnections.dec();
  }

  measureDbQuery(operation: string, table: string): () => void {
    return this.dbQueryDuration.startTimer({ operation, table });
  }

  async getMetrics(): Promise<string> {
    return this.registry.metrics();
  }
}

// Expose on /metrics endpoint for Prometheus to scrape
// In your controller:
@Get('metrics')
async metrics(@Res() res: Response) {
  res.set('Content-Type', 'text/plain');
  res.send(await this.metricsService.getMetrics());
}

Pillar 3: Traces — Where in the System

Analogy: Package Tracking

When you order online and a package is delayed, you open the tracking page. You see every checkpoint: "Left warehouse 10:00, arrived sorting facility 14:00, out for delivery 08:00 next day — delayed at customs." Each checkpoint is a span. The entire journey from order to doorstep is a trace. Distributed tracing works the same way for a request that passes through multiple services.

In a microservices architecture, a single user-facing request might touch an API gateway, an auth service, a user service, a product service, and a payment service. Logs and metrics tell you that something is slow — but they can't tell you which service on the call chain added the latency. Traces can.

OpenTelemetry is the industry standard for instrumentation. Here is how you propagate trace context between two NestJS services:

// tracing.ts — initialize OpenTelemetry at app startup (before any imports)
import { NodeSDK } from '@opentelemetry/sdk-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { HttpInstrumentation } from '@opentelemetry/instrumentation-http';
import { NestInstrumentation } from '@opentelemetry/instrumentation-nestjs-core';

const sdk = new NodeSDK({
  serviceName: 'order-service',
  traceExporter: new OTLPTraceExporter({
    url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT, // e.g. http://jaeger:4318/v1/traces
  }),
  instrumentations: [
    new HttpInstrumentation(),     // auto-instruments outgoing HTTP calls
    new NestInstrumentation(),     // auto-instruments NestJS request lifecycle
  ],
});

sdk.start(); // must be before requiring @nestjs/core

// When order-service calls payment-service via HTTP,
// OpenTelemetry automatically injects trace context headers:
//   traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
// payment-service reads this header and continues the same trace.

// Adding a custom span inside a service method:
import { trace, SpanStatusCode } from '@opentelemetry/api';

async processOrder(orderId: string): Promise<Order> {
  const tracer = trace.getTracer('order-service');
  return tracer.startActiveSpan('processOrder', async (span) => {
    try {
      span.setAttribute('order.id', orderId);
      const result = await this.orderRepository.find(orderId);
      span.setAttribute('order.items_count', result.items.length);
      return result;
    } catch (err) {
      span.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
      throw err;
    } finally {
      span.end();
    }
  });
}

With traces flowing into Jaeger or Grafana Tempo, you get a flame graph showing exactly where time was spent across every service in a single request's journey. A 2-second response that felt mysterious in logs becomes instantly obvious: the payment service spent 1.8 seconds waiting for a database query that was missing an index.

How logs, metrics, and traces work together

Metrics tell you something is wrong: error rate spiked at 14:32. Traces tell you where: the 14:32 spike started in the inventory service. Logs tell you what: "Connection refused to database replica-3" repeated 847 times between 14:32 and 14:45. You need all three. Any two is not enough.

4. Graceful Shutdown in Kubernetes

Analogy: Closing a Restaurant at End of Service

A good restaurant manager doesn't flip the "CLOSED" sign and immediately kick out customers who are mid-meal. They stop seating new customers. They let the kitchen finish the existing orders. They allow diners to finish eating. Then they close. Kubernetes does the same thing to your pod — it sends a polite signal to stop accepting new work, waits for in-flight requests to finish, then terminates the process.

When Kubernetes needs to terminate a pod — during a rolling deployment, a node drain, or a scale-down — it sends a SIGTERM signal to the process. Your app has terminationGracePeriodSeconds (default: 30 seconds) to finish its work. After that, Kubernetes sends SIGKILL and the process is forcibly ended, dropping any in-flight requests.

Without graceful shutdown, every rolling deployment causes a burst of 502 or 503 errors for users whose requests were in-flight when the old pod was killed. Here is a correct implementation:

// main.ts — NestJS app with graceful shutdown
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // Enable shutdown hooks — NestJS will call onModuleDestroy() lifecycle
  // methods on all modules, giving them a chance to close connections cleanly.
  app.enableShutdownHooks();

  const server = await app.listen(3000);
  console.log('Application started on port 3000');

  // Handle SIGTERM (Kubernetes pod termination)
  process.on('SIGTERM', async () => {
    console.log('SIGTERM received — starting graceful shutdown');

    // Step 1: stop the HTTP server from accepting new connections.
    // Existing connections can finish their current request.
    server.close(async () => {
      console.log('HTTP server closed');

      // Step 2: close the NestJS application (closes DB connections,
      // message queue consumers, etc. via lifecycle hooks)
      await app.close();

      console.log('Application shut down cleanly');
      process.exit(0);
    });

    // Safety net: if shutdown takes longer than 25 seconds, force exit.
    // This should never happen if your in-flight requests are fast.
    setTimeout(() => {
      console.error('Graceful shutdown timed out — forcing exit');
      process.exit(1);
    }, 25_000);
  });
}

bootstrap();
# kubernetes/deployment.yaml — matching Kubernetes configuration
spec:
  terminationGracePeriodSeconds: 30   # Must be greater than your app's timeout
  containers:
    - name: api
      image: my-service:latest
      lifecycle:
        preStop:
          exec:
            # Kubernetes removes the pod from the Service's endpoint list
            # before sending SIGTERM. This sleep gives load balancers time
            # to stop routing new traffic to this pod before it shuts down.
            command: ["/bin/sh", "-c", "sleep 5"]
      readinessProbe:
        httpGet:
          path: /health/ready
          port: 3000
        initialDelaySeconds: 5
        periodSeconds: 10

The preStop hook timing gap

There is a race condition in Kubernetes: SIGTERM is sent at the same time the pod is removed from the Service endpoints — but load balancers may continue routing requests to the pod for a few hundred milliseconds while they catch up. The preStop: sleep 5 hook closes this window by delaying SIGTERM slightly, giving time for traffic to drain before shutdown begins. Do not skip it in production.

5. Rate Limiting for Reliability

Analogy: A Highway On-Ramp Meter

During rush hour, highway on-ramps use metering lights — one car per green light — to prevent the highway from becoming a parking lot. The individual delay is small, but without it the entire system grinds to a halt. Rate limiting works the same way: a small delay or rejection for one client protects the entire system from being overwhelmed.

Rate limiting has two contexts. Client-side rate limiting is what your code does when calling an external API — you add exponential backoff and jitter so you don't hammer a service when it's struggling. Server-side rate limiting is what your API does to protect itself from overload or abuse.

The two main algorithms are fixed window (count requests in the current minute, reset at the start of the next minute — simple but allows burst at window boundary) and sliding window (count requests in the last N seconds at any moment — more accurate, no boundary burst). Here is a Redis sliding window rate limiter:

// rate-limiter.ts — Redis sliding window implementation
import { Redis } from 'ioredis';

export class SlidingWindowRateLimiter {
  constructor(
    private readonly redis: Redis,
    private readonly windowMs: number = 60_000,  // 1 minute window
    private readonly maxRequests: number = 100,   // 100 requests per window
  ) {}

  /**
   * Returns true if the request should be allowed, false if rate-limited.
   * Uses a Redis sorted set where the score is the timestamp (ms).
   */
  async isAllowed(identifier: string): Promise<boolean> {
    const now = Date.now();
    const windowStart = now - this.windowMs;
    const key = `rate_limit:${identifier}`;

    // Atomic pipeline: remove old entries, count current, add new entry
    const pipeline = this.redis.pipeline();

    // Remove all entries older than the window
    pipeline.zremrangebyscore(key, 0, windowStart);

    // Count remaining entries (requests in current window)
    pipeline.zcard(key);

    // Add this request with current timestamp as score
    pipeline.zadd(key, now, `${now}-${Math.random()}`);

    // Set key expiry to clean up after window passes
    pipeline.expire(key, Math.ceil(this.windowMs / 1000));

    const results = await pipeline.exec();
    const currentCount = results![1][1] as number;

    // currentCount is BEFORE adding this request
    return currentCount < this.maxRequests;
  }
}

// NestJS guard using the rate limiter
@Injectable()
export class RateLimitGuard implements CanActivate {
  private readonly limiter: SlidingWindowRateLimiter;

  constructor(private readonly redis: Redis) {
    this.limiter = new SlidingWindowRateLimiter(redis, 60_000, 100);
  }

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const req = context.switchToHttp().getRequest();

    // Use user ID if authenticated, fall back to IP
    const identifier = req.user?.id ?? req.ip;
    const allowed = await this.limiter.isAllowed(identifier);

    if (!allowed) {
      throw new HttpException(
        { error: 'Too Many Requests', retryAfter: 60 },
        HttpStatus.TOO_MANY_REQUESTS,
      );
    }

    return true;
  }
}

6. How to Debug a Production Outage

This is the question that separates engineers who have been on-call from those who have not. The wrong answer is "I'd look at the logs." The right answer is a structured process that prioritizes mitigation over root-cause analysis.

Here is the process senior engineers follow:

  1. Check dashboards first. What is the error rate? Latency? Throughput? Is this a sudden spike or a gradual degradation? What time did it start? This tells you the shape of the problem before you look at any code.
  2. Check recent deployments. What deployed in the last 60 minutes? The majority of production outages are deploy-related. Check your CI/CD pipeline. If a deploy happened right before the incident, that is your prime suspect.
  3. Check infrastructure. Are pods healthy? Are nodes under memory or CPU pressure? Are databases accepting connections? Is disk full? Is a third-party dependency (payment processor, email provider, AWS service) having an incident?
  4. Check logs. Now that you know roughly when and where, filter logs to the affected service and time window. What new errors are appearing? What's different from 2 hours ago?
  5. Isolate the scope. Is it all users or only specific users, regions, or account types? Is it all endpoints or one endpoint? Does the error reproduce in staging? Narrowing the scope tells you whether this is data-specific, code-specific, or infrastructure-specific.
  6. Mitigate before you fix. If it's a bad deploy, roll it back — now. Don't spend 30 minutes root-causing while users are down. Roll back first, investigate second. If it's a load problem, scale up the pods. If it's a database issue, fail over to a replica. Your first job is to restore service. Understanding why comes second.

The most expensive debugging mistake

Jumping straight to code during an active incident. You start reading a function that looks suspicious. Forty minutes pass. Users are still down. The actual cause was a full disk on the database server. Always check infrastructure before code. It takes 5 minutes and eliminates 40% of outage causes.

Senior vs. Junior: "How do you know if your system is healthy?"

Junior answer

"I check if there are errors in the logs. If nothing is throwing exceptions, I assume the system is healthy."

Senior answer

"We define health in terms of SLOs. Our core SLIs are API error rate below 0.1% and P99 latency below 500 ms. We have Prometheus alerts that fire within 2 minutes of either metric drifting past 80% of the SLO threshold — that's an early warning before we've actually violated the SLO.

Beyond reactive metrics, I also watch leading indicators — things that predict problems before users feel them. Database connection pool saturation above 80% predicts a connection timeout storm. Queue depth growing monotonically predicts consumer lag turning into dropped messages. Pod restart counts increasing predicts an OOM issue that's about to get worse. I'd rather catch these at 9 AM on a Tuesday than at 2 AM on a Saturday."

Practice Questions

Beginner

What is the difference between P50 and P99 latency? Why does P99 matter more for user experience than the average?

Practice answer: P50 is the median — half of requests are faster, half are slower. P99 means 99% of requests completed faster than this value; 1% were slower. Average hides outliers: a handful of very slow requests raise the average only slightly but represent a real user experience problem. P99 captures those slow requests directly. In large-scale systems, 1% of users can be hundreds of thousands of people.

Intermediate

Your team's SLO is 99.9% availability for the month. It's the 20th of the month and you've already used 30 minutes of downtime. How do you approach the remaining 10 days?

Practice answer: Total allowed downtime for 30 days at 99.9% is ~43.8 minutes. With 30 minutes used by day 20, I have ~13.8 minutes left. I would freeze risky deployments, require extra review for any infrastructure changes, delay any planned maintenance to next month, and make the SLO status visible to the whole team. This is error budget management in practice — the budget is the forcing function for technical conservatism when it's running low.

Advanced

Users are reporting slow responses for the past 20 minutes. Dashboards show P99 latency jumped from 150 ms to 4,500 ms. Walk through your debugging process step by step.

Practice answer: First, note the exact time the spike started (say, 14:32). Check if any deployment happened between 14:15 and 14:32 — if yes, that's the prime suspect. Check pod health: are pods restarting? Check database connection pool metrics: are connections saturated? Check external dependency dashboards. Then filter logs to 14:32 onward: what error messages are new? If it looks deploy-related, roll back immediately rather than waiting for root cause. After service is restored, use distributed traces to find which service contributed the most latency to slow requests, then drill into that service's logs for the specific error. Write a postmortem: what broke, how long it took to detect, how long to mitigate, and what changes prevent recurrence.

Chapter Summary

  • Average latency hides suffering. Always measure and communicate latency in percentiles — P50, P95, P99. Tail latency (P99) is the experience of your most frustrated users.
  • SLI is what you measure, SLO is the target you set, SLA is the business contract. The error budget (1 − SLO) is how you decide when to move fast and when to be conservative.
  • The three pillars of observability are logs (what happened), metrics (how much over time), and traces (where in the system). You need all three; any two is insufficient for diagnosing complex distributed failures.
  • Graceful shutdown — stop accepting new requests, drain in-flight work, close connections, exit — prevents dropped requests during Kubernetes rolling deployments. The preStop sleep hook closes the load-balancer timing gap.
  • During a production outage, mitigate before you root-cause. Check dashboards, check recent deployments, check infrastructure — in that order. Roll back a bad deploy before spending an hour reading code.
