Part 4 · Chapter 19

Microservices and Distributed Systems

Microservices are not an architecture — they're a tradeoff. Know exactly what you're trading before you commit.

What you'll learn

  • The honest tradeoffs between monoliths and microservices, and when each wins
  • Conway's Law and why org structure drives architecture
  • Synchronous vs asynchronous service communication with real code
  • CQRS, Saga, Event Sourcing, and Circuit Breaker patterns with TypeScript examples
  • CAP Theorem and why you can never have everything

"The moment someone says 'we should use microservices' without finishing the sentence '...because our team is too large to coordinate on a monolith,' you should be worried. Microservices are not an architecture. They're a tradeoff."

Most engineers learn microservices by working with them. Fewer learn why they exist, and almost none can articulate clearly when they're the wrong call. In interviews, examiners are looking for the latter. This chapter gives you the honest, nuanced answer.

1. The Big Idea: Swiss Army Knife vs Toolbox

Analogy: A monolith is a Swiss Army knife — everything in one tool, easy to carry, convenient for most jobs. But if the corkscrew breaks, the entire knife has to come in for repair. Microservices are a toolbox of specialized tools. Each tool is independently replaceable and upgradeable. But you're now carrying a heavy box, tools have to be handed between workers, and coordinating who uses what requires real effort.

Neither is universally better. The question is always: what are you optimizing for?

2. Monolith vs Microservices — Honest Tradeoffs

DimensionMonolithMicroservices
DeploymentOne artifact, simpleMany artifacts, complex orchestration
Local developmentOne process to runRun 10+ services locally (docker-compose)
LatencyIn-process function calls (nanoseconds)Network calls (milliseconds)
Team independenceShared codebase — coordination requiredTeams own services — can deploy independently
Failure blast radiusOne bug can take down everythingOne service fails, rest can degrade gracefully
ScalingScale the whole app (wasteful)Scale only the bottleneck service
Data consistencySingle database, ACID transactions trivialDistributed transactions are hard; eventual consistency
DebuggingOne log stream, simple stack tracesDistributed tracing required across many services
Operational overheadLowHigh — CI/CD per service, service mesh, observability
When to use a monolith: Early-stage startups, small teams (<10 engineers), unclear domain boundaries, or any time speed of iteration matters more than independent deployability. Amazon, Netflix, and Airbnb all started as monoliths.
When to use microservices: Large organizations where multiple teams need to deploy independently. Domains with wildly different scaling requirements (e.g., image processing vs user auth). When a monolith has become genuinely painful to deploy.

3. Conway's Law

Analogy: If you have three teams working on a compiler, you'll get a three-pass compiler. Software architecture mirrors the communication structure of the organization that builds it. This isn't metaphor — it's observed reality.

Melvin Conway wrote in 1968: "Organizations which design systems are constrained to produce designs which are copies of the communication structures of these organizations."

The practical implication: asking "should we use microservices?" is really asking "is our organization structured to operate distributed systems?" If your team has one ops person and five developers all working together on Slack, deploying ten microservices will not make you faster. It will make you slower and more brittle.

Mature microservices organizations (Netflix, Spotify, Uber) have dedicated platform teams, mature internal developer platforms, and service ownership models baked into the culture. The architecture follows the org, not the reverse.

4. Service Communication Patterns

Synchronous: REST and gRPC

Analogy: A phone call. Both parties must be present and available. The caller waits on the line until the recipient responds. Use this when the caller needs the result before it can do anything else.

REST over HTTP is familiar and easy to debug with tools like curl and Postman. gRPC is faster (binary Protobuf encoding), supports streaming, and enforces a typed contract via .proto files — preferred for high-throughput internal service-to-service calls.

Synchronous call — direct API call from Order Service to Inventory Service:

// order-service/src/order.service.ts
import { Injectable, HttpException } from '@nestjs/common';
import { HttpService } from '@nestjs/axios';
import { firstValueFrom } from 'rxjs';

@Injectable()
export class OrderService {
  constructor(private readonly http: HttpService) {}

  async placeOrder(userId: string, productId: string, qty: number): Promise {
    // Synchronous call — we WAIT for inventory to respond before continuing
    const { data } = await firstValueFrom(
      this.http.post('http://inventory-service/reserve', { productId, qty })
    );

    if (!data.success) {
      throw new HttpException('Insufficient stock', 409);
    }

    // Only after a successful inventory response do we save the order
    await this.saveOrder({ userId, productId, qty, status: 'confirmed' });
  }
}

Asynchronous: Kafka and RabbitMQ

Analogy: An email. You write it, hit send, and immediately move on to your next task. You don't wait for a reply before doing something else. The recipient processes it when they're ready. Use this for fire-and-forget operations, fan-out (one event, many consumers), and decoupling services so one slowdown doesn't cascade.

Asynchronous — publishing an event after order is saved:

// order-service/src/order.service.ts
import { Injectable } from '@nestjs/common';
import { ClientKafka } from '@nestjs/microservices';

@Injectable()
export class OrderService {
  constructor(private readonly kafkaClient: ClientKafka) {}

  async placeOrder(userId: string, productId: string, qty: number): Promise {
    // Save immediately — don't wait for downstream services
    const order = await this.saveOrder({ userId, productId, qty, status: 'pending' });

    // Fire event and move on. Inventory, notification, analytics services all consume independently.
    this.kafkaClient.emit('order.placed', {
      orderId: order.id,
      userId,
      productId,
      qty,
      timestamp: new Date().toISOString(),
    });

    // Returns immediately — no waiting for inventory or email to finish
  }
}

// inventory-service/src/inventory.consumer.ts
import { Controller } from '@nestjs/common';
import { EventPattern, Payload } from '@nestjs/microservices';

@Controller()
export class InventoryConsumer {
  @EventPattern('order.placed')
  async handleOrderPlaced(@Payload() event: { orderId: string; productId: string; qty: number }) {
    await this.reserveStock(event.productId, event.qty);
    // If this fails, a dead-letter queue captures it for retry — order service never knows
  }
}
Key rule: Use synchronous communication when the caller cannot proceed without the response (e.g., checking if a user exists before creating a session). Use asynchronous when downstream work is independent — notifications, analytics, audit logs, inventory updates.

5. CQRS — Command Query Responsibility Segregation

Analogy: A library has two desks. The check-in/check-out desk handles commands — returning books, borrowing books, state changes. The reference desk handles queries — "where is the science fiction section?", "do you have this title?" — purely reading information. They're staffed differently, optimized differently. CQRS applies this to your codebase and optionally your databases.

Simple CQRS (same database, separate code paths) — the most common and practical starting point:

// commands/create-order.command.ts
export class CreateOrderCommand {
  constructor(
    public readonly userId: string,
    public readonly productId: string,
    public readonly qty: number,
  ) {}
}

// command-handlers/create-order.handler.ts
import { CommandHandler, ICommandHandler } from '@nestjs/cqrs';
import { CreateOrderCommand } from '../commands/create-order.command';

@CommandHandler(CreateOrderCommand)
export class CreateOrderHandler implements ICommandHandler {
  async execute(command: CreateOrderCommand): Promise {
    // Writes go through this handler — validates, applies business rules, writes to DB
    await this.orderRepository.save({
      userId: command.userId,
      productId: command.productId,
      qty: command.qty,
      status: 'pending',
    });
  }
}

// queries/get-orders.query.ts
export class GetOrdersQuery {
  constructor(public readonly userId: string) {}
}

// query-handlers/get-orders.handler.ts
@QueryHandler(GetOrdersQuery)
export class GetOrdersHandler implements IQueryHandler {
  async execute(query: GetOrdersQuery): Promise {
    // Reads can be optimized independently — add caching, read replicas, denormalized views
    return this.orderReadRepository.findByUser(query.userId);
  }
}

Full CQRS adds a separate read database (e.g., write to PostgreSQL, sync to Elasticsearch or a Redis read model). This enables independent scaling of reads vs writes and optimized query models. The tradeoff is eventual consistency between write and read stores.

When NOT to use CQRS: Simple CRUD applications where reads and writes have the same shape. If you're not hitting scale problems and your domain logic is straightforward, CQRS adds complexity for no return. Start simple.

6. The Saga Pattern — Distributed Transactions

Analogy: Booking a holiday package — flights, hotel, car rental. If the hotel is fully booked after you've confirmed the flight, you can't atomically roll back your flight in a single database transaction. Each booking system is separate. You must call each vendor and cancel individually. That's a saga: a sequence of local transactions with compensating actions when something goes wrong.

There are two saga coordination styles:

Choreography saga: each service listens for events and reacts. No central coordinator. Simple but hard to track the overall flow.

Orchestration saga: a central orchestrator tells each service what to do and handles compensation if anything fails. Easier to reason about. Example:

// order-saga.orchestrator.ts
@Injectable()
export class OrderSagaOrchestrator {
  async execute(order: CreateOrderDto): Promise {
    const sagaId = uuidv4();

    try {
      // Step 1: Reserve inventory
      await this.inventoryService.reserve({ sagaId, productId: order.productId, qty: order.qty });

      // Step 2: Charge payment
      await this.paymentService.charge({ sagaId, userId: order.userId, amount: order.totalAmount });

      // Step 3: Confirm order
      await this.orderRepository.updateStatus(order.id, 'confirmed');

      // Step 4: Send notification (fire-and-forget — failure here doesn't roll back the order)
      this.notificationService.send({ userId: order.userId, event: 'order_confirmed' }).catch(() => {
        this.logger.warn('Notification failed, will retry via queue');
      });

    } catch (error) {
      this.logger.error(`Saga ${sagaId} failed at step: ${error.step}`, error);

      // Compensating transactions — undo in reverse order
      await this.compensate(sagaId, error.step, order);
      throw new Error('Order placement failed — all steps rolled back');
    }
  }

  private async compensate(sagaId: string, failedStep: string, order: CreateOrderDto): Promise {
    if (failedStep === 'payment' || failedStep === 'confirm') {
      // Release the inventory reservation
      await this.inventoryService.release({ sagaId, productId: order.productId, qty: order.qty });
    }
    if (failedStep === 'confirm') {
      // Refund the payment
      await this.paymentService.refund({ sagaId, userId: order.userId, amount: order.totalAmount });
    }
  }
}
Idempotency is critical. Every saga step must be idempotent — safe to retry. The orchestrator may retry on transient failures. Use the sagaId to deduplicate repeat calls.

7. Event Sourcing

Analogy: Your bank account doesn't store "current balance: $1,234." It stores every transaction that ever happened: deposit $500, withdraw $200, deposit $934. The balance is derived by replaying those events. Event sourcing applies this to your entire application state.

Instead of storing current state, you store a log of every event that caused state changes. To get current state, replay the events.

Benefits: Full audit trail for free. Time travel — reconstruct state at any point in the past. Events become the integration point for other services. Easier to debug because you have the full history.

Challenges: Querying is non-trivial — you need projections (read models). Event schema evolution is hard. Storage grows unbounded (use snapshots after N events). Not suitable for every domain.

When to use event sourcing: Domains where audit trails are critical (banking, healthcare, compliance). High-value domains where you might need to replay history. Do not reach for it by default — the operational overhead is real.

8. Circuit Breaker

Analogy: Your home's electrical circuit breaker. Under normal load, current flows freely (Closed state). If a surge occurs, the breaker trips and cuts power immediately (Open state) — protecting the wiring from damage. After a timeout, it allows a small test current (Half-Open state) to check if conditions have normalized.

In distributed systems, if Service A calls Service B and B is slow or down, without a circuit breaker, A accumulates blocked threads and slow requests — eventually A falls over too. The circuit breaker detects failure and short-circuits calls to B immediately, returning a fallback response until B recovers.

Node.js example with opossum:

import CircuitBreaker from 'opossum';
import axios from 'axios';

const options = {
  timeout: 3000,           // If the request takes longer than 3s, count as failure
  errorThresholdPercentage: 50,  // Open circuit if 50% of requests fail
  resetTimeout: 10000,     // After 10s in Open, try Half-Open
};

async function callInventoryService(productId: string) {
  return axios.get(`http://inventory-service/stock/${productId}`);
}

const breaker = new CircuitBreaker(callInventoryService, options);

// Fallback: return a safe default when circuit is open
breaker.fallback(() => ({ available: false, reason: 'inventory-service unavailable' }));

breaker.on('open', () => console.warn('Circuit OPEN: inventory-service is failing'));
breaker.on('halfOpen', () => console.log('Circuit HALF-OPEN: testing inventory-service'));
breaker.on('close', () => console.log('Circuit CLOSED: inventory-service recovered'));

// Use the breaker instead of calling directly
const result = await breaker.fire('product-123');
console.log(result); // Safe even if inventory-service is down

9. CAP Theorem

Analogy: You're running a restaurant with multiple chefs. You can guarantee that all chefs have the same up-to-date menu (Consistency), that chefs can always take orders (Availability), and that the kitchen keeps running even if the comms system between stations fails (Partition Tolerance). But if communication between stations breaks down, you have to choose: stop taking orders until menus are re-synced (CP) or keep taking orders with potentially stale menus (AP). You can't do both.

CAP states: in a distributed system, you can guarantee at most two of Consistency, Availability, and Partition Tolerance. Since network partitions are a fact of distributed life, you effectively choose between CP and AP.

CP systems (Consistent + Partition-tolerant): when a network partition occurs, the system refuses to respond rather than return stale data. Examples: HBase, Zookeeper, traditional RDBMS in a cluster. Good for banking, financial ledgers, anything where stale data causes real harm.

AP systems (Available + Partition-tolerant): when a network partition occurs, the system continues serving requests but may return stale data. Examples: DNS, Cassandra, CouchDB, shopping carts (Amazon's Dynamo paper chose AP — better to show a slightly stale cart than refuse to serve the page). Data eventually becomes consistent once the partition heals.

Interview framing: "It depends on the use case. For a bank transfer I'd choose CP — stale balance data causes real financial harm. For a product recommendations system I'd choose AP — showing slightly stale recommendations is completely acceptable. The tradeoff is always between correctness and availability."

Senior vs Junior

Question: "When would you NOT recommend microservices?"

Junior Answer "I'd always recommend microservices because they're more scalable and modern."
Senior Answer "There are several clear cases where I'd push back. If it's an early-stage startup with a team under ten engineers, the operational overhead of microservices — separate CI/CD pipelines, service discovery, distributed tracing, managing multiple deployments — will eat velocity that the business can't afford. If the domain boundaries aren't well-understood yet, splitting into microservices means you'll need to do expensive refactoring later when you discover the boundaries were wrong. I'd also push back if the team doesn't have a mature DevOps culture. Microservices require platform engineering investment. Without it, you get all the downsides — distributed system complexity, network failures, cascading errors — without the benefits. I'd start with a well-structured modular monolith and extract services only when a specific pain point — team autonomy, independent scaling, or a genuinely distinct technical requirement — makes the tradeoff clearly worth it."

Practice Questions

Beginner

What is the difference between a monolith and microservices? Name one advantage of each.

A monolith is a single deployable unit where all code runs in one process — simple to develop and deploy. A microservice architecture splits the application into independently deployable services. Advantage of monolith: simple local development and easy refactoring across boundaries. Advantage of microservices: teams can deploy their service independently without coordinating with other teams.

Intermediate

Explain CQRS. When would you use it and when would you avoid it?

CQRS separates the code paths (and optionally the databases) for reads and writes. Commands mutate state; queries return data. Use it when read and write workloads have different scale requirements, when you need denormalized read models for performance, or when your domain logic benefits from explicit command/event modeling. Avoid it for simple CRUD applications — the added indirection creates overhead without payoff.

Advanced

Design an order placement flow using the Saga pattern. The flow involves payment, inventory reservation, and notification. What happens if payment succeeds but inventory reservation fails?

Use an orchestration saga. The orchestrator:

  1. Calls inventory.reserve() first — if it fails, no money has moved. Return error to user.
  2. Calls payment.charge() — if it fails, call inventory.release() as compensation, then return error.
  3. Calls order.confirm() — if it fails, call payment.refund() and inventory.release().
  4. Fires notification.send() as a non-critical fire-and-forget event — failure does not roll back the order.

Each compensation action must be idempotent. Use a unique sagaId (UUID) as an idempotency key passed to every service so retries don't double-charge or double-reserve. Store saga state in a durable store so the orchestrator can resume after its own crash.

Chapter Summary

  • Microservices are a tradeoff, not a default. Choose them when team independence and independent scalability genuinely justify the operational complexity.
  • Conway's Law: your architecture will mirror your org structure. Fix the org before you fix the architecture.
  • Use synchronous communication (REST/gRPC) when the caller needs the response to continue; use async (Kafka/RabbitMQ) for decoupled, fire-and-forget, or fan-out operations.
  • Sagas solve the distributed transaction problem with compensating actions. Always use idempotency keys.
  • CAP Theorem forces a choice between CP (consistent but may be unavailable) and AP (always available but possibly stale). Match the choice to the business requirement.
