1. OOP in Java — The 4 Pillars

Analogy — Four Engineering Principles: Think of a well-run factory. Workers are told what buttons to press (abstraction), not how the machines work internally (encapsulation). Different machines can fill the same role (polymorphism). A new machine inherits the base factory setup and adds specializations (inheritance). These aren't buzzwords — they're practical design decisions that determine how easy your code is to change six months later.

Encapsulation — the vending machine

You press buttons. You can't reach into the mechanism. That's encapsulation: private state, public interface. The machine can change its internal mechanism (a new motor) without breaking how you interact with it.

public class BankAccount {
    // Private state — callers can't manipulate directly
    private final String accountId;
    private double balance;
    private final List<String> transactionLog = new ArrayList<>();

    public BankAccount(String accountId, double initialBalance) {
        if (initialBalance < 0) throw new IllegalArgumentException("Initial balance cannot be negative");
        this.accountId = accountId;
        this.balance = initialBalance;
    }

    // Public interface — controlled access
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Deposit must be positive");
        this.balance += amount;
        transactionLog.add("DEPOSIT: " + amount);
    }

    public void withdraw(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Withdrawal must be positive");
        if (amount > this.balance) throw new IllegalStateException("Insufficient funds");
        this.balance -= amount;
        transactionLog.add("WITHDRAWAL: " + amount);
    }

    public double getBalance() { return this.balance; }

    // Return unmodifiable view — internal list stays private
    public List<String> getTransactionLog() {
        return Collections.unmodifiableList(transactionLog);
    }
}

Inheritance — family traits with @Override

public abstract class Animal {
    protected final String name;

    public Animal(String name) { this.name = name; }

    public abstract String makeSound(); // Must be implemented by subclasses

    public String introduce() {
        return "I am " + name + " and I say: " + makeSound();
    }
}

public class Dog extends Animal {
    private final String breed;

    public Dog(String name, String breed) {
        super(name); // Call parent constructor first
        this.breed = breed;
    }

    @Override
    public String makeSound() { return "Woof!"; }

    @Override
    public String introduce() {
        return super.introduce() + " (I'm a " + breed + ")";
    }
}

public class Cat extends Animal {
    public Cat(String name) { super(name); }

    @Override
    public String makeSound() { return "Meow!"; }
}

Polymorphism — the universal TV remote

One remote, many devices. You press "volume up" and each device handles it its own way. In Java: one reference type, many implementations.

public class Main {
    public static void main(String[] args) {
        // Animal reference — Dog/Cat behavior at runtime
        List<Animal> animals = List.of(
            new Dog("Rex", "German Shepherd"),
            new Cat("Whiskers"),
            new Dog("Buddy", "Golden Retriever")
        );

        // Same method call — different behavior per object
        for (Animal animal : animals) {
            System.out.println(animal.introduce());
        }
        // I am Rex and I say: Woof! (I'm a German Shepherd)
        // I am Whiskers and I say: Meow!
        // I am Buddy and I say: Woof! (I'm a Golden Retriever)
    }
}

Abstraction — abstract class vs interface

Rule: Use an abstract class when subclasses share state or partial implementation. Use an interface when you're defining a pure contract. Java allows implementing multiple interfaces but only one abstract class. 
// Interface — pure contract, no state
public interface Payable {
    double calculatePayment();
    String getPaymentDescription();

    // Default method (Java 8+) — optional override
    default String getFormattedPayment() {
        return String.format("Payment: $%.2f — %s", calculatePayment(), getPaymentDescription());
    }
}

// Abstract class — shared state + partial implementation
public abstract class Employee implements Payable {
    protected final String name;
    protected final String employeeId;

    public Employee(String name, String employeeId) {
        this.name = name;
        this.employeeId = employeeId;
    }

    // Common behavior shared by all employees
    @Override
    public String getPaymentDescription() {
        return "Employee: " + name + " (ID: " + employeeId + ")";
    }
    // calculatePayment() still abstract — each type computes it differently
}

public class SalariedEmployee extends Employee {
    private final double annualSalary;

    public SalariedEmployee(String name, String employeeId, double annualSalary) {
        super(name, employeeId);
        this.annualSalary = annualSalary;
    }

    @Override
    public double calculatePayment() { return annualSalary / 12.0; }
}

public class HourlyEmployee extends Employee {
    private final double hourlyRate;
    private final int hoursWorked;

    public HourlyEmployee(String name, String employeeId, double hourlyRate, int hoursWorked) {
        super(name, employeeId);
        this.hourlyRate = hourlyRate;
        this.hoursWorked = hoursWorked;
    }

    @Override
    public double calculatePayment() { return hourlyRate * hoursWorked; }
}

2. Java Collections — With Complexity and the Classic Gotchas

Analogy — Choosing the Right Tool: A screwdriver and a wrench are both tools, but using a screwdriver as a wrench will strip the bolt. Each Java collection has a different performance profile. Choosing ArrayList when you need a sorted map is like using a screwdriver on a nut — it'll technically work until it doesn't.
CollectionGetInsert (end)Insert (middle)Notes
ArrayListO(1)O(1) amortizedO(n)Grows by 1.5x; backed by array
LinkedListO(n)O(1)O(1) with iteratorHigh memory overhead; avoid for most use cases
HashMapO(1) avgO(1) avg—O(n) worst case (all same hash bucket)
LinkedHashMapO(1)O(1)—Maintains insertion order
TreeMapO(log n)O(log n)—Sorted by key; uses red-black tree
HashSetO(1)O(1)—Backed by HashMap; no duplicates
PriorityQueueO(1) peekO(log n)—Min-heap; does not guarantee order on iteration
 
import java.util.*;

public class CollectionExamples {
    public static void main(String[] args) {

        // ArrayList — use when random access matters
        // Specify expected size to avoid unnecessary resizing
        List<String> names = new ArrayList<>(100); // avoids resizing up to 100 elements
        names.add("Alice");
        names.add("Bob");
        System.out.println(names.get(0)); // O(1)

        // HashMap — use for key-value lookups
        Map<String, Integer> scores = new HashMap<>();
        scores.put("Alice", 95);
        scores.put("Bob", 87);
        scores.getOrDefault("Charlie", 0); // 0 — avoids NullPointerException

        // LinkedHashMap — when insertion order matters (e.g., LRU cache base)
        Map<String, String> config = new LinkedHashMap<>();
        config.put("host", "localhost");
        config.put("port", "5432");
        config.put("db", "myapp");
        // Iteration preserves: host, port, db order

        // TreeMap — when sorted order matters
        Map<String, Integer> sorted = new TreeMap<>();
        sorted.put("banana", 2);
        sorted.put("apple", 5);
        sorted.put("cherry", 1);
        sorted.forEach((k, v) -> System.out.println(k + "=" + v));
        // apple=5, banana=2, cherry=1 — alphabetically sorted

        // PriorityQueue — min-heap for task scheduling, Dijkstra's, etc.
        PriorityQueue<Integer> pq = new PriorityQueue<>();
        pq.add(5); pq.add(1); pq.add(3);
        System.out.println(pq.poll()); // 1 — smallest first
        System.out.println(pq.poll()); // 3
    }
}

== vs .equals() — the interview trap

Warning: == compares references (memory addresses). .equals() compares content. For Strings, always use .equals(). This is one of the most common Java interview gotchas at every level. 
public class EqualityGotcha {
    public static void main(String[] args) {

        // String pool — literals are interned
        String a = "hello";
        String b = "hello";
        System.out.println(a == b);       // true — same object in string pool
        System.out.println(a.equals(b));  // true

        // new String() bypasses the pool
        String c = new String("hello");
        System.out.println(a == c);       // FALSE — different objects!
        System.out.println(a.equals(c));  // true — same content

        // Integer caching gotcha — Java caches Integer values -128 to 127
        Integer x = 127;
        Integer y = 127;
        System.out.println(x == y);       // TRUE — cached, same object

        Integer p = 128;
        Integer q = 128;
        System.out.println(p == q);       // FALSE — beyond cache range, new objects!
        System.out.println(p.equals(q));  // true — always use .equals()

        // RULE: Never use == for Integer, Long, String, or any object comparison
        // Always use .equals() or Objects.equals() (null-safe)
        System.out.println(Objects.equals(p, q)); // true — null-safe equals
    }
}

3. Java Concurrency

Analogy — Shared Whiteboard: Three workers share one whiteboard. Worker A reads the current number (42), walks to get a marker, and by the time they write 43, Worker B has already changed it to 50. Both workers read the same value and wrote different results. This is a race condition. Coordination mechanisms (locks, atomic operations) ensure only one worker writes at a time.

The race condition — and the fix

import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

// BROKEN — race condition
class UnsafeCounter {
    private int count = 0;

    public void increment() {
        count++; // Read-modify-write is NOT atomic — three separate operations
    }

    public int getCount() { return count; }
}

// FIX 1: synchronized — mutual exclusion (heavier, fine for low contention)
class SynchronizedCounter {
    private int count = 0;

    public synchronized void increment() {
        count++; // Only one thread enters at a time
    }

    public synchronized int getCount() { return count; }
}

// FIX 2: AtomicInteger — lock-free, CAS-based (prefer for simple counters)
class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet(); // Atomic compare-and-swap — no locks needed
    }

    public int getCount() { return count.get(); }
}

// Demo — race condition in action
public class ConcurrencyDemo {
    public static void main(String[] args) throws InterruptedException {
        UnsafeCounter unsafe = new UnsafeCounter();
        AtomicCounter safe = new AtomicCounter();

        ExecutorService pool = Executors.newFixedThreadPool(10);
        List<Future<?>> futures = new ArrayList<>();

        for (int i = 0; i < 1000; i++) {
            futures.add(pool.submit(() -> { unsafe.increment(); safe.increment(); }));
        }

        for (Future<?> f : futures) f.get(); // Wait for all tasks
        pool.shutdown();

        System.out.println("Unsafe count: " + unsafe.getCount()); // Often < 1000 — race!
        System.out.println("Safe count:   " + safe.getCount());   // Always 1000
    }
}

volatile — memory visibility

// volatile ensures all threads see the latest value — NOT atomicity
class ServiceRunner {

            doWork();

    }

CompletableFuture — async composition

import java.util.concurrent.CompletableFuture;

public class AsyncDemo {

    // Simulated async operations
    static CompletableFuture<String> fetchUser(String userId) {
        return CompletableFuture.supplyAsync(() -> {
            // Simulates DB call
            return "User:" + userId;
        });
    }

    static CompletableFuture<String> fetchUserOrders(String userId) {
        return CompletableFuture.supplyAsync(() -> "Orders for " + userId);
    }

    static CompletableFuture<Double> fetchAccountBalance(String userId) {
        return CompletableFuture.supplyAsync(() -> 1500.00);
    }

    public static void main(String[] args) throws Exception {
        // Sequential chain: fetch user, then fetch their orders
        CompletableFuture<String> userThenOrders = fetchUser("u1")
            .thenCompose(user -> fetchUserOrders("u1"))   // Like Promise.then returning Promise
            .thenApply(orders -> "Summary: " + orders);  // Transform the result

        // Parallel: fetch user and balance simultaneously, combine when both done
        CompletableFuture<String> userFuture = fetchUser("u1");
        CompletableFuture<Double> balanceFuture = fetchAccountBalance("u1");

        CompletableFuture<String> combined = userFuture.thenCombine(
            balanceFuture,
            (user, balance) -> user + " — Balance: $" + balance
        );

        // allOf — wait for multiple completions in parallel
        CompletableFuture.allOf(userFuture, balanceFuture).join();

        System.out.println(userThenOrders.get());   // Summary: Orders for u1
        System.out.println(combined.get());         // User:u1 — Balance: $1500.0
    }
}
Tip: Never create raw new Thread() in production. Always use an ExecutorService with a managed thread pool. Raw threads are unmanaged, can proliferate unchecked, and don't give you lifecycle control or resource limits.

4. Spring Boot Patterns

Analogy — A Well-Organized Restaurant Kitchen: The @Controller is the front-of-house waiter taking orders. The @Service is the head chef with the business logic. The @Repository is the pantry — it knows where the ingredients are stored. Spring Boot's IoC container is the restaurant manager who hires them, wires them together, and makes sure everyone has what they need without anyone having to call the manager themselves. 
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.web.bind.annotation.*;
import org.springframework.stereotype.*;
import org.springframework.transaction.annotation.Transactional;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.data.jpa.repository.Query;
import jakarta.persistence.*;
import java.util.*;

// ============================================================
// Entry point
// ============================================================
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

// ============================================================
// JPA Entity
// ============================================================
@Entity
@Table(name = "users")
public class UserEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private String id;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(nullable = false)
    private String name;

    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL, fetch = FetchType.LAZY)
    private List<OrderEntity> orders = new ArrayList<>();

    // Getters and setters (or use Lombok @Data)
    public String getId() { return id; }
    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public List<OrderEntity> getOrders() { return orders; }
}

// ============================================================
// Spring Data JPA Repository
// ============================================================
public interface UserRepository extends JpaRepository<UserEntity, String> {
    Optional<UserEntity> findByEmail(String email);
    boolean existsByEmail(String email);
}

// ============================================================
// Request/Response DTOs
// ============================================================
record CreateUserRequest(String email, String name) {}
record UserResponse(String id, String email, String name) {}

// ============================================================
// Service layer — business logic lives here
// ============================================================
@Service
public class UserService {
    private final UserRepository userRepository;

    // Constructor injection — PREFER over @Autowired on fields
    // Reasons: testable without Spring context, immutable dependencies, clear contract
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    @Transactional
    public UserResponse createUser(CreateUserRequest request) {
        if (userRepository.existsByEmail(request.email())) {
            throw new IllegalArgumentException("Email already registered: " + request.email());
        }

        UserEntity entity = new UserEntity();
        entity.setEmail(request.email());
        entity.setName(request.name());

        UserEntity saved = userRepository.save(entity);
        return toResponse(saved);
    }

    // @Transactional — read-only hint for query optimization
    @Transactional(readOnly = true)
    public UserResponse findUser(String id) {
        return userRepository.findById(id)
            .map(this::toResponse)
            .orElseThrow(() -> new NoSuchElementException("User not found: " + id));
    }

    private UserResponse toResponse(UserEntity e) {
        return new UserResponse(e.getId(), e.getEmail(), e.getName());
    }
}

// ============================================================
// Controller — HTTP concerns only (no business logic)
// ============================================================
@RestController
@RequestMapping("/api/users")
public class UserController {
    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @PostMapping
    @ResponseStatus(org.springframework.http.HttpStatus.CREATED)
    public UserResponse createUser(@RequestBody CreateUserRequest request) {
        return userService.createUser(request);
    }

    @GetMapping("/{id}")
    public UserResponse getUser(@PathVariable String id) {
        return userService.findUser(id);
    }
}
@Transactional gotcha: If a method inside the same class calls another @Transactional method on this, the transaction does NOT activate — Spring's proxy is bypassed. To trigger a new transaction, you must call through the Spring proxy (inject the service into itself, or extract to a separate class). This is one of the most common Spring bugs in production.

5. Java Gotchas for Interviews

Analogy — Trapdoors in the Floor: Java's gotchas are trapdoors you don't see until you step on one. Knowing where they are lets you walk around them confidently — and spots you as an experienced engineer when you proactively mention them.

NullPointerException — use Optional

import java.util.Optional;

public class OptionalDemo {
    // Old style — risky
    public String getUpperCaseNameUnsafe(Map<String, String> map, String key) {
        String name = map.get(key); // Could be null!
        return name.toUpperCase(); // NullPointerException if key absent
    }

    // Safe with Optional
    public Optional<String> getUpperCaseName(Map<String, String> map, String key) {
        return Optional.ofNullable(map.get(key))
            .map(String::toUpperCase);
    }

    // At call site — no null checks
    public void demo() {
        Map<String, String> users = Map.of("u1", "alice");

        // Option 1: get with default
        String name = getUpperCaseName(users, "u999").orElse("Unknown");

        // Option 2: throw custom exception if absent
        String name2 = getUpperCaseName(users, "u999")
            .orElseThrow(() -> new NoSuchElementException("User not found"));

        // Option 3: execute only if present
        getUpperCaseName(users, "u1").ifPresent(n -> System.out.println("Found: " + n));
    }
}

Checked vs unchecked exceptions

// Checked exception — compiler forces you to declare or handle
// Subclass of Exception (not RuntimeException)
public class DatabaseException extends Exception {
    public DatabaseException(String message, Throwable cause) {
        super(message, cause);
    }
}

// You MUST declare it or catch it
public UserEntity loadUser(String id) throws DatabaseException {
    try {
        return repository.findById(id).orElseThrow();
    } catch (Exception e) {
        throw new DatabaseException("Failed to load user: " + id, e);
    }
}

// Unchecked exception — subclass of RuntimeException — no declaration needed
// Use these for programming errors (invalid args, state violations)
public void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("Age out of range: " + age);
    }
    this.age = age;
}

Autoboxing — the silent performance and NPE risk

public class AutoboxingDemo {
    public static void main(String[] args) {
        // Autoboxing: int -> Integer (implicit)
        List<Integer> numbers = new ArrayList<>();
        numbers.add(42); // autoboxed: Integer.valueOf(42)

        // Unboxing: Integer -> int
        int sum = 0;
        for (int n : numbers) {
            sum += n; // unboxed — tiny overhead per iteration
        }

        // The dangerous case: unboxing null
        Integer value = null; // Legal — Integer is an object
        int primitive = value; // NullPointerException at runtime!
        // Java tries to call value.intValue() on null

        // Fix: check before unboxing or use primitive int instead
        int safe = (value != null) ? value : 0;
    }
}

6. JVM Basics — What Interviewers Actually Ask

Analogy — Building Manager for Memory: The JVM is the building manager. The heap is the office space where objects live. The stack is each worker's personal desk — small, fast, cleared when the worker leaves. The garbage collector is the cleaning crew — they don't clean during meetings (GC pauses), and they focus on areas with the most garbage (generational GC).

Senior vs Junior Answer

Interview question: "What happens when you call new ArrayList()?"
Junior answer

"It creates an empty list."

Senior answer

"new ArrayList() creates an ArrayList backed by an array with an initial capacity of 10. When you add elements beyond that capacity, ArrayList allocates a new array at 1.5× the current size and copies all elements over — an O(n) operation, though amortized across many adds it's O(1) per add.

If you know the expected size upfront, you should call new ArrayList<>(expectedSize) to avoid the resizing overhead entirely. For example, if you're building a list from a result set of 500 rows, preallocate 500.

It's also worth noting that due to type erasure, ArrayList<String> and ArrayList<Integer> are the exact same class at runtime — the generic type parameter is a compile-time construct only. This means you can't do instanceof ArrayList<String>; you can only check instanceof ArrayList<?>."

Beginner

Question 1: Reverse a linked list iteratively

Write a Java method that reverses a singly linked list iteratively (not recursively). Include the Node class definition.

Show solution 
public class LinkedListReversal {

    static class Node<T> {
        T data;
        Node<T> next;

        Node(T data) {
            this.data = data;
            this.next = null;
        }
    }

    // Iterative reversal — O(n) time, O(1) space
    public static <T> Node<T> reverse(Node<T> head) {
        Node<T> prev = null;
        Node<T> current = head;

        while (current != null) {
            Node<T> next = current.next; // Save next before overwriting
            current.next = prev;          // Reverse the pointer
            prev = current;               // Move prev forward
            current = next;               // Move current forward
        }

        return prev; // prev is now the new head
    }

    // Helper to print the list
    public static <T> String listToString(Node<T> head) {
        StringBuilder sb = new StringBuilder("[");
        Node<T> current = head;
        while (current != null) {
            sb.append(current.data);
            if (current.next != null) sb.append(" -> ");
            current = current.next;
        }
        return sb.append("]").toString();
    }

    public static void main(String[] args) {
        // Build: 1 -> 2 -> 3 -> 4 -> 5
        Node<Integer> head = new Node<>(1);
        head.next = new Node<>(2);
        head.next.next = new Node<>(3);
        head.next.next.next = new Node<>(4);
        head.next.next.next.next = new Node<>(5);

        System.out.println("Before: " + listToString(head));
        // Before: [1 -> 2 -> 3 -> 4 -> 5]

        Node<Integer> reversed = reverse(head);
        System.out.println("After:  " + listToString(reversed));
        // After:  [5 -> 4 -> 3 -> 2 -> 1]
    }
}

Question 2: Thread-safe counter with AtomicInteger

Implement a thread-safe counter using AtomicInteger. Demonstrate why it's preferable to synchronized for this use case. Include a benchmark comparing both approaches under concurrent load.

Show solution 
import java.util.concurrent.*;
import java.util.concurrent.atomic.*;

public class CounterComparison {

    // Approach 1: synchronized — uses OS-level lock (mutex)
    // Every thread must acquire the lock, even read operations
    static class SynchronizedCounter {
        private int count = 0;

        public synchronized void increment() { count++; }
        public synchronized int getCount() { return count; }
    }

    // Approach 2: AtomicInteger — uses CPU-level CAS (Compare-And-Swap)
    // No OS lock needed — hardware instruction handles atomicity
    // Lower overhead, especially at high concurrency
    static class AtomicCounter {
        private final AtomicInteger count = new AtomicInteger(0);

        public void increment() { count.incrementAndGet(); }
        public int getCount() { return count.get(); }

        // Additional atomic operations not available with synchronized:
        public int incrementAndGet() { return count.incrementAndGet(); }
        public boolean compareAndSet(int expected, int update) {
            return count.compareAndSet(expected, update); // No-op if not expected value
        }
    }

    public static void main(String[] args) throws InterruptedException {
        final int THREADS = 50;
        final int INCREMENTS_PER_THREAD = 10_000;
        final int EXPECTED = THREADS * INCREMENTS_PER_THREAD; // 500,000

        // Test synchronized
        SynchronizedCounter syncCounter = new SynchronizedCounter();
        ExecutorService syncPool = Executors.newFixedThreadPool(THREADS);
        long syncStart = System.currentTimeMillis();

        for (int i = 0; i < THREADS; i++) {
            syncPool.submit(() -> {
                for (int j = 0; j < INCREMENTS_PER_THREAD; j++) {
                    syncCounter.increment();
                }
            });
        }
        syncPool.shutdown();
        syncPool.awaitTermination(30, TimeUnit.SECONDS);
        long syncTime = System.currentTimeMillis() - syncStart;

        // Test AtomicInteger
        AtomicCounter atomicCounter = new AtomicCounter();
        ExecutorService atomicPool = Executors.newFixedThreadPool(THREADS);
        long atomicStart = System.currentTimeMillis();

        for (int i = 0; i < THREADS; i++) {
            atomicPool.submit(() -> {
                for (int j = 0; j < INCREMENTS_PER_THREAD; j++) {
                    atomicCounter.increment();
                }
            });
        }
        atomicPool.shutdown();
        atomicPool.awaitTermination(30, TimeUnit.SECONDS);
        long atomicTime = System.currentTimeMillis() - atomicStart;

        System.out.println("Expected: " + EXPECTED);
        System.out.printf("Synchronized: %d (correct: %b) in %dms%n",
            syncCounter.getCount(), syncCounter.getCount() == EXPECTED, syncTime);
        System.out.printf("Atomic:        %d (correct: %b) in %dms%n",
            atomicCounter.getCount(), atomicCounter.getCount() == EXPECTED, atomicTime);
        // AtomicInteger is typically 2-5x faster under high contention
    }
}

// Why AtomicInteger wins for simple counters:
// synchronized: acquire lock (system call) -> do work -> release lock
//              Other threads BLOCK (context switch overhead)
// AtomicInteger: CPU executes LOCK CMPXCHG instruction (single atomic cycle)
//               If CAS fails (another thread won), retry — no thread sleeping
// Use synchronized when: protecting multiple operations that must be atomic together
// Use AtomicInteger when: single read-modify-write on one value

Question 3: Spring Boot endpoint — no N+1, transactional, sorted

Write a complete Spring Boot endpoint that: accepts a list of user IDs as a query parameter, fetches ALL users in a single database query (no N+1), returns them sorted by last name, is properly transactional, and handles cases where some IDs don't exist. Include the entity, repository, service, and controller.

Show solution 
import org.springframework.web.bind.annotation.*;
import org.springframework.stereotype.*;
import org.springframework.transaction.annotation.Transactional;
import org.springframework.data.jpa.repository.*;
import jakarta.persistence.*;
import java.util.*;
import java.util.stream.*;

// ============================================================
// Entity
// ============================================================
@Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private String id;

    @Column(name = "first_name", nullable = false)
    private String firstName;

    @Column(name = "last_name", nullable = false)
    private String lastName;

    @Column(nullable = false, unique = true)
    private String email;

    // Getters
    public String getId() { return id; }
    public String getFirstName() { return firstName; }
    public String getLastName() { return lastName; }
    public String getEmail() { return email; }
}

// ============================================================
// Repository — ONE query for all IDs using IN clause
// ============================================================
public interface UserRepository extends JpaRepository<User, String> {

    // Spring Data generates: SELECT * FROM users WHERE id IN (:ids)
    // This is the critical method — one query regardless of list size
    List<User> findAllByIdIn(Collection<String> ids);

    // Alternative with explicit JPQL for complex cases
    @Query("SELECT u FROM User u WHERE u.id IN :ids ORDER BY u.lastName ASC, u.firstName ASC")
    List<User> findByIdsOrderedByLastName(@Param("ids") Collection<String> ids);
}

// ============================================================
// Response DTO
// ============================================================
public record UserSummary(
    String id,
    String fullName,
    String email
) {}

// ============================================================
// Result type for partial-hit handling
// ============================================================
public record BulkUserResult(
    List<UserSummary> found,
    List<String> notFoundIds
) {}

// ============================================================
// Service
// ============================================================
@Service
public class UserBulkService {
    private final UserRepository userRepository;

    public UserBulkService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    @Transactional(readOnly = true) // readOnly hint: no dirty-checking overhead
    public BulkUserResult getUsersByIds(List<String> requestedIds) {
        if (requestedIds == null || requestedIds.isEmpty()) {
            return new BulkUserResult(Collections.emptyList(), Collections.emptyList());
        }

        // Deduplicate incoming IDs to avoid wasted work
        List<String> uniqueIds = requestedIds.stream()
            .distinct()
            .collect(Collectors.toList());

        // SINGLE query — IN clause (no N+1)
        List<User> foundUsers = userRepository.findByIdsOrderedByLastName(uniqueIds);

        // Determine which IDs were not found
        Set<String> foundIds = foundUsers.stream()
            .map(User::getId)
            .collect(Collectors.toSet());

        List<String> notFoundIds = uniqueIds.stream()
            .filter(id -> !foundIds.contains(id))
            .collect(Collectors.toList());

        // Map to response DTO — already sorted by last name from query
        List<UserSummary> summaries = foundUsers.stream()
            .map(u -> new UserSummary(
                u.getId(),
                u.getLastName() + ", " + u.getFirstName(),
                u.getEmail()
            ))
            .collect(Collectors.toList());

        return new BulkUserResult(summaries, notFoundIds);
    }
}

// ============================================================
// Controller
// ============================================================
@RestController
@RequestMapping("/api/users")
public class UserController {
    private final UserBulkService userBulkService;

    public UserController(UserBulkService userBulkService) {
        this.userBulkService = userBulkService;
    }

    // GET /api/users/bulk?ids=u1,u2,u3,u4
    @GetMapping("/bulk")
    public BulkUserResult getUsersByIds(
        @RequestParam List<String> ids // Spring auto-splits comma-separated values
    ) {
        if (ids.size() > 100) {
            throw new IllegalArgumentException("Cannot request more than 100 users at once");
        }
        return userBulkService.getUsersByIds(ids);
    }
}

Chapter Summary

  • OOP's four pillars — encapsulation, inheritance, polymorphism, abstraction — are not academic concepts. They're practical decisions: private fields with public interfaces prevent state corruption; polymorphism enables extension without modification; interfaces define contracts, abstract classes share implementation.
  • == vs .equals() is the most common Java bug in interviews. == compares references; .equals() compares content. Always use .equals() for Strings and object comparisons. Integer caching applies only to values -128 to 127 — beyond that, new Integer() objects are not reference-equal.
  • Never create raw threads in production — always use ExecutorService. For simple counters, prefer AtomicInteger over synchronized (lock-free CAS vs OS mutex). For protecting multiple operations atomically, synchronized or ReentrantLock is appropriate.
  • Spring Boot's three golden rules: prefer constructor injection over field @Autowired (testable, immutable), always use @Transactional on the service layer (not the controller), and never call @Transactional methods on yourself within the same class (proxy bypass is a silent bug).
  • Preventing N+1 queries is a signal of senior experience — use Spring Data's findAllByIdIn, JPQL with IN clauses, or JOIN FETCH for eager associations; never loop-query inside a service method. Set -Xms == -Xmx in production to avoid heap resizing overhead, and know the difference between StackOverflowError (deep recursion) and OutOfMemoryError (heap exhausted).
